An issue kind is a stable, machine-readable code that classifies a single audit finding (for example `expired` or `identity_mismatch`). The taxonomy is shared across agents so that downstream consumers — the partner results service, the portal, and Octy — can group and prioritize findings without parsing free text.

## IssueKindService

`IssueKindService` (`issue-kinds/issue-kind.service.ts`) is the runtime authority:

- On `onModuleInit` it loads `issue-kinds/issueKinds.yaml`, caches entries by `code`, and best-effort upserts them into the `issue_kinds` table (columns `code` unique, `category`, `description`).
- `getInfo(code)` returns `{ code, category, description? }`.
- `ensure(code)` throws `"Unknown issue_kind"` for any code not in the YAML.

<Note>
The YAML file is the source of truth. The in-memory cache survives a DB failure, so a transient database problem during boot does not break issue-kind resolution.
</Note>

## The catalog

`issueKinds.yaml` defines these codes, grouped by category:

<AccordionGroup>
  <Accordion title="PII">
    | Code | Meaning |
    |---|---|
    | `missing` | Field present in MISMO but absent in the supporting doc |
    | `inconsistent` | Value mismatch between MISMO and the supporting doc |
    | `identity_mismatch` | Doc belongs to a different person than the MISMO borrower (potential fraud) |
  </Accordion>
  <Accordion title="Validation">
    | Code | Meaning |
    |---|---|
    | `invalid` | Invalid value |
  </Accordion>
  <Accordion title="Credit">
    | Code | Meaning |
    |---|---|
    | `credit_low` | Middle credit score below the product minimum |
  </Accordion>
  <Accordion title="Document Health">
    | Code | Meaning |
    |---|---|
    | `expired` | Document exceeded its maximum age and requires update |
    | `expiring_soon` | Document will expire before closing |
    | `date_missing` | Date required for expiration validation not found |
    | `date_invalid` | Date found but format is invalid or unreadable |
    | `missing_document` | Required document not found in the loan file |
    | `conditional_document_unknown` | Conditional requirement could not be evaluated due to missing loan context |
    | `page_mismatch` | Declared page count differs from pages found |
    | `multi_doc_mismatch` | Multi-document bundle page total off |
    | `composite_mismatch` | Composite bundle page total off |
    | `independent_collection_incomplete` | Independent collection missing required pages |
    | `indeterminate` | Could not determine a final outcome from available evidence |
    | `signature_missing` | Required signature not detected on the document pages |
  </Accordion>
</AccordionGroup>

## How agents emit issue kinds

Agents write their findings to per-agent result tables (see [Reading audit results](/pipeline/results)), and each issue-bearing row carries an issue-kind value:

- The [PII agent](/agents/pii-validation) writes `pii_validation_results.issue_kind`. Its canonical enum is `missing | inconsistent | identity_mismatch`.
- [Document health](/agents/document-health) writes `doc_expiration_results.issue_kind` and `doc_completeness_results` codes (`expired`, `expiring_soon`, `page_mismatch`, and so on). Signature verification surfaces `doc_signature_verification_results.status` (`all_signed`, `partially_signed`, `unsigned`, `verification_failed`) as the issue kind; `verification_failed` is the non-blocking state added in migration 0200 for a failed vision call (distinct from a real `unsigned` reading).
- [Property red flags](/agents/property-red-flags) surface a `rule_id` as the issue kind rather than a taxonomy code.
- [Credit evaluation](/agents/credit-evaluation) surfaces `complianceResults[].ruleCode` (the `CRED_*` rule codes, for example `CRED_001`) with severity derived from each rule's `outcome` via `deriveSeverityFromOutcome` (`FAIL→HIGH`, `WARNING→MEDIUM`). These rule codes are not `issueKinds.yaml` codes; `credit_low` is the only credit entry in the YAML.

## Adjacent taxonomies (not issue kinds)

The issue-kind taxonomy above governs the audit-pipeline agents. Two adjacent classification surfaces are easy to confuse with it but are separate enums:

- **Credit rule codes** — `CRED_*` codes on `credit_runs.complianceResults[].ruleCode`, governed by the credit rule bundles, not `issueKinds.yaml`.
- **Statement Analyzer finding types** — the bank-statement / asset pipeline writes the `findings` table with its own `finding_type` CHECK constraint, a wholly separate taxonomy. Migration 0209 (`0209_extend_findings_check_for_new_types`) brought it to its current 15 values: `large_deposit`, `anomaly`, `fraud_flag`, `income_classification`, `validation_balance_mismatch`, `validation_date_out_of_range`, `validation_duplicate`, `validation_invalid_amount`, `validation_missing_field`, `fraud_pdf_integrity`, `fraud_font_consistency`, `fraud_continuity`, `llm_source_mismatch`, `nsf_overdraft`, and `income_trend` (`nsf_overdraft` and `income_trend` were the additions in 0209). The canonical list lives in `VALID_FINDING_TYPES` (`modules/bank-statement/bank-statement.processor.ts`). See the [enums reference](/data-model/enums-reference#finding-type-unions-bank-statement-pipeline) and [income analysis](/agents/income-analysis).

## Relation to result tables and severity buckets

The partner results service (`AuditResultsService`) reads each result table's issue-kind value and buckets it into critical or warning using `CRITICAL_ISSUE_KINDS` and `WARNING_ISSUE_KINDS`. See [Critical vs warning bucketing](/pipeline/results#critical-vs-warning-bucketing).

<Warning>
The agent-emitted enums and the YAML taxonomy are related but not identical surfaces. The results service's bucketing sets include codes the agents emit that are not all present in `issueKinds.yaml` — `mismatch`, `missing_required_field`, `unsigned`, `partially_signed`, and `verification_failed`. Note also that the PII agent does not emit the older single `mismatch` code; it uses `missing | inconsistent | identity_mismatch`.
</Warning>
