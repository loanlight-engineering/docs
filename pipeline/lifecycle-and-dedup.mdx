An `audit_runs` row tracks the lifecycle of a single audit. This page covers its statuses, the guards that protect terminal states, the deduplication mechanisms, and how run numbers are assigned under concurrency. The schema lives in `database/schema/audit-runs.ts`; see also [Core tables](/data-model/core-tables).

## Status lifecycle

`AuditRunStatus = "pending" | "processing" | "complete" | "failed"`.

```text
pending ──▶ processing ──▶ complete   (terminal)
                       └─▶ failed      (terminal)
```

The column defaults to `pending` at insert, but creation code inserts `processing` directly. Transitions are guarded:

| Method | Effect | Guard |
|---|---|---|
| `tryCompleteAuditRun(id)` | completes the run once every expected `agent_runs` row and `audit_infra_steps` row is terminal (calls `completeAuditRun`) | watch-based, idempotent; vacuous when the ledger is empty |
| `completeAuditRun(id)` | sets `completed_at`, `status="complete"` | `WHERE id = ? AND status IN ('pending','processing')` |
| `failAuditRun(id, step, msg)` | sets `completed_at`, `status="failed"` | `WHERE status IN ('pending','processing')` (idempotent) |
| `markAuditAsInterrupted(id)` | unconditionally sets `failed` | retained, but no longer invoked by a startup cleanup |
| `findIncompleteAudits(threshold=30)` | reads rows with `completed_at IS NULL AND started_at < threshold` | stuck-audit detection |

<Note>
`completeAuditRun` is guarded so a late success cannot overwrite a `failed` terminal state. This prevents a phantom success when an agent times out, the run is failed, and the agent then resolves late. `tryCompleteAuditRun` is the watch-based driver the orchestrator and queue workers call; its plan-aware guard reads `audit_runs.agent_config` so a phase-0 flip cannot complete a run whose later-phase rows have not been seeded yet.
</Note>

`AuditProcessingStep` (reported in failure events) runs through: `initializing, downloading_files, creating_loan, creating_audit_run, classifying_documents, preparing_pipeline, uploading_documents, waiting_for_pipeline, executing_agents, completing_audit`.

## Audit-run deduplication

On the EPC path, the `correlationId` is the `audit_runs.id`, which makes the row itself the dedup key. `AuditService.createAuditRun` decides what to do when a row already exists:

<AccordionGroup>
  <Accordion title="status is complete or failed">
    Returns `{ created: false, existingStatus }`. The request is rejected as a duplicate; the orchestrator emits `WORKFLOW.ERROR` with `errorCode: "DUPLICATE_AUDIT"` and returns early so Encompass resets its spinner.
  </Accordion>
  <Accordion title="status is processing, age < 55 min">
    Returns `{ created: false }`. Another task is actively working the run.
  </Accordion>
  <Accordion title="status is processing, age >= 55 min (stale)">
    Treated as orphaned. Atomically resets `started_at`, `status="processing"`, `completed_at=null`, and `mismo_xml_hash` (guarded `WHERE status='processing' AND completed_at IS NULL`) and returns `{ created: true }` for retry.
  </Accordion>
  <Accordion title="no existing row">
    Calls `createAuditRunImplementation(correlationId, loanId, lenderId, "epc", {...})`.
  </Accordion>
</AccordionGroup>

<Info>
The 55-minute stale threshold (`STALE_PROCESSING_THRESHOLD_MINUTES`) is intentionally below the SQS 60-minute visibility timeout, so an orphaned run is reset before the message redelivers.
</Info>

There is also an in-memory `processingAudits = new Set<string>()` keyed by `correlationId`, but it is **warn-only** — it logs "Duplicate concurrent execution detected". The real duplicate guard is the DB unique constraint surfaced by `created === false`. The `finally` block deletes the key.

## Stage deadlines and the false-kill invariant

Every stage is bounded so the stuck-audit cleanup cron can never false-kill a still-running healthy audit (#2412). Before this, only credit had a deadline, so an unbounded stage could outrun the cron — which marked it `failed`, after which the status-guarded `completeAuditRun` silently no-op'd the real success (a lossy failure, terminal for partner audits with no SQS redelivery).

- The four non-credit agents (`piiValidation`, `propertyData`, `documentHealth`, `redFlags`) are wrapped in `raceWithDeadline` against `NON_CREDIT_AGENT_BUDGET_MS` (default 15m, env-tunable). A timeout throws a tagged `NonCreditAgentDeadlineError`, recorded as an `audit_errors` row — the run still completes.
- URLA extraction is bounded by `URLA_EXTRACTION_BUDGET_MS` (default 45m). Because its output feeds every downstream agent, a timeout throws `UrlaExtractionDeadlineError` and **fails** the run rather than degrading it.
- The mechanism is abandonment (JavaScript cannot force-kill a promise): the race stops the wait and threads `config.signal` best-effort so abort-aware nodes can unwind.

The cleanup cron `cleanupStuckAuditRuns` (`llama-poller.service.ts`) marks a run `failed` once it is older than `AUDIT_RUN_CLEANUP_DEADLINE_MS` (165m). The headroom invariant is that this deadline provably exceeds the sum of the per-stage budgets, asserted at boot so an env override that closes the headroom fails fast.

## Recovery on deploy restart

Boot-time recovery used to be source-blind and false-killed healthy in-flight integration/partner audits on every deploy (#2451): the `processing_jobs` and `audit_runs` boot sweeps (`resumeInterruptedJobs`, `resumeIncompleteAuditRuns`) selected any non-terminal row regardless of source and failed runs older than 30 minutes, even though integration audits legitimately run up to ~57m and partner up to ~95m.

Now those boot paths recover **only `source = "los"`** work; integration and partner audits are left to the source-agnostic, headroom-aware cron above and to time-windowed orphan reapers:

- `AgentRunOrphanReaperService` — sweeps every 5m, reaping `agent_runs` rows stuck longer than a 3h window (`reapStuckAgentRuns`). The window must exceed the slowest live agent's P99.9 so a healthy run is never reaped.
- `AuditInfraStepOrphanReaperService` — the same pattern for `audit_infra_steps` rows on a 70m window.
- The orchestrator's phase barrier (`PhaseBarrierTimeoutError`) has a 4h ceiling, deliberately above the 3h reaper window so a genuinely-stuck row is reaped to `failed` (which the barrier reads as terminal) before the barrier fires.

`markAuditAsInterrupted` still exists but is no longer wired into a startup cleanup.

## Classify-readiness gate and the snapshot

The phase-0 classify-readiness gate (`awaitClassifyReadiness`) bounds each in-flight audit against a snapshot captured at intake — `audit_runs.snapshot_doc_ids` and `snapshot_taken_at` over the live, non-superseded `loan_documents`. A `classify_split_runs` row is excluded when its `started_at` is after the snapshot, so a classify run created inside the same audit cannot satisfy its own gate.

<Note>
The gate predicate must stay in lockstep with the classification reader (`getActiveClassificationsForLoan`). A re-audit of an already-classified loan produces only a `skipped` classify run for the current audit, so the loan's real classifications belong to a prior audit; the predicate is therefore status-scoped (`csr.status IN ('complete','partial')`) with no exclusive `audit_run_id` filter, matching the reader. Misaligning the two reintroduces the `snapshot_doc_no_csr` deadline that failed every integration re-audit before #2296.
</Note>

## Run numbering and races

`createAuditRunImplementation` assigns `run_number` per loan. `getNextRunNumber` is an unlocked `SELECT max(run_number)`, so it races. The unique constraint `audit_runs_loan_run_key` on `(loan_id, run_number)` (where `loan_id IS NOT NULL`) catches collisions:

- On a `23505` unique violation matching `audit_runs_loan_run_key`, it retries up to `MAX_RUN_NUMBER_RETRIES = 3`, re-reading the max each time.
- A primary-key race (same `id`) matched by `error.message.includes(auditRunId)` returns `{ created: false, existingStatus: "processing" }`.

Loanless runs have a null `loan_id` and null `run_number`; the CHECK `audit_runs_run_number_requires_loan` enforces `loan_id IS NULL OR run_number IS NOT NULL`.

## MISMO XML hash

`computeMismoXmlHash(mismoXml)` (`common/utils/hash.util.ts`) is stored on `audit_runs.mismo_xml_hash`. It drives URLA-extraction reuse: `UrlaPreprocessingService.checkForCachedUrlaExtraction` finds a prior completed run for the same loan with a byte-identical hash and reuses its `urla_extraction_results`, avoiding re-extraction on reprocess. The partner path sets the hash post-creation via `updateMismoXmlHash`. See [Pre-audit stage](/pipeline/pre-audit-stage#mismo-hash-caching).

## Per-document change detection

`classifySplitPersistence.shouldReclassify(context, registrationResult, allClassifyDocs, forceReclassify)` decides which documents need (re)classification:

- `forceReclassify` ⇒ classify all.
- Otherwise it selects distinct `loan_document_id`s already classified (joined through `classify_split_runs` and `audit_runs`, `is_dry_run = false`, any prior status).
- Classifies `newDocuments ∪ updatedDocuments` (from `DocumentRegistrationResult`, which compares source identity plus content hash) ∪ unchanged docs with no classification yet.
- If nothing needs classification, it records a skipped run and returns `{ needsClassification: false }`.

`hasInProgressRunsForLoan(loanId)` (pending/running, non-dry-run) gates partner audit creation with a `409`.

## Legacy LOS change interceptor

On the legacy path, `POST /api/v1/los/aus/webhook` uses `EncompassFieldChangeInterceptor` to decide whether the webhook represents a meaningful change before triggering an audit. The interceptor gates on the `CX.LLAUS` field so that unrelated Encompass field edits do not spawn audits. See [Triggering an audit](/pipeline/triggers#legacy-los-pull-path).

## Creation entry points

Runs are now minted by `AgentRouterService.route(...)`, which delegates to `AuditRunFactory.create(...)` — the canonical allocator — and surfaces a `created` flag for the dedup short-circuit above.

| Allocator | Path | `trigger_source` | Notes |
|---|---|---|---|
| `AuditRunFactory.create({...})` | all | any | the canonical allocator the router calls; dispatches by `trigger_source` |
| → `createAuditRun(correlationId, ...)` | EPC | `epc` | the `correlationId`-keyed dedup wrapper; lender resolved from the loan |
| → `createAuditRunDirect(auditRunId, loanId\|null, lenderId, triggerSource, {...})` | Partner / LOS | `partner`, `los` | fresh `id = uuidv4`; loan-optional |

The legacy `createAuditRunForPartner` is no longer the partner entry point. `CreateAuditRunResult = { auditRunId, created, runNumber: number|null, existingStatus?, startedAt? }`. `AuditTriggerSource = "epc" | "los" | "partner" | "portal" | "admin_rerun" | "shadow"`.
