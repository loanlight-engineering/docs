After [phase 0](/pipeline/pre-audit-stage), the [agent orchestrator](/pipeline/overview#router-then-orchestrator) runs the remaining phases of the router's `ExecutionPlan`. Each phase's agents fire concurrently via `Promise.allSettled` and are dispatched **by identity** through `AgentDispatcher` (`switch(agentType)`). A ledger-observed barrier separates the phases.

<Note>
The legacy `AuditAgentsExecutor.executeAuditAgents` no longer drives execution. `AuditAgentsExecutor` survives as a helper (for example `resolveLoanProgram`); the per-agent invocation that used to live in it now lives in the dispatcher's per-identity handlers.
</Note>

## Agent membership

Membership and dependencies come from `AGENT_PIPELINE` (`langgraph/agents/agent-pipeline.ts`), filtered to the lender's entitled set by the router.

| Phase | Agent type | Depends on | Agent page |
|---|---|---|---|
| 1 | `piiValidation` | `preAuditClassification`, `urlaExtraction` | [PII validation](/agents/pii-validation) |
| 1 | `propertyData` | `preAuditClassification`, `urlaExtraction` | [Property data](/agents/property-data) |
| 1 | `creditEvaluation` | `preAuditClassification`, `urlaExtraction` | [Credit evaluation](/agents/credit-evaluation) |
| 2 | `documentHealth` | `preAuditClassification`, `propertyData`, `urlaExtraction` | [Document health](/agents/document-health) |
| 2 | `redFlags` | `propertyData`, `urlaExtraction` | [Property red flags](/agents/property-red-flags) |
| standalone | `appraisalEvaluation` | — | [Appraisal evaluation](/agents/appraisal-evaluation) |

`piiValidation`, `propertyData`, and `documentHealth` carry `requiresIndexing` (they read the LlamaCloud pipeline). `creditEvaluation` and `redFlags` do not — credit loads S3 binary refs, red flags consume the property output.

## Dispatch and context handoff

For each in-plan agent the orchestrator calls `runAgent`, which dispatches by identity. Before phase 1 runs, the orchestrator rehydrates the phase-0 URLA handoff into the execution context so the phase-1 agents reuse it instead of re-extracting:

- `piiValidation` receives `fieldsOverride` from the URLA `transformedPiiFields` (falls back to LLM extraction if absent).
- `propertyData` receives `addressOverrides` and the primary borrower's housing-expense type (which feeds the refinance ATTOM gate).
- The dispatcher resolves and persists the expected closing date via `resolveExpectedClosingDate({ override, estimated, scheduled, closing })`, written to `audit_runs.expected_closing_date`.

Between phase 1 and phase 2, the property output enriches the shared context (for example `title_held_in_trust` from `propertyData.consolidatedSubject.ownership.trustIndicator`) before `documentHealth` and `redFlags` read it.

### Unmet dependencies

A phase advances only when the prior phase's `agent_runs` rows are terminal, but a dependency can still be `failed`. The channel decides per agent: an agent listed in `ctx.skipWhenDepsUnmet` whose in-plan predecessor did not reach `complete` is set `skipped` (for example LOS `redFlags` when `propertyData` failed); an agent not in the list runs **degraded** on undefined inputs (the EPC/Partner default for `documentHealth`/`redFlags`).

## Credit gating and fan-out

The `creditEvaluation` handler skips work (no graph run) when:

- `hasClassifiedCreditReportForLoan(loanId)` is false. This joins `document_classifications` → `loan_documents` → `classify_split_runs`, requiring `classification_type IN CREDIT_REPORT_CLASSIFICATION_TYPES`, `classification_status = "classified"`, `is_dry_run = false`, `deleted_at IS NULL`, and `superseded_by IS NULL`.
- `creditS3Bucket` is missing.
- The credit loan context resolves to null (fails closed when `loans.lender_id` is null).

<Warning>
`CREDIT_REPORT_CLASSIFICATION_TYPES` is a three-value allowlist (`integration-events/constants/credit-report-classifications.ts`): `credit_report`, `credit_report_trimerge`, `credit_report_single`. A strict `eq("credit_report")` would silently drop most real credit reports in production.
</Warning>

When credit runs, each downloaded document spawns a `createCreditEvaluationGraphWithSharedAuditRun(...)` call over the loaded docs (`Promise.allSettled`). Graph rejections are collected into an `AggregateError` with `auditedCreditDocumentId` on each `cause`; per-document failures write terminal `credit_runs` rows via `writeFailedSharedAuditCreditRun(...)`. S3-partial download failures write a terminal `credit_runs` row per rejected doc (no `audit_errors`), and the survivors continue to graph evaluation. Credit runs on shadow audits the same as any other agent.

## The standalone lane

After all phases settle, the orchestrator runs the standalone lane (`plan.standalone`) with no barrier. `appraisalEvaluation` is a `queue_worker` agent: the dispatcher enqueues a BullMQ job and the background worker owns its own `agent_runs` lifecycle (`processing → terminal`) and drives `tryCompleteAuditRun`.

## Per-agent error handling

The dispatcher owns each agent's status and error recording:

- An agent's `agent_runs` row transitions `pending → processing → complete | failed` inside the dispatcher; the orchestrator never writes those.
- `dispatch()` never throws for a per-agent failure — it records the failed `agent_runs` row and the `audit_errors` row itself and tolerates the failure.
- A credit `AggregateError` produces one `audit_errors` row per inner error, carrying `auditedCreditDocumentId` from `err.cause`.

<Warning>
Per-agent failures do not fail the audit. The run still reaches `complete` via `tryCompleteAuditRun`; failures are recorded only in `audit_errors`. `failAuditRun` is reserved for infrastructure failures that escape `AgentOrchestrator.run` — the `llama_index` infra step, a phase-barrier timeout, config drift, or the fatal URLA-extraction deadline (#2412). The canonical ADR is `docs/adr/ADR-audit-pipeline-error-semantics.md`.
</Warning>

See [Runs ledger](/agents/runs-ledger) for the `agent_runs` status model and [Router and entitlements](/agents/router-and-entitlements) for how the plan is resolved.
