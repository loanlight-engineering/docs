An audit run starts from one of three entry paths. Each path resolves to a `trigger_source` value on the `audit_runs` row, which lets you tell after the fact how a run was initiated.

| Path | Trigger | Orchestrator | `trigger_source` |
|---|---|---|---|
| EPC (production) | SQS message on `audit-requested-events` | `IntegrationOrchestrationService.processAuditRequest` | `epc` |
| Partner API | `POST /api/v1/partner/audits` | `IntegrationOrchestrationService.processAuditForPartner` | `partner` |
| Legacy LOS-pull | `POST /api/v1/los/aus` and `/aus/webhook` | `EncompassAuditOrchestrationService` / `DevConnectAuditOrchestrationService` | `los` |

The `trigger_source` CHECK allowlist also includes `portal`, `admin_rerun`, and `shadow` for runs created through other internal flows.

## EPC path (production)

This is the production revenue path. The external `loanlight-integrations` Encompass plug-in pushes the MISMO XML and PDFs to S3 and publishes an "Audit Requested" EventBridge event. That event lands on the SQS queue `audit-requested-events` (`dev-`-prefixed only when `DEPLOYED_ENVIRONMENT=dev`).

`QueueConsumerService.handleAuditRequested` (`messaging/queue-consumer.service.ts`, an `@SqsMessageHandler`) drains the queue:

<Steps>
  <Step title="Parse the EventBridge envelope">
    Reads `parsedBody.detail` from the message body (`{ detail, "detail-type", source }`).
  </Step>
  <Step title="Resolve and validate the bucket">
    `resolveIntegrationAuditS3Bucket(detail.s3Bucket, configService)` enforces the `INTEGRATION_AUDIT_S3_BUCKET_ALLOWLIST`.
  </Step>
  <Step title="Build and validate the request">
    Builds an `IntegrationAuditRequest` from `detail` and validates it with `IntegrationAuditRequestSchema.parse(...)` (Zod).
  </Step>
  <Step title="Orchestrate">
    Calls `orchestration.processAuditRequest(validated)`. Any error is logged and re-thrown to trigger the SQS dead-letter queue.
  </Step>
</Steps>

### Request shape — `IntegrationAuditRequestSchema`

<ParamField path="correlationId" type="string (UUID)" required>
Used as both the `loans.id` and the `audit_runs.id` on the EPC path. This is the dedup key.
</ParamField>

<ParamField path="requestorId" type="string" required>
Encompass client ID. Maps to `lender_auth_clients.client_id` for tenant attribution and gates portal access on `GET /audit_runs/:id/results`.
</ParamField>

<ParamField path="environment" type="string">
Used for logging and observability; should match the S3 bucket suffix.
</ParamField>

<ParamField path="s3Bucket" type="string">
Refined against the integration-audit bucket allowlist.
</ParamField>

<ParamField path="s3Prefix" type="string">
Regex-guarded (`^[a-zA-Z0-9][a-zA-Z0-9._\-/]{0,199}\/?$`), rejects `..` segments and leading `/`. Falls back to `correlationId` if absent. This is a path-traversal guard against a cross-tenant S3 read primitive.
</ParamField>

<ParamField path="loanMetadata" type="object">
Loan number, lender code, loan program, closing dates, ratios, citizenship, doc types, and more.
</ParamField>

<ParamField path="documents[]" type="array">
Optional manifest with per-file SHA-256 `fileHash` for change detection. Each entry: `docId?, docMapType?, fileName?, originalName?, s3Key?, mimeType?, isMismo?, fileHash?, isNew?`.
</ParamField>

<ParamField path="forceReclassify" type="boolean">
Forces re-classification even if documents are unchanged.
</ParamField>

<ParamField path="requestingParty" type="object">
Lender info (`name` required). `name` is the fallback for lender resolution when `requestorId` does not map.
</ParamField>

<ParamField path="additionalData" type="object">
Passthrough carrying shadow-audit flags: `isShadowAudit`, `shadowOfTransactionId`, `shadowAuditLenderName`.
</ParamField>

<Note>
The HTTP endpoint `POST /api/v1/integration-events/process` (`IntegrationEventsController`, guarded by `InternalApiKeyGuard`, `@HttpCode(202)`) runs the same Zod validation and fires `processAuditRequest` fire-and-forget. It is internal/test-only and slated for removal in LOA-174. Production ingress is SQS.
</Note>

## Partner API path

A lender or LOS calls `POST /api/v1/partner/audits` on a loan they have already uploaded. The endpoint (`PartnerAuditsController.createAudit`, `@HttpCode(202)`) is guarded by `PartnerAuthInterceptor` and `PartnerRateLimitInterceptor` (default 100 requests/min/partner).

### Request shape — `CreatePartnerAuditDto`

<ParamField path="loanId" type="string (UUID)" required>
The previously uploaded loan to audit.
</ParamField>

<ParamField path="agents" type="string[]">
Requested agents. The router intersects this with the lender's entitled set and expands the dependency closure. If omitted, all entitled agents run.
</ParamField>

<ParamField path="options.estimatedClosingDate" type="string (YYYY-MM-DD)">
Overrides the closing date used for document expiration checks.
</ParamField>

### Preconditions and routing

<Steps>
  <Step title="Loan lookup">
    `loanService.getLoanByIdAndLender(loanId, lenderId)` → `404` if not found.
  </Step>
  <Step title="Error gate">
    The only `409 Conflict` is when the loan's `processing_status` is `error` — typically a prior failed audit that must be explicitly cleared. There is no longer a status gate on `ready_for_audit`/`ready` or a classification-in-progress gate.
  </Step>
  <Step title="Route">
    `agentRouter.route(...)` is the only path (no feature flag). It mints the run, applies the entitlement filter (requested ∩ enabled), expands the required-closure (a named downstream agent pulls in its upstreams), seeds the `agent_runs` ledger, and writes `audit_runs.agent_config`. If the resolved plan is empty (`plannedAgentRuns.length === 0`), the freshly-minted run is deleted best-effort and the endpoint returns `403` with `denied[]`.
  </Step>
</Steps>

The response is `{ auditId, loanId, status: "processing", agents, denied, startedAt, progressUrl: "/api/v1/partner/audits/:auditId/progress" }`, where `agents` is the resolved plan (the entitled request plus its required-closure) and `denied` lists any requested agents filtered out (empty when nothing was denied). Execution runs in the background via `processAuditForPartner` → `AgentOrchestrator.run`; its `.catch` calls `failAuditRun` to cover pre-try throws.

See [Partner API audits](/partner-api/audits) and [Router and entitlements](/agents/router-and-entitlements) for detail.

## Legacy LOS-pull path

`POST /api/v1/los/aus/webhook` (Encompass) and `POST /api/v1/los/aus` (Dev Connect) in `los/los.controller.ts` pull the loan and documents directly from the LOS provider API rather than from S3. The webhook uses `EncompassFieldChangeInterceptor` (alongside signature, requestor-id, and rate-limit interceptors) to decide whether the change is meaningful before triggering `EncompassAuditOrchestrationService.perform`.

These orchestrators delegate to `DocumentProcessingService`, which now also routes through `AgentRouterService.route(...)` → `AgentOrchestrator.run(...)` (`trigger_source = "los"`, `skipIndexing = true` since the LOS path indexes upstream) and completes the run by watching the ledger.

<Warning>
Treat the LOS-pull path as deprecated. The Encompass `LosProvider` is a non-functional stub. New work targets the EPC and Partner paths. See [Legacy Dev Connect](/integrations/legacy-dev-connect).
</Warning>

## Statement Analyzer standalone runs

The [Statement Analyzer](/agents/income-analysis) has its own portal-driven entry path that produces an **asset run**, not an audit run. Portal staff upload bank statements through `POST /api/v1/portal/bank-statements/enqueue` (`BankStatementController`, guarded by `PortalAuthGuard`, roles `client_admin` / `client_user`). The request carries an optional free-text `loanId`.

When no loan is linked, the run is a **standalone (loanless) run**: the typed value is stored as `borrower_files.loan_reference`, and the resulting `asset_runs` row has `loan_id = NULL` and binds to `borrower_file_id` instead (migration 0207). Before #2489 a loanless upload extracted successfully but never created a run — the work silently evaporated.

<Note>
A Statement Analyzer run is a separate pipeline from the audit run. It runs the `statementAnalyzer` LangGraph agent (not part of the audit `AGENT_PIPELINE`) and writes `asset_runs`, `asset_accounts`, `bank_statements`, and `findings` — there is no `audit_runs` row and no `trigger_source`. See [income analysis](/agents/income-analysis) and [bank statement and DSCR API](/portal/api-bank-statement-dscr).
</Note>
