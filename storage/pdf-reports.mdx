LoanLight renders per-agent audit reports as PDFs and pushes them back to the loan origination system (Encompass) as document attachments. The rendering lives in the `pdf/` module of loanlight-api (`PdfTemplateService`, `pdf/pdf-template.service.ts`).

<Warning>
Audit PDFs are **not stored in S3.** They are generated in-process and uploaded to the LOS. The only file-system copy is a local debug write under `./pdf-output/` when `NODE_ENV=development`.
</Warning>

## Engine

The service uses **Handlebars** for templating and **Puppeteer** (headless Chromium) for HTML to PDF.

- Templates are resolved at runtime via `path.join(__dirname, "..", "pdf", "templates", "{name}.hbs")`. Compiled templates are cached in an in-memory `Map` (`compiledTemplates`); `clearTemplateCache()` clears it.
- `htmlToPdf(html, options)` launches Puppeteer with `{ headless: true, args: ["--no-sandbox", "--disable-setuid-sandbox", "--disable-dev-shm-usage", "--disable-gpu"] }` (LOA-1727: `--disable-dev-shm-usage` and `--disable-gpu` were added to prevent intermittent "Failed to launch the browser process" crashes caused by the container's default 64 MB `/dev/shm`), calls `page.setContent(html, { waitUntil: ["networkidle0", "domcontentloaded"], timeout: 30000 })`, then `page.pdf(...)`. The browser is closed in a `finally` block. On launch failure the sequence retries once before giving up.
- `registerHandlebarsHelpers()` runs in the constructor and registers helpers on the Handlebars singleton: `formatDate`, `severityBadge` (high/medium/low → `badge-*`), `issueIcon` (missing / inconsistent / default), `percentage`, `eq`, `gt`, and others.

<Warning>
Templates MUST be fully self-contained — inline CSS, no CDN/font/image fetches. Any external request stalls `networkidle0` for the full 30s timeout.
</Warning>

## The three templates

`TemplateType` is one of `data-discrepancy-audit-report`, `document-expiration-audit-report`, or `property-data-report`. `getTemplateNameForAgent(agentType)` maps an agent to its template:

| agentType | Template (`.hbs`) | Report |
|---|---|---|
| `PII` / `piiValidation` | `data-discrepancy-audit-report` | PII / data-discrepancy audit |
| `documentExpiration` | `document-expiration-audit-report` | Document expiration audit |
| `propertyData` | `property-data-report` | Property data report |
| anything else | `data-discrepancy-audit-report` (fallback) | — |

## Live render path: the FromNormalized builders

All production callers route through the `...FromNormalized` builders into a single entry point, `generateAgentDetailReportPdf(templateData, agentType, options): Promise<Buffer>`.

<AccordionGroup>
<Accordion title="buildPiiTemplateDataFromNormalized(normalized, reportMetadata)">
Builds PII template data from `pii_validation_results` (plus nested `pii_document_validations`). It defensively sorts `normalized.results` by `(field, borrower_index, id)` and validations by `(document_name, id)` before dedupe. Dedupe key: `field|borrower_index|encompass_attachment_name|page_ref|normalizeVal(doc_value)`.
</Accordion>
<Accordion title="buildDocumentExpirationTemplateDataFromNormalized(normalized, reportMetadata)">
Sorts results by `(document_type, document_name, id)` and nested `doc_expiration_evidence` by `(created_at, id)`, then uses `.find()` to pick the canonical `date_extraction` / `expiration_calculation` / `flag_type` evidence rows. It also pulls `fetchDocumentCompletenessResults`.
</Accordion>
<Accordion title="fetchDocumentCompletenessResults(auditRunId)">
Reads `doc_completeness_results` ordered `(document_name, id)` and classifies documents into failed (`page_mismatch`, `multi_doc_mismatch`, `composite_mismatch`, `independent_collection_incomplete`, or status `invalid_*`) versus warning.
</Accordion>
</AccordionGroup>

<Note>
`generateAgentDetailReportPdfFromRawData` and its helpers (`enhanceWithMultiDocumentData`, `fetchMultiDocumentAuditData`, `transformRawAuditData`, `transformDocumentExpirationData`, `createTemplateExtensions`) plus the `RawAuditData` / `RawAuditEntry` types are **dead code**. They have no production callers and survive only for the spec file. All live paths use the `...FromNormalized` builders.
</Note>

## Where the PDF goes

PDFs are generated in-process and uploaded to the LOS (Encompass) as attachments. In development the buffer is also written locally for inspection.

### Property data PDF

Generated in `langgraph/langgraph.service.ts` after property data is persisted. It lazily resolves `PdfTemplateService` via `moduleRef`, builds `PdfTemplateData` (`reportType: "Property Data"`), and calls `generateAgentDetailReportPdf(templateData, "propertyData", { format: "A4", orientation: "portrait" })`.

- In `NODE_ENV=development` it writes `./pdf-output/property-data-{loanId}-{auditRunId}.pdf`.
- When a loan-processing context with a token exists, it uploads to the LOS via `DevConnectDocumentService.ensureDocumentWithTitle(originalLoanId, "LLUE - Property Data", …)` then `uploadPdfAttachment(originalLoanId, docId, "property-data-report-{auditRunId8}.pdf", pdfBuffer, …)`.
- All PDF failures are caught and logged; they are non-fatal to the audit run.

### PII and document-expiration PDFs

Generated in `los/providers/dev-connect/dev-connect-document.service.ts`.

- For `documentExpiration`: `readDocumentExpirationResultsNormalized(db, auditRunId)` → `buildDocumentExpirationTemplateDataFromNormalized` → `generateAgentDetailReportPdf`.
- Otherwise (PII): query `pii_validation_results` (ordered `field, borrower_index, id`; nested `pii_document_validations` ordered `document_name, id`) → `buildPiiTemplateDataFromNormalized` → `generateAgentDetailReportPdf`.
- Uploaded to Encompass via `POST {baseUrl}/encompass/v3/loans/{loanId}/attachmentUploadUrl?view=id` (docGroup `Processing Docs`, docClassification `Other`). Filename: `{reportTitle-slug}-report-{auditRunId8}.pdf`. Existing attachments are de-duped via `checkForExistingAuditAttachment`. LOS container documents are titled `LLUE - ...`.

For more detail on the standalone PDF library used by the integrations service, see [integrations audit PDF](/integrations/audit-pdf).

<Note>
The AUS audit-report PDF generated by the integrations [audit-pdf](/integrations/audit-pdf) library is separate from the `pdf/` module above. It gained a Credit section and new Loan Details fields; the api side feeds it via `GET /api/v1/audit_runs/:id/results?includeEncompassLoanDetails=true`, which surfaces the Encompass-only Loan Details fields (Income Type, Note Rate) in `summary.loanDetails` for the Encompass split-delivery PDF fetcher (LOA-1625). The PDF structure itself lives in audit-pdf (owned elsewhere).
</Note>

## Appraisal Review Summary export PDF

A separate PDF path lives in the portal-appraisal module (`AppraisalExportDataService.renderReviewPdf`), distinct from the three Handlebars templates above. It renders an appraisal run's **Review summary** (property facts, AI findings, and program-rule outcomes) as a LoanLight-branded PDF. The same `renderReviewPdf` entry point is used by two separate delivery paths:

**Portal export (user-triggered):**
- **Endpoint** — `POST /api/v1/portal/appraisal/:id/export` (`PortalAppraisalController`, roles `client_admin` / `client_user` / `super_admin`, throttled 5/min because the render is expensive). Only `completed` / `reviewed` runs export; others 409.
- **Delivery** — uploaded to S3 under the 4-segment `exports/{lenderId}/{runId}/{ts}/{filename}` key (tenant-isolated, lowercased to satisfy `EXPORT_KEY_PATTERN`) and returned as a signed URL via `SignedUrlService`. It is a portal download, not pushed to the LOS.

**Inline EPC delivery (automatic, LOA-1677):**
- When an EPC audit has an appraisal agent that completes successfully before audit completion, `appraisal.processor.ts` calls `AppraisalExportDataService.renderReviewPdf` and PUTs the PDF to the shared `integration-files` S3 bucket, stashing `{ s3Key, filename, propertyAddress }` on the agent-run output.
- `AuditService.finalizeAuditCompletion` reads those fields and carries them on the `workflow:complete` EventBridge event as `appraisalReviewPdfS3Key` / `appraisalReviewPdfFilename` / `appraisalReviewPropertyAddress`.
- `loanlight-integrations` best-effort fetches the PDF from the `integration-files` bucket and files it into the Encompass eFolder inline with the AUS + property-data push, routed to its own "Appraisal — LoanLight Review" container (`appraisal_review_summary` routing type, migration 020 routing seed).
- A failed or timed-out appraisal completes the audit without the PDF — AUS + property-data delivery is never blocked.

**Common engine:**
- `renderAppraisalReviewHtml` (`appraisal-pdf-template.ts`) emits self-contained HTML (hand-rolled, not Handlebars); the `bank-statement-export` `PdfWriterService` (Puppeteer) renders it to a buffer.
- **Contents** — header (address, form type, loan + program); key facts; an **Issues Found** section surfacing failed/warning rules; a Findings section (ATTOM-enrichment AVM/market findings plus deny-list and Mashvisor rental findings); Sales Comparables (all forms) and, for form 1025 only, Unit Details and Operating Income tables; an AI "Summary of Appraiser Comments" (read cache-only); and the Program Rules outcomes.

<Note>
Despite the "Review Summary" PDF and the audit-report PDFs both being appraisal-adjacent, they are different mechanisms: the export here renders HTML with the Puppeteer writer; the per-agent appraisal audit report goes through the `pdf/` Handlebars/Puppeteer service (`data-discrepancy-audit-report` fallback) and is pushed to the LOS.
</Note>

## Dev preview server

`pdf/dev/template-dev-server.ts` (with `start-dev-server.ts`) serves the templates at `http://localhost:3001` (`pnpm template:dev`) using fixtures in `pdf/dev/fixtures/` (`pii-sample-enhanced.json`, `document-expiration-sample.json`, `property-data-sample.json`). Refresh the browser manually to see template changes.
