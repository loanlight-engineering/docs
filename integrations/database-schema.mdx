The encompass service owns its own database: Aurora Serverless v2 PostgreSQL, schema `loanlight_epc`. It is separate from the loanlight-api / AUS database.

## Migrations

Migrations live in `packages/encompass/migrations/*.sql` and are applied by the `DbMigrate` Lambda custom resource on every deploy.

<Warning>
There is **no applied-migrations ledger.** Every `.sql` is re-run whenever the migrations' content hash changes, so **every migration must be idempotent** (`CREATE TABLE/INDEX IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`, guarded `ALTER`). A non-idempotent statement wedges `LLIntegrationsDatabaseStack` in `UPDATE_ROLLBACK_FAILED` and blocks all deploys (LOA-1004).
</Warning>

The current schema has **14 tables**. Migration `001_baseline_schema.sql` creates the original baseline; the numbered migrations 002 through 022 add and amend the rest. Both runners order migrations **lexically** (`DbMigrate` via `Array.prototype.sort()`, `run-migrations.sh` via the shell glob), so there are two `014_*` files and a final `zzzz_reapply_role_grants.sql` that — because digits sort before lowercase letters — always runs last.

<Note>
The `encompass/README.md` "expected tables" list is stale — it names only the 6 baseline tables. The nine tables added by migrations 003–019 (`audit_details`, `transaction_document`, `document_push_record`, `lender`, `audit_delivery_log`, `document_push_routing_config`, `materialized_split_record`, `lender_feature`) are missing from it. Trust the schema, not the README.
</Note>

<Note>
`zzzz_reapply_role_grants.sql` exists because `DbMigrate` runs every migration as the **app** user, while `db-init` set `ALTER DEFAULT PRIVILEGES` as the **master** user. Default privileges are keyed to the creating role, so tables created by the app user never inherit the read-role grants — a newly added table is invisible to `db_readonly` / `db_readwrite` until something re-grants it (this is how `lender_feature` first landed `permission denied` for `db_readwrite`). Re-grant lives only in this file; do not add per-table `GRANT`s to individual migrations.
</Note>

|---|---|
| `id` | PK |

| `document_id`, `s3_key`, `classification_type` | |

<Note>
The `epc_delivery_records` table and the `epc-delivery` package were **dropped** in migration `022` (LOA-1678). They were dead scaffolding for a durable outbox delivery approach that was de-scoped in favor of the synchronous model. `transaction_state.delivery_cancelled_after` was removed in the same migration.
</Note>

### lender

Per-lender delivery behaviour (added in 012).

| Column | Notes |
|---|---|
| `id` | PK |
| `name` | |
| `audit_delivery_mode` | `single` / `split` / `pipeline`, default `single`. **Retired** — the `document_push` flag on the `lender_feature` table is now the sole switch for routed delivery (LOA-948). This column remains in the schema as a legacy artifact. |
| `workos_org_id` | text, nullable; added in 014. The lender's WorkOS org id, sent as the `X-Requestor-ID` on the trusted internal audit-PDF data fetch. Unprovisioned lenders fall back to `provider_client_id`. |

Shadow audits resolve a lender by name via this table — see [Shadow audits](/integrations/shadow-audits). `workos_org_id` is also how the [routing-config admin API](/integrations/routing-config-admin) maps an `:orgId` path param to a `lender.id`.

### audit_delivery_log

Per-document outcome ledger for split deliveries (added in 013). Since LOA-989 (migration 021), the ledger write happens **before** the terminal Encompass PATCH — not after — so a PATCH failure or ambiguous PATCH result still records what was uploaded. The upsert key `(transaction_state_id, file_name)` makes a later re-delivery safe; the upsert is monotonic (a `delivered` row is never downgraded to `failed`). Core documents only (`aus_report` / `property`); the best-effort appraisal review-summary is excluded.

| Column | Notes |
|---|---|
| `id` | PK |
| `transaction_state_id` | |
| `document_kind` | `aus_report` / `property` |
| `property_address`, `file_name`, `s3_key`, `epc_resource_id`, `container_type` | |
| `status` | `delivered` / `failed` (added in 021; existing rows default `delivered`) |
| `error_class` | Upload failure class: `network` / `rate_limit` / `http_error` / `circuit_open` / `unknown`; NULL on success (added in 021) |
| `error_detail` | Error message text; NULL on success (added in 021) |
| `attempts` | Total upload attempts including retries; 1 on first-attempt success (added in 021) |

### document_push_routing_config

Versioned EPC document-routing config (added in 015): a global default plus optional per-lender overrides. Authored and activated through the [routing-config admin API](/integrations/routing-config-admin); onboarding or editing a lender is pure DML against this table, never a schema migration.

| Column | Notes |
|---|---|
| `id` | PK (uuid) |
| `lender_id` | FK → `lender`; **NULL = global default scope**, non-NULL = per-lender override |
| `version` | integer; unique within a scope |
| `rules` | jsonb (`{ defaultContainer, rules: { <classificationType>: rule } }`) |
| `is_active` | boolean, default `false` |
| `created_at` | |

Three partial unique indexes enforce the invariants: at most one active global row (`is_active AND lender_id IS NULL`), at most one active row per lender, and version uniqueness within each scope (global and per-lender versions advance independently).

### materialized_split_record

De-dup ledger for the split-document materializer (added in 018; LOA-1102). One row per materialized split, so re-audit and `forceReclassify` re-runs do not re-carve.

| Column | Notes |
|---|---|
| `id` | PK (uuid) |
| `loan_id`, `original_doc_id` | text |
| `page_start`, `page_end` | integer, nullable (NULL = whole-doc) |
| `content_hash` | SHA-256, nullable |
| `classification_id` | text; threaded onto the `documents.materialized` return event |
| `split_s3_key` | text |
| `created_at` | |

The de-dup key is a unique index over `(loan_id, original_doc_id, COALESCE(page_start,-1), COALESCE(page_end,-1), COALESCE(content_hash,''))`. It deliberately does **not** key on `classification_id` (which `forceReclassify` regenerates).

### lender_feature

Per-lender feature gates (added in 019; LOA-1336). Column names mirror octy's `lender_feature_flags`. The only key used today is `document_push`.

| Column | Notes |
|---|---|
| `lender_id`, `feature_key` | composite **PK** |
| `enabled` | boolean, default `false` |
| `feature_value` | text, nullable |
| `created_at`, `updated_at` | |

<Note>
Default OFF is the **absence of a row** — a lender is enabled for a feature only when a row exists with `enabled = true`. There is no backfill of existing lenders. The [document-push consumer](/integrations/messaging) checks `EpcClientRepository.isLenderFeatureEnabled(lenderId, "document_push")` before materializing.
</Note>

## Access

Database roles: `db_readonly` (all envs) and `db_readwrite` (int only). The bastion is shared with the API DB; connect on local port 5433 to avoid the 5432 conflict (`db:connect:int:read` / `db:connect:int:write` scripts). See [Integrations infrastructure](/integrations/infrastructure).
