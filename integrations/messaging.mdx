The encompass service is event-driven. Inbound webhooks fan out to SQS; the handoff to the AUS and the return of progress and results travel over EventBridge, routed back into SQS by rules. This page documents every queue, rule, and event.

## SQS queues

Defined in `constants/queue-names.ts` and provisioned by `worker-stack.ts`. Consumers use `batchSize: 1`, and SQS handler errors re-throw to trigger the DLQ.

|| Logical | Queue name | Visibility timeout | Consumer | Producer / source |
||---|---|---|---|---|
||| Health check | `health-check` | 30 s | `handleHealthCheck` (SELECT 1) | `WorkerService.enqueueHealthCheck` |
||| Transaction events | `transaction-event-handling` | 900 s | `handleTransactionEvent` | `WorkerService` (from webhook) |
||| AUS updates | `handle-aus-updates` | 30 s | `handleAusUpdate` | EventBridge `AusUpdateRule` |
||| Document push | `document-push` | 900 s | `handleDocumentPush` | EventBridge `DocumentPushRule` |
||| Lender events | `lender-event` | 30 s | `handleLenderEvent` | EventBridge `LenderEventRule` (LOA-1015) |

Queue config env vars are validated at boot: `SQS_{HEALTH_CHECK,TRANSACTION_EVENT,AUS_UPDATE,DOCUMENT_PUSH,LENDER_EVENT}_QUEUE_{URL,NAME}` plus `AWS_REGION`.

The `WORKER_TYPES` enum (`messaging/constants.ts`) names the job types carried in messages: `handle-health-check`, `handle-transaction-created-event`, `handle-transaction-updated-event`, `handle-aus-updates`.


## EventBridge rules

Three rules (`worker-stack.ts`) route external events into SQS.

### AusUpdateRule

Matches source `["loanlight.aus"]` and a fixed allow-list of detail-types, routing to the `handle-aus-updates` queue:

```text
workflow:start
workflow:complete
workflow:error
document_processing:initialization
document_processing:downloading
document_processing:uploading_to_ai
document_processing:pipeline_monitoring
document_processing:documents_ready
agent:pii_validation_{start|analyzing|complete}
agent:document_expiration_*
agent:credit_evaluation_*
agent:liabilities_assessment_*
```

`QueueConsumerService.handleAusUpdate` parses both nested (`detail.data.*`) and flat (`detail.*`) formats, extracts `correlationId` (falling back to `auditRunId`), and dispatches:

- `workflow:complete` â†’ `AusUpdateService.handleAusUpdate({ auditState: Complete })`. When the audit's appraisal agent succeeded, `Detail.data` also carries `appraisalReviewPdfS3Key`, `appraisalReviewPdfFilename`, and `appraisalReviewPropertyAddress` â€” the consumer threads these into `DeliverCompleteContext` so `AusUpdateService.deliverComplete` can best-effort fetch the PDF and file it inline in the same eFolder batch as the AUS + property-data push (LOA-1677). Fields are absent for non-appraisal audits or a failed/timed-out appraisal.
- `workflow:error` â†’ `auditState: Error` (recoverable); `DUPLICATE_AUDIT` is ignored; skipped if already in a final state.
- `workflow:start` â†’ reset `audit_details` to `Processing` and PATCH Encompass to `Processing`; shadow runs are skipped entirely.
- progress / `document_processing:*` / `agent:*` â†’ `updateAuditProgress(correlationId, eventType, message, percent)`. API-supplied message/percent are preferred; otherwise fallback maps in `getProgressMessage` / `getProgressPercent` apply. Agent event names parse as `agent:{name}_{start|analyzing|complete}`.

### DocumentPushRule

Matches source `["loanlight.aus"]`, detail-type `["loanlight.documents.classified"]`, routing to the `document-push` queue. This drives the document-push pipeline, where the AUS asks encompass to file classified documents back into the Encompass eFolder.

<Note>
The source is `loanlight.aus`, not `loanlight.api`. loanlight-api emits `loanlight.documents.classified` with `Source: "loanlight.aus"` (its `publishDocumentsClassified` path). The rule matched `loanlight.api` for a while, which meant the event never routed to the push queue â€” fixed during the LOA-1102 / LOA-1041 contract reconciliation. The in-code comment in `worker-stack.ts` still flags the old value.
</Note>

<Note>
`DocumentsClassifiedEvent.documents` (the array of classified docs in the event payload) is now **optional** (LOA-1340, thin-event migration). The consumer no longer reads this field â€” it pulls authoritative classification rows from `GET /api/v1/audit_runs/{auditRunId}/classifications` instead. The field will be removed from the event once all producers are thinned.
</Note>

`QueueConsumerService.handleDocumentPush` runs the consumer:

1. Checks the global kill-switch `appConfig.documentPushEnabled` (`DOCUMENT_PUSH_ENABLED`). When `false` it throws, returning the message to the queue (a prolonged stop eventually drains the backlog into the DLQ). It is `true` in INT.
2. Validates the `detail` with `DocumentsClassifiedEventSchema` (invalid payloads are dropped).
3. Resolves the transaction from the integrations DB by `correlationId` â€” `getTransactionRecordByCorrelationIdOrThrow(event.loanId)`, where `event.loanId == transaction_state.id == auditRunId == correlationId` (LOA-1491). The S3 path emits `encompassLoanId: null`, so the real Encompass loan GUID is read from the joined `loan` row, never the event field. Previously the consumer gated on `event.encompassLoanId`, which was always null, so every push was dropped.
4. Skips shadow audits â€” when `transaction_record.is_shadow`, it logs `"Shadow audit; skipping document push"` and consumes the message (shadow audits never touch Encompass). See [Shadow audits](/integrations/shadow-audits).
5. Drops the message when the resolved loan has no `encompass_loan_id` (no Encompass destination).
6. Applies the per-lender enablement gate (LOA-1336): it resolves the lender from `transaction_record.client_id` and checks `EpcClientRepository.isLenderFeatureEnabled(lenderId, "document_push")` against the [`lender_feature` table](/integrations/database-schema) **before** any materialize work. Default off (no row); a not-enabled lender consumes the message. Enabled in INT.
7. Calls `DocumentMaterializerService.materialize` (carve splits â†’ S3 â†’ persist to `materialized_split_record` â†’ push into the Encompass eFolder). `parseS3Key` strips an `s3://bucket/` prefix so bare keys reach the S3 SDK unmodified.
8. On a non-empty result, publishes the `documents.materialized` return leg (below) so loanlight-api can record where each split landed.

## Events published by encompass

`EventPublishingService` uses `publishEventToEventBridge` from `@loanlight-engineering/shared-cdk`, gated by `DEPLOYED_ENVIRONMENT` plus `PUBLISH_EVENTS_LOCALLY`.

### Audit Requested

The handoff to the AUS. See [Audit lifecycle](/integrations/audit-lifecycle).

```json
{
  "Source": "loanlight.encompass",
  "DetailType": "Audit Requested",
  "Detail": {
    "correlationId": "...",
    "requestorId": "...",
    "environment": "...",
    "s3Bucket": "...",
    "s3Prefix": "...",
    "loanMetadata": {},
    "requestingParty": {},
    "documents": [],
    "additionalData": {},
    "timestamp": "..."
  }
}
```

<Info>
On the loanlight-api side, this event lands on the AUS's own `audit-requested-events` SQS queue, which `IntegrationOrchestrationService.processAuditRequest` drains. The `Audit Requested` detail-type and `audit-requested-events` queue are the canonical production entry path into the audit pipeline.
</Info>

### workflow:error

Published by encompass (not only the AUS) when `processTransaction` fails after the `transactionStateId` is set, so the UI reflects the failure.

```json
{
  "Source": "loanlight.aus",
  "DetailType": "workflow:error",
  "Detail": {
    "correlationId": "...",
    "auditRunId": "...",
    "environment": "...",
    "error": "...",
    "errorCode": "TRANSACTION_PROCESSING_FAILED",
    "errorStep": "transaction_processing",
    "errorTimestamp": "...",
    "timestamp": "..."
  }
}
```

### documents.materialized

The return leg of the document-push pipeline (LOA-1337). After `DocumentMaterializerService` carves and stores the split documents, `EventPublishingService.publishDocumentsMaterialized` publishes this event so loanlight-api can persist each split's storage path.

```json
{
  "Source": "loanlight.encompass",
  "DetailType": "loanlight.documents.materialized",
  "Detail": {
    "type": "loanlight.documents.materialized",
    "loanId": "...",
    "auditRunId": "...",
    "documents": []
  }
}
```

The `DetailType` is the shared-types constant `DOCUMENTS_MATERIALIZED_TYPE`. On the loanlight-api side an EventBridge rule routes it to the dedicated SQS queue `documents-materialized-events`, which loanlight-api consumes (it is the only consumer of this event).

For the consuming side and the audit flow these events drive, see [Audit lifecycle](/integrations/audit-lifecycle) and [Orchestration pipeline](/integrations/orchestration-pipeline).

### LenderEventRule (LOA-1015)

Matches source `["loanlight.aus"]` and detail-types `["LenderUpserted", "LenderClientLinked"]`, routing to the `lender-event` queue. These events are emitted by loanlight-api when WorkOS provisions or links a lender organisation.

`QueueConsumerService.handleLenderEvent` dispatches on the `detail-type`:

- **`LenderUpserted`** â€” `LenderRepository.upsertLenderByWorkosOrgId` â€” creates or updates the `lender` row, matching in priority order: `workos_org_id` â†’ `api_lender_id` â†’ unambiguous name. The `api_lender_id` secondary key (migration `023_add_lender_api_lender_id`) was added as a follow-up because the `workos_org_id`-only lookup is order-dependent: if `LenderUpserted` arrives before `LenderClientLinked` for a legacy lender (no `workos_org_id`), a new duplicate row would be inserted instead of adopting the existing one.
- **`LenderClientLinked`** â€” `LenderRepository.linkClient` â€” associates a `client` row to the resolved lender row, idempotently.

Both event schemas (`LenderUpsertedEventSchema`, `LenderClientLinkedEventSchema`) are validated with Zod against the contract types in `@loanlight-engineering/shared-types`. An invalid payload is logged and the message is re-thrown (routes to DLQ). Unrecognised `detail-type` values are dropped (logged, not re-thrown).
