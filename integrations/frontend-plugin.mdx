`frontend` (workspace package `frontend`, in `packages/frontend/`) is the live Encompass plug-in UI — the screen loan officers see inside Encompass when they run and review a LoanLight audit. It runs as an Encompass SSF (Smart Server Framework) embedded app via `@elliemae/em-ssf-guest`.

<Warning>
The legacy `packages/web-app/` frontend was **deleted** in commit `5b65c1b` (LOA-649, PR #365). `git ls-files packages/web-app` returns nothing; only stale build artifacts remain on disk. `frontend/` is the one and only live frontend. Any doc or system prompt that still references `web-app` as a frontend is stale.
</Warning>

## Stack

- Next.js **16.1.1**, React **19.2.3**, dev on **port 8000** (`next dev -p 8000`).
- Tailwind v4, Storybook 9.
- Deployed as a **static export to S3 + CloudFront** (build → `dist`; the `infra/` CDK uploads it). It is a pure client — there are no Next API routes (`route.ts`).
- State via `zustand` (`store/useEllimae.ts`, `store/useAudit.ts`).
- Datadog browser RUM/logs + PostHog for analytics, `pdfjs-dist` for in-app PDF viewing, Playwright for E2E.
- Consumes `@loanlight-engineering/shared-types` (`^1.13.0`) and `@loanlight-engineering/shared-ui` (`^0.8.0`).

## The two backends it talks to

The frontend calls **both** backends directly.

<Tabs>
<Tab title="encompass service">
Via `encompass.service.ts` / `epc-api-service.ts`. Bootstraps with `POST /integrations/origin`, then polls or streams `GET /integrations/audit-details/:id[/stream]` (`audit-progress.service.ts` handles the SSE), lists and streams the result PDF via `/integrations/documents/...` (`document.service.ts`), and opens the rich pop-out results view via `/integrations/audits/:auditId/external-url` (`external-results.service.ts`). See the [HTTP API](/integrations/http-api).
</Tab>
<Tab title="loanlight-api directly">
Via `audit-api.service.ts`, against `NEXT_PUBLIC_LL_WEB_AUS_API_HOST`. It calls `GET /api/v1/audit_runs/{id}/results?includeNonIssues=true`, `/property_data`, `/urla_completeness`, `getCreditData` → `GET /api/v1/audit_runs/{id}/credit`, and `POST /bug-reports`. Requests send `X-Requestor-ID` plus a bearer token.
</Tab>
</Tabs>

## Credit and liabilities

The `/audit/credit-liabilities` route renders the shared `CreditLiabilitiesView` from `@loanlight-engineering/shared-ui/audit`, fed by `AuditApiService.getCreditData` (`GET /api/v1/audit_runs/{id}/credit`). The sidebar nav item and the Overview card are shown when `creditData` is a non-empty array (LOA-1722: the `credit-liabilities` PostHog flag was removed; the section now gates purely on data presence).

The sidebar (`MergedAuditSidebar`) lists, in order: **Overview**, **Data Consistency**, **Document Health**, **Properties Check** (flagged on `properties-check`), **Credit & Liabilities** (data-presence gate: `!!creditData?.length`), **Appraisal** (data-presence gate: `appraisalData != null`). Credit & Liabilities moved after Properties Check. The 1003-check is a route but not a sidebar item.

The Document Health card on the Overview route now displays the **estimated closing date** when `auditData.summary.estimatedClosingDate` is present (LOA-1875). The date is parsed from `YYYY-MM-DD` using the numeric constructor (local time, no UTC day-shift) and validated against calendar roll-over before rendering; an invalid or absent date renders nothing. The integrations frontend plugin reads this from the same `audit_runs.summary` payload the portal uses, so the two UIs stay in sync.

<Note>
LOA-1722 removed the `appraisal` and `credit-liabilities` PostHog feature flags. Both sections are now gated on whether an agent run exists for the loan — entitlement is enforced server-side at run creation, so an unentitled lender never gets a run and the section stays hidden without a client-side flag.
</Note>

<Note>
The credit issue count on the sidebar badge and the Overview credit card is **FAIL-only** (`countCreditFindings` counts `outcome === "FAIL"`; `WARNING`, `NEEDS_INFO`, `PASS`, and `SKIPPED` are excluded). This matches the portal's credit issue count. The detail `CreditLiabilitiesView` still lists FAIL and WARNING in its own findings.
</Note>

## Component conventions

From the package `CLAUDE.md`:

- Catalyst components live in `components/ui`.
- Route-local code is co-located: `_components/` (JSX) + `_helpers/` (pure functions).
- Every new component needs a Storybook story plus a gallery entry.
- Design tokens only — the Figma → Tailwind mapping is `accent` → `brand` and `grays` → `zinc`.
- SVGs are imported through the `assets/svg/index.ts` barrel.

For the audit data shapes the frontend renders, see [Audit lifecycle](/integrations/audit-lifecycle) and [Integrations database schema](/integrations/database-schema).
