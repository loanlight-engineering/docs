The agent platform is the infrastructure that decides which LangGraph agents an audit should run, records what each agent did, and drives the container audit run to completion. It is intentionally separate from the agents' business logic: this layer is about registration, routing, gating, sequencing, recording, and reconciliation.

It has four moving parts: a **DB-backed catalog** of agents and their dependencies, a **pure-functional router** that turns a request into a plan, an additive **`agent_runs` ledger** that records one row per (audit run, agent), and a **reconciler** that advances ledger rows and completes the container.

<Warning>
The router does **not** yet own agent selection or execution by default. In production the platform runs in **observer mode**: the legacy fan-out still decides what runs ("run all"), and the platform only backfills ledger rows. Router ownership is gated by per-lender feature flags that default OFF. Do not document the router as selecting agents in production unless a lender flag is on. See [Feature flags and rollout](/agents/feature-flags-and-rollout).
</Warning>

## The platform classes

All six classes are provided and exported by `LangGraphModule` (`langgraph/langgraph.module.ts`). Five live in `langgraph/agents/`; `GraphRegistryService` lives at `langgraph/graph-registry.service.ts`.

| Class | File | Role |
|---|---|---|
| `GraphRegistryService` | `langgraph/graph-registry.service.ts` | In-memory registry of LangGraph build factories (lazy import + memoized compile). |
| `AgentRegistryService` | `langgraph/agents/agent-registry.service.ts` | Loads the DB catalog + dependency DAG; computes the required closure; validates the DAG at boot. |
| `AgentRouterService` | `langgraph/agents/agent-router.service.ts` | Pure planner (`buildPlan` / `buildFlatPlan`) + transactional persistence (`route`, `instrumentExistingRun`) + parity harness (`diffPlanVsActual`). |
| `AgentRunReconciler` | `langgraph/agents/agent-run-reconciler.ts` | `agent_runs` row lifecycle: `flip`, `ensure`, `reopen`, `tryCompleteAuditRun`. |
| `AgentRunLedgerObserver` | `langgraph/agents/agent-run-ledger-observer.service.ts` | Env-flag-gated, best-effort wrapper over the reconciler for observer-mode ledger writes. |
| flag helpers | `langgraph/agents/agent-router-flags.ts` | Per-lender feature-flag keys + the fail-closed reader for router-ownership flips. |

## Two-phase rollout

The platform is rolling out in two phases. The phase a given lender and path is in depends on which flags are set.

<Steps>
  <Step title="Observer mode (live default)">
    The router does not own selection. The live EPC / pre-audit fan-out still decides what runs. When the env flag `AGENT_RUNS_LEDGER_ENABLED` is `true`, the `AgentRunLedgerObserver` backfills per-agent `agent_runs` rows as agents settle, so the ledger reflects reality with zero change to which agents run. Phase-0 foundation rows are terminalized at the pre-audit barrier so a later container-completion check does not hang.
  </Step>
  <Step title="Router ownership (per-lender, default OFF)">
    Per-lender feature flags let `AgentRouterService` own selection on a path-by-path basis (EPC, LOS, partner, appraisal/portal). When ON, the router computes the entitled required-closure, writes the ledger plus an `audit_runs.agent_config` compatibility bridge, and the existing executor honors that selection. Every flip is fail-closed and reversible.
  </Step>
</Steps>

## What the router does and does not do

<Note>
`buildPlan` / `route` produce a `PipelineBlueprint`, but the router does **not** execute phases. Handing a blueprint to a phase executor is future work. The existing `AuditAgentsExecutor` still drives execution, honoring `audit_runs.agent_config` (the compatibility bridge), not the blueprint. `agent_config` is a bridge, not the source of truth — the durable model is the blueprint plus the `agent_runs` ledger.
</Note>

## Status vocabulary

Both the container (`audit_runs.status`) and the ledger (`agent_runs.status`) use the value `complete`, never `completed`. The CHECK constraints pin this.

- `agent_runs.status`: `pending`, `processing`, `complete`, `failed`, `skipped`. Terminal: `complete`, `failed`, `skipped`.
- `audit_runs.status`: `pending`, `processing`, `complete`, `failed`.

<Warning>
The string is `complete`. The appraisal pipeline separately uses `completed` for its own `appraisal_runs.status` — that is a different table. The reconciler only ever touches the container via `AuditService.completeAuditRun`, which emits `complete`.
</Warning>

## Boot-time drift guard

`AgentRegistryService.onModuleInit` fails the app boot if the code and the DB catalog diverge. Every non-hyphenated registered graph name and every `EXECUTOR_AGENT_KEYS` literal must map to an **active** catalog row, the DAG must be acyclic, and every dependency edge must point to a strictly lower phase. This keeps the seed and the code in sync. See [Agent catalog and dependency DAG](/agents/catalog-and-dependencies).

## Observability

Every LangGraph node across every agent is wrapped in a dd-trace span by central instrumentation, with a stall watchdog and a LangSmith PII anonymizer. See [LangGraph observability](/agents/observability), [Datadog](/ops/datadog), and [LangSmith](/ops/langsmith).
