The agent platform is the infrastructure that decides which agents an audit should run, runs them in dependency order, records what each did, and drives the audit run to completion. It is separate from the agents' business logic: this layer is registration, routing, gating, sequencing, execution, and recording.

Its organizing principle is an **audit-centric primitive**. `audit_runs` is now lender-anchored and loan-optional, so every channel — EPC, LOS, partner, portal — drives one universal, **channel-blind** orchestrator over the same primitive, with a per-lender agent catalog and entitlements deciding what runs.

## Catalog in the database, structure in code

An agent's definition is split. The `agents` table is **identity only** — display name, billable, active — keyed by the natural string `agent_type`. The pipeline **structure** — which phase an agent runs in, what it depends on, how it executes, and whether it reads the indexed pipeline — lives in code in `AGENT_PIPELINE` (`langgraph/agents/agent-pipeline.ts`). `AgentRegistryService` joins them at boot.

<Note>
There is no `agent_dependencies` table, and `agents` has no `kind`, `audit_phase`, or `requires_llama_indexing` column. Changing where an agent runs or its dependencies is a code edit; only adding/retiring an agent or toggling its attributes touches the database. Earlier drafts of these docs described a DB-stored DAG and those columns — they no longer exist.
</Note>

## Channel-blind core, adapter-owned plan edges

The core — `AgentRouterService`, `AgentOrchestrator`, `AgentDispatcher`, `AgentRunReconciler` — has zero `triggerSource` branches. The orchestrator takes an `ExecutionPlan` and an `ExecutionContext` and dispatches agents by identity. Each per-channel adapter owns its own **plan edges**: entitlement resolution, phase stripping (`deriveOrchestratorPlan`), whether it owns indexing at audit time (`ctx.skipIndexing`), and the final completion. So this is a channel-blind core plus adapter-owned edges, not a single end-to-end primitive.

## The platform classes

The execution classes are provided by `LangGraphModule` (`langgraph/langgraph.module.ts`); the entitlement-management service lives in `AgentPlatformModule`. `IndexingService` is provided by `IndexingModule`.

| Class | File | Role |
|---|---|---|
| `AgentRegistryService` | `langgraph/agents/agent-registry.service.ts` | Joins the DB catalog with the `AGENT_PIPELINE` code map; computes the required closure; validates the DAG at boot. |
| `AgentRouterService` | `langgraph/agents/agent-router.service.ts` | Pure planner (`buildPlan` / `buildFlatPlan`) + transactional persistence (`route`, `recordAgentRunsForExistingAudit`). |
| `AgentOrchestrator` | `langgraph/orchestrator/agent-orchestrator.service.ts` | The execution control flow: phase-by-phase dispatch, ledger-observed barriers, the phase-0 indexing gate, the standalone lane. |
| `AgentDispatcher` | `langgraph/orchestrator/agent-dispatcher.ts` | Dispatches one agent by its runtime — enqueue `queue_worker` agents, run `inproc_graph` agents by identity. |
| `AgentRunReconciler` | `langgraph/agents/agent-run-reconciler.ts` | `agent_runs` / `audit_infra_steps` lifecycle, `tryCompleteAuditRun`, the phase-barrier reads, and the orphan reapers. |
| `IndexingService` | `integration-events/services/indexing.service.ts` | The phase-0 `llama_index` infra step + classify-readiness gate. |
| `AgentEntitlementService` | `agent-platform/agent-entitlement.service.ts` | Reads/writes per-lender entitlement intents; serves Octy's management UI. |

Supporting providers: `QueueAgentRegistry` (worker-agent enqueue adapters), `AuditErrorSinkService` and `CreditDocumentLoaderService` (injected directly by the dispatcher), `AgentRunOrphanReaperService` and `AuditInfraStepOrphanReaperService`, and `GraphRegistryService` (LangGraph build factories).

## How a run flows

<Steps>
  <Step title="Route">
    The channel calls `route` (EPC/partner/portal) or `recordAgentRunsForExistingAudit` (LOS). The router filters by entitlements, expands the required closure, annotates data-gate skips, allocates the `audit_runs` row, and seeds the `agent_runs` ledger (`pending`/`skipped`) plus the `agent_config` bridge.
  </Step>
  <Step title="Orchestrate">
    The channel builds an `ExecutionContext` and calls `orchestrator.run(auditRunId, executionPlan, ctx, hooks?)`. Phase 0 may run a `llama_index` infra step concurrently with the classify/URLA agents. Each phase's agents fire concurrently; a ledger barrier gates advancement.
  </Step>
  <Step title="Dispatch + own status">
    The dispatcher runs each agent by identity. Each agent owns its own `processing → complete/failed` transition; the orchestrator only writes `skipped` for a dependency-gated agent. Producers persist a cross-phase handoff to `agent_runs.output`.
  </Step>
  <Step title="Complete">
    `tryCompleteAuditRun` completes the container once every expected agent (per `agent_config`) and every infra step is terminal. After `run()` returns, the channel issues the final guarded `completeAuditRun`.
  </Step>
</Steps>

## Status vocabulary

Both the container (`audit_runs.status`) and the ledger (`agent_runs.status`) use the value `complete`, never `completed`. The CHECK constraints pin this.

- `agent_runs.status`: `pending`, `processing`, `complete`, `failed`, `skipped`. Terminal: `complete`, `failed`, `skipped`.
- `audit_infra_steps.status`: `pending`, `processing`, `complete`, `failed` (no `skipped`).
- `audit_runs.status`: `pending`, `processing`, `complete`, `failed`.

<Warning>
The string is `complete`. The appraisal pipeline separately uses `completed` for `appraisal_runs.status` — a different table. The reconciler only ever touches the container via `AuditService.completeAuditRun`, which emits `complete`.
</Warning>

## Boot-time drift guard

`AgentRegistryService.onModuleInit` fails the app boot if the code and the DB catalog diverge. Every non-hyphenated registered graph name and every `EXECUTOR_AGENT_KEYS` literal must map to an **active** catalog row, the dependency DAG must be acyclic, and every edge must point to a strictly lower phase. See [Agent catalog and pipeline](/agents/catalog-and-dependencies).

## Observability

Every LangGraph node across every agent is wrapped in a dd-trace span by central instrumentation, with a stall watchdog and a LangSmith PII anonymizer. See [LangGraph observability](/agents/observability), [Datadog](/ops/datadog), and [LangSmith](/ops/langsmith).
