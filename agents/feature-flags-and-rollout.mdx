The agent platform shipped as the **only** execution path. There is no observer mode and no per-lender router feature flag — every channel drives the orchestrator directly. The rollout risk is not a runtime toggle; it is the schema migration that makes `audit_runs` lender-anchored, which has to land on a database full of loan-anchored rows. That is handled by an EXPAND → backfill → CONTRACT deploy order.

<Note>
Earlier drafts of this page described an `AGENT_RUNS_LEDGER_ENABLED` env var and four per-lender flags (`agent_router_epc_enabled`, `agent_router_los_enabled`, `PARTNER_VIA_ROUTER`, `APPRAISAL_VIA_ROUTER`). Those flags, the observer-mode ledger writer, and the parity harness were removed when the orchestrator became the default path. The catalog and pipeline are still gated at boot by the [drift guard](/agents/catalog-and-dependencies), not by flags.
</Note>

## The deploy order

`audit_runs.lender_id` and `trigger_source` are load-bearing (NOT NULL) at the end state, but a database with existing rows has them NULL until they are populated. A single SET NOT NULL would fail. The migration is therefore split, and the deploy applies all three steps in one `cdk deploy`:

<Steps>
  <Step title="EXPAND — 0202_audit_runs_lender_anchor.sql">
    Adds `lender_id`, `loan_program_id`, and `trigger_source` as **nullable** columns with their FKs and CHECKs, relaxes `loan_id`/`run_number` to nullable, swaps the unique index to the partial `audit_runs_loan_run_key`, and best-effort backfills `lender_id` from the loan's tenant and `trigger_source` from existing signals. Behaviorally inert — existing writers keep working.
  </Step>
  <Step title="Schema — 0203_agent_platform_schema.sql">
    Additively creates `agents`, `agent_runs`, `lender_agent_entitlements`, and `audit_infra_steps`, links `appraisal_runs.audit_run_id`, and seeds the catalog. See [Agent platform schema](/data-model/agent-platform-schema).
  </Step>
  <Step title="Backfill — scripts/backfill-agent-platform.ts">
    Populates `lender_id` on every legacy row (the EXPAND backfill covers only loan-bound rows) and seeds the platform tables. Run **inline** by the DbMigrate Lambda immediately before the CONTRACT migration.
  </Step>
  <Step title="CONTRACT — 0204_audit_runs_tighten_not_null.sql">
    A gate that `RAISE EXCEPTION`s if any `audit_runs.lender_id` is still NULL, then tightens `lender_id` and `trigger_source` to NOT NULL via the `NOT VALID → VALIDATE → SET NOT NULL → DROP CONSTRAINT` pattern. On a fresh database with zero rows the gate trivially passes.
  </Step>
</Steps>

<Warning>
The order is load-bearing: the CONTRACT gate only passes once the backfill has populated `lender_id` on every row. The deploy must run EXPAND + schema → backfill → CONTRACT.
</Warning>

### Inline, retry-safe backfill

`infra/lambda/db-migrate/index.ts` runs `runBackfill()` in-line, keyed off the migration tag suffix `_audit_runs_tighten_not_null` (`isBackfillGateMigration`), so it stays correct if migrations are renumbered. Because the gate migration stays pending until it succeeds, every deploy attempt re-runs the idempotent backfill before re-attempting the gate. The DbMigrate Lambda timeout was raised 5 → 15 minutes and memory 256 → 512 MB for the prod-scale backfill, and an `infra/tsconfig` `@/*` alias lets the backfill's schema barrel resolve.

## What the backfill does

`scripts/backfill-agent-platform.ts` is idempotent (inserts use `onConflictDoNothing`; updates are guarded by already-terminal / canonical-set predicates) and runs these parts in order:

<Steps>
  <Step title="Containerize orphaned standalone runs">
    Mint a parent `audit_run` and link for historical standalone appraisal runs so they flow through the unified model.
  </Step>
  <Step title="Fill lender_id">
    A fallback chain ending in a sentinel tenant (`Legacy / Unattributed Tenant (backfill)`), so the CONTRACT gate holds with zero residual NULLs instead of aborting the deploy. Sentinel-pointed rows are durable data the backfill never auto-reassigns — a warning log flags them for manual reconciliation.
  </Step>
  <Step title="Seed entitlements">
    Seed `lender_agent_entitlements` from each lender's run history plus legacy flags.
  </Step>
  <Step title="Seed the ledger">
    Reconstruct `agent_runs` from the per-agent output tables, plus `failed` rows from the `audit_errors` ledger for agents that failed without leaving an output row.
  </Step>
  <Step title="Align legacy identifiers">
    Canonicalize historical agent identifiers in `octy_issues` JSONB and `audit_errors.agent_name` (e.g. `PII-Validation` → `piiValidation`).
  </Step>
  <Step title="Terminalize orphans">
    Flip historical orphan `agent_runs` (non-terminal under a terminal parent) to `failed`, clearing the never-flipped backlog before the orphan reaper becomes authoritative.
  </Step>
</Steps>

## Entitlement rollout

The backfill seeds **explicit** `lender_agent_entitlements` rows for every historical lender, covering the agents active at backfill time. This matters for how new agents reach the installed base:

- A lender with **no** rows runs the platform default — every active agent (default-ON). See [Agent router and entitlements](/agents/router-and-entitlements).
- Once a lender has explicit rows it is an **allowlist**. Activating a new agent later (`active` false → true) does **not** auto-grant it to backfilled lenders; only un-seeded lenders pick it up via default-ON.
- Granting a newly-activated agent to the installed base requires a follow-up entitlement migration that inserts the new `agent_type` for existing lenders.

## Loanless bug reports

A side effect of the loan-optional primitive: migration `0205_octy_issues_loan_id_nullable.sql` drops NOT NULL on `octy_issues.loan_id` so bug reports can be filed against loanless audit runs (standalone portal appraisal/income). A report's anchor is `audit_run_id` (still NOT NULL); `loan_id` is now optional.
