The execution side of the platform is three pieces: the `agent_runs` and `audit_infra_steps` **ledgers**, the `AgentRunReconciler` that owns their lifecycle and drives container completion, and the `AgentOrchestrator` that runs the [router](/agents/router-and-entitlements)'s execution plan. The orchestrator is **channel-blind** — it takes an `ExecutionPlan` and an `ExecutionContext` and dispatches agents by identity, never branching on the trigger source. Each channel adapter builds the context and owns its own plan edges (entitlement resolution, phase stripping, completion).

## The ledger tables

`agent_runs` records one row per `(audit run, agent)`. `audit_infra_steps` records non-agent infra steps that gate a phase (today only LlamaCloud indexing). Both schemas are in [Agent platform schema](/data-model/agent-platform-schema); the load-bearing columns:

- `agent_runs.status`: `pending`, `processing`, `complete`, `failed`, `skipped`. Terminal: `complete`, `failed`, `skipped`.
- `agent_runs.output` (jsonb): the cross-phase handoff payload a producer agent persists on completion, rehydrated into the next phase.
- `audit_infra_steps.status`: `pending`, `processing`, `complete`, `failed` (no `skipped`).

<Warning>
The success value is `complete`, not `completed`. The CHECK constraints on `agent_runs.status` and `audit_runs.status` pin this. The appraisal pipeline separately uses `completed` for `appraisal_runs.status` — a different table.
</Warning>

## Status ownership

The split of who writes which status is the core invariant:

- The **router** seeds `pending` rows (and `skipped` for data-gated agents) when it records the plan.
- The **orchestrator** writes `skipped` for an agent whose channel-declared dependency was unmet, and never anything else.
- Each **agent** owns its own `processing → complete/failed` transitions inside the dispatcher.
- A **queue_worker** agent's row stays `pending` after enqueue — the background worker owns the rest.

## AgentRunReconciler

File: `langgraph/agents/agent-run-reconciler.ts`. Injects the Drizzle db and `AuditService`. Constants: `TERMINAL_STATUSES = ["complete","failed","skipped"]`, `NON_TERMINAL_STATUSES = ["pending","processing"]`, `WORKER_OWNED_AGENT_TYPES = ["appraisalEvaluation"]`. `FlipOptions = { errors?, output? }`.

<AccordionGroup>
  <Accordion title="setAgentRunStatus(auditRunId, agentType, status, options?)">
    Advance a single row. Sets `started_at = now` on `processing`, `completed_at = now` on a terminal status, plus optional `errors` / `output`. The UPDATE is **guarded** to only advance a non-terminal row (`inArray(status, [pending, processing])`), so re-setting a terminal row is a no-op and a late `complete` cannot stomp a terminal `failed`. Moving terminal → processing is `reopenAgentRun`'s job only.
  </Accordion>
  <Accordion title="ensureAgentRun(auditRunId, agentType)">
    `INSERT ... VALUES(status:'pending') ON CONFLICT (audit_run_id, agent_type) DO NOTHING`. Idempotent; never resets an existing row.
  </Accordion>
  <Accordion title="reopenAgentRun(auditRunId, agentType)">
    Rerun-after-fail. In a transaction, flips the agent row `complete | failed → processing` (clears `completed_at`, guarded so a `skipped` row is left untouched) and flips the container `audit_runs` row `complete | failed → processing`. Used by the portal appraisal re-claim path. (Caveat in code: it reopens the shared container, correct only for a single-agent container.)
  </Accordion>
  <Accordion title="tryCompleteAuditRun(auditRunId): boolean">
    Drives the container terminal once the run's work is done. Reads all `agent_runs` statuses, then applies three guards before calling `auditService.completeAuditRun`:

    - **Vacuous guard:** returns `false` on zero rows — never completes an empty ledger.
    - **Non-terminal guard:** returns `false` if any `agent_runs` row is non-terminal.
    - **Plan-aware guard:** reads the expected agent set from `audit_runs.agent_config` and returns `false` until **every** expected agent has a row — so a phase-0 terminal flip cannot complete a run whose phase-1/2 rows are not seeded yet. Inert when `agent_config` is absent.
    - **Infra-step guard:** returns `false` unless every `audit_infra_steps` row is terminal too.

    An all-failed run still becomes `complete` (failed is terminal). Idempotent.
  </Accordion>
  <Accordion title="isPhaseTerminal / isInfraStepTerminal">
    The barrier reads. `isPhaseTerminal(auditRunId, agentTypes, { infraCapability? })` is true only when every named agent's row is terminal — a missing row counts as not terminal — and (when an `infraCapability` is given) when its infra step is terminal. `isInfraStepTerminal` returns true when the row is absent (no infra step → does not gate) or `complete`/`failed`.
  </Accordion>
  <Accordion title="getAgentRunOutput / setAgentRunOutput / getAgentRunStatus">
    `setAgentRunOutput` persists a producer's handoff to `agent_runs.output` (guarded to non-terminal rows). `getAgentRunOutput` reads it back for rehydration. `getAgentRunStatus` reads one row's status (null when absent) for the dependency-aware skip.
  </Accordion>
</AccordionGroup>

`AuditService.completeAuditRun` sets `status='complete'` and `completed_at`, guarded `inArray(status, [pending, processing])`, and emits the CHECK-pinned `complete`.

## The orchestrator

File: `langgraph/orchestrator/agent-orchestrator.service.ts`. `AgentOrchestrator.run(auditRunId, plan, ctx, hooks?)` executes the plan. Each phase's agents fire concurrently (`Promise.allSettled`) by identity through the dispatcher, then a ledger-observed barrier gates advancement; the standalone lane runs without a barrier.

<Steps>
  <Step title="Pre-flight (assertResolvable)">
    Every planned agent must resolve to a catalog record, or `run()` throws **before** any dispatch — so config drift is a loud failure routed through the channel's error cascade, not a silently-tolerated per-agent settle.
  </Step>
  <Step title="Indexing gate decision">
    `needsIndexing = !ctx.skipIndexing && <any planned agent has requiresIndexing>`. EPC owns indexing at audit time (`skipIndexing=false`); LOS and partner index upstream (`skipIndexing=true`), so the gate no-ops for them even though the agents they run carry `requiresIndexing`.
  </Step>
  <Step title="Per phase">
    For phase 0 with indexing, `indexing.beginInfraStep(auditRunId)` materializes the `llama_index` row **before** any agent can reach `tryCompleteAuditRun`. The phase's agents and `indexing.runIndexStep(...)` run concurrently (indexing uploads raw docs, independent of classification). After the agents settle, the orchestrator awaits the index promise and propagates the resolved `pipelineId` onto `ctx` for later phases.
  </Step>
  <Step title="Fatal-failure surfacing">
    Every per-agent failure is tolerated except a URLA-extraction **deadline**: if a settled result is an `isUrlaExtractionDeadlineError`, `run()` rethrows it so the channel's outer catch fails the audit rather than completing it with degraded downstream findings.
  </Step>
  <Step title="Barrier + classify readiness">
    `awaitPhaseTerminal` polls `isPhaseTerminal` (exponential backoff 250ms → 5s, ceiling 4h) until every phase agent_run row — and, on phase 0, the `llama_index` infra step — is terminal. On phase 0 it then runs `indexing.awaitClassifyReadiness(...)` (the bounded classify-readiness gate).
  </Step>
  <Step title="afterPhase hook + rehydrate">
    The optional `afterPhase(phase, ctx)` hook fires at the boundary (EPC uses it to freeze the LOA-936 snapshot with the resolved pipeline). Then this phase's agents' `output` rows are rehydrated and merged into `upstreamOutput` for the next phase.
  </Step>
  <Step title="Standalone lane">
    After the phases, standalone agents are dispatched with no barrier — a queue worker completes them and drives `tryCompleteAuditRun` on its own.
  </Step>
</Steps>

A wedged phase that exceeds the 4h ceiling throws `PhaseBarrierTimeoutError` out of `run()` into the channel's error cascade. The ceiling deliberately exceeds the agent-run orphan-reaper window (3h) so a genuinely-stuck row is reaped to `failed` (terminal for the barrier) before the ceiling fires.

### Dependency-aware skip

Before dispatching a phase agent, the orchestrator checks `dependenciesUnmet`: an agent the channel lists in `ctx.skipWhenDepsUnmet` whose in-plan `dependsOn` predecessor did not reach `complete` is flipped `skipped` instead of dispatched. An agent absent from that list **runs degraded** on an unmet dependency (undefined inputs — the EPC/partner legacy for `documentHealth`/`redFlags`). LOS sets `skipWhenDepsUnmet: ["redFlags"]` so `redFlags` is skipped when `propertyData` failed, while `documentHealth` still runs. A dependency absent from the plan (no row) does not gate.

## AgentDispatcher

File: `langgraph/orchestrator/agent-dispatcher.ts`. Routes one agent on its **runtime**:

- `queue_worker` → resolves the agent's adapter from `QueueAgentRegistry` and **enqueues** the background job. The row stays `pending`; the worker owns `processing → terminal` and drives completion. A missing adapter or an enqueue throw is recorded as a per-agent failure so the audit can still complete.
- `inproc_graph` → a `switch(agentType)` onto the existing `LangGraphService.create<Agent>…WithSharedAuditRun` methods, wrapped in `owned()`.

`owned()` is the per-agent status lifecycle: flip the row `processing`, run the work, flip `complete`; on a throw flip `failed`, record the `errors` blob, and route the error to `AuditErrorSinkService` (which fans a credit `AggregateError` out to one `audit_errors` row per failed document). It **never rethrows** — except a URLA-extraction deadline, which it rethrows after recording so the orchestrator can fail the audit.

Producer agents persist their cross-phase handoff via `persistHandoff` → `reconciler.setAgentRunOutput`. Only the fields downstream phases consume are extracted: `urlaExtraction` persists `transformedPiiFields`, `addressOverrides`, `loanPurpose`, the URLA view, borrower names, and credit identity; `propertyData` persists `consolidatedSubject`/`consolidatedBorrower` and the `trustIndicator` that `documentHealth` reads as `title_held_in_trust`.

### Collaborators

The dispatcher injects two helpers directly (no DI-token seams): `CreditDocumentLoaderService` (a self-contained S3 client that materializes credit document bytes only inside the credit fan-out — the bytes never ride the context or LangSmith) and `AuditErrorSinkService` (per-agent error recording). Worker-owned adapters self-register into `QueueAgentRegistry` from their feature module (portal appraisal), so `langgraph` never imports the feature module.

## Indexing as an infra step

File: `integration-events/services/indexing.service.ts`. `IndexingService` runs LlamaCloud indexing as a tracked phase-0 infra step (`LLAMA_INDEX_CAPABILITY = "llama_index"`), writing `audit_infra_steps` directly.

- `beginInfraStep(auditRunId)` — idempotent insert + flip to `processing` (re-pends a non-complete row; no-ops a `complete` one, so EPC SQS redelivery re-enters cleanly).
- `runIndexStep(auditRunId, ctx)` — `getOrCreatePipeline` + association + `uploadAndWaitForDocuments` (the poller-readiness wait) with the single `PIPELINE_NOT_FOUND` retry. On success flips the row `complete` and returns the resolved `pipelineId`; on any throw flips it `failed` (ledger-only) and rethrows into the channel's error cascade.
- `awaitClassifyReadiness(auditRunId, ctx)` — the bounded classify-readiness gate. On deadline it runs the inline `saveAuditError → failAuditRun → WORKFLOW.ERROR` cascade (each its own try/catch) and `markCascadeHandled` so the channel outer catch does not double-cascade.

## Orphan reapers

Three reapers (boot reap + unref'd `setInterval`, 5-minute cadence) keep the ledger from deadlocking on dead workers and stuck completions. The `agent_runs` and `audit_infra_steps` reapers join on the parent `audit_runs` status, so they only unwedge runs that are themselves non-terminal — a no-op in steady state and storm-proof against a `pending` backlog under already-completed parents.

| Reaper | File | Window | Reaps |
|---|---|---|---|
| `AgentRunOrphanReaperService` | `orchestrator/agent-run-orphan-reaper.service.ts` | 3h | Stuck non-terminal `agent_runs` rows → `failed` (skips `WORKER_OWNED_AGENT_TYPES`). |
| `AuditInfraStepOrphanReaperService` | `orchestrator/audit-infra-step-orphan-reaper.service.ts` | 70min | Stuck non-terminal `audit_infra_steps` rows → `failed`. |
| `AuditCompletionReaperService` | `orchestrator/audit-completion-reaper.service.ts` | 60min | `audit_runs` stuck non-terminal after **all** their `agent_runs` and `audit_infra_steps` are already terminal — the one case the other two reapers cannot cover (LOA-1752, #2719). Calls `reconciler.reapCompletableAuditRuns(windowMs)`. |

The work itself is in `reconciler.reapStuckAgentRuns(windowMs)` and `reconciler.reapStuckInfraSteps(windowMs)`; both drive each affected run toward completion after reaping. Each window must exceed the slowest legitimate work it covers (credit fan-out for agents; the 57-minute index ceiling for infra steps) so a live-but-slow operation is never false-failed.

## How channels wire it

Every channel resolves a plan, builds the context, and calls `orchestrator.run`. After `run()` returns (every phase barrier already awaited), the channel calls `completeAuditRun` for the final completion.

| Channel | Plan source | Context flags | Notes |
|---|---|---|---|
| EPC | `route` (`expandClosure` default) | `skipIndexing=false`, `indexingInputs` set | Owns phase 0 (classify/URLA + `llama_index`); `afterPhase(0)` freezes the snapshot. |
| Partner | `route` | `skipIndexing=true` | `deriveOrchestratorPlan({ stripPhase0: true })` strips `preAuditClassification` (classified upstream); `urlaExtraction` still runs. |
| LOS | `recordAgentRunsForExistingAudit` (`expandClosure:false`) | `skipIndexing=true`, `skipWhenDepsUnmet:["redFlags"]` | Flat fixed suite, no phase-0 closure. |
| Portal appraisal | `route` (`requestedAgents:["appraisalEvaluation"]`) | standalone lane | Worker-owned; the `AppraisalQueueAdapter` enqueues the extraction job. |

When a channel strips agents from execution that the router already seeded (partner's `preAuditClassification`, or EPC under `SKIP_LLAMA_INGESTION`), `markStrippedAgentsSkipped` terminalizes those rows `skipped` so no phase-0 row lingers `pending` and pollutes ledger projections.
