An agent's definition is split across two places. The `agents` table is the **catalog** — identity and attributes only (display name, billable, active), keyed by a natural string `agent_type`. The **pipeline structure** — which phase each agent runs in, what it depends on, how it executes, and whether it reads the indexed pipeline — lives in code in the `AGENT_PIPELINE` map (`langgraph/agents/agent-pipeline.ts`). `AgentRegistryService` joins the two at boot: identity from the row, placement from the map.

This split is deliberate. Changing where an agent runs or its dependencies is a **code edit**, never a migration; only adding/retiring an agent or toggling its `active`/`is_billable` attributes touches the database.

<Note>
There is no `agent_dependencies` table and the `agents` table has no `kind`, `audit_phase`, or `requires_llama_indexing` column. The dependency DAG and phase placement are code (`AGENT_PIPELINE`). The schema is in [Agent platform schema](/data-model/agent-platform-schema).
</Note>

## `agents` — the catalog

File: `database/schema/agents.ts`. The natural string PK `agent_type` is shared by the child tables (`agent_runs`, `lender_agent_entitlements`).

| Column | Type | Notes |
|---|---|---|
| `agent_type` | `varchar(64)` PK | Natural key, e.g. `piiValidation`. |
| `display_name` | `text` NOT NULL | Human label, e.g. `PII Validation`. |
| `is_billable` | `boolean` NOT NULL | Default `false`. |
| `active` | `boolean` NOT NULL | Default `true`. Inactive agents are excluded from closure and selection. |
| `created_at`, `updated_at` | `timestamptz` NOT NULL | Default `now()`. |

The catalog is seeded by migration `0203_agent_platform_schema.sql` (`ON CONFLICT DO NOTHING`, idempotent).

## `AGENT_PIPELINE` — the pipeline structure

File: `langgraph/agents/agent-pipeline.ts`. `AGENT_PIPELINE` is a `Record<string, AgentPipelineEntry>` keyed by `agent_type`. Each entry carries:

| Field | Type | Meaning |
|---|---|---|
| `phase` | `number \| null` | `0`/`1`/`2` = a phased member (run in ascending phase order); `null` = the standalone lane. |
| `dependsOn` | `string[]` | Agents that must be terminal before this one runs. Each must be a strictly-lower phase. |
| `runtime` | `"inproc_graph" \| "queue_worker"` | How the agent executes. Default `inproc_graph`. Orthogonal to phase placement. |
| `requiresIndexing` | `boolean` | True when the agent reads the indexed LlamaCloud pipeline. Drives the orchestrator's phase-0 indexing gate. Default `false`. |

`runtime` and `phase` are independent axes: a `queue_worker` agent can be phased, an `inproc_graph` agent can be standalone. The [dispatcher](/agents/runs-ledger) routes on `runtime`; the [router](/agents/router-and-entitlements) groups and orders on `phase`/`dependsOn`.

### The pipeline

This is the current `AGENT_PIPELINE`. `requiresIndexing` agents are the ones whose graphs query the indexed pipeline; the others read MISMO XML or upstream output.

| `agent_type` | `phase` | `dependsOn` | `runtime` | `requiresIndexing` |
|---|---|---|---|---|
| `preAuditClassification` | 0 | — | `inproc_graph` | `true` |
| `urlaExtraction` | 0 | — | `inproc_graph` | `false` |
| `piiValidation` | 1 | `preAuditClassification`, `urlaExtraction` | `inproc_graph` | `true` |
| `propertyData` | 1 | `preAuditClassification`, `urlaExtraction` | `inproc_graph` | `true` |
| `creditEvaluation` | 1 | `preAuditClassification`, `urlaExtraction` | `inproc_graph` | `false` |
| `documentHealth` | 2 | `preAuditClassification`, `propertyData`, `urlaExtraction` | `inproc_graph` | `true` |
| `redFlags` | 2 | `propertyData`, `urlaExtraction` | `inproc_graph` | `false` |
| `appraisalEvaluation` | standalone | — | `queue_worker` | `false` |
| `liabilitiesAssessment` | standalone | — | `inproc_graph` | `false` |

<Note>
`preAuditClassification` carries `requiresIndexing` because classify drives the LlamaCloud upload of the raw documents. `creditEvaluation` loads its document bytes from S3 (not the indexed pipeline), and `redFlags` consumes `propertyData` output — so neither reads the index. The catalog keys here are the identity that flows into `agent_runs.agent_type`; the LangSmith run-name labels are a separate presentation form (e.g. `piiValidation` → `PII-Validation`) — see [observability](/agents/observability).
</Note>

### Catalog notes and traps

<AccordionGroup>
  <Accordion title="liabilitiesAssessment is seeded inactive">
    `liabilitiesAssessment` is seeded with `active=false` and no phase. Because it is inactive it never appears in a required closure or a plan. Liabilities and DTI live inside the fully-implemented `creditEvaluation` agent. See [Liabilities assessment](/agents/liabilities-assessment).
  </Accordion>
  <Accordion title="urlaCompletenessRulesEngine is NOT a catalog agent">
    The URLA completeness rules engine is deliberately absent from the catalog and `AGENT_PIPELINE`. It is an embedded subgraph of `urlaExtraction` and is never scheduled or billed independently. See [URLA completeness](/agents/urla-completeness).
  </Accordion>
  <Accordion title="appraisalEvaluation is the only queue_worker agent">
    `appraisalEvaluation` runs as a `queue_worker`: the dispatcher enqueues a BullMQ job and the background worker owns the `agent_runs` lifecycle. It is standalone (no phase, no edges). See [Appraisal evaluation](/agents/appraisal-evaluation).
  </Accordion>
  <Accordion title="creditEvaluation is fully implemented">
    Despite older descriptions calling it a stub, `creditEvaluation` is a fully-implemented agent, seeded active and billable. See [Credit evaluation](/agents/credit-evaluation).
  </Accordion>
</AccordionGroup>

## Boot-time loading and validation

`AgentRegistryService` (`langgraph/agents/agent-registry.service.ts`) implements `OnModuleInit`. At boot it:

<Steps>
  <Step title="load()">
    Select all rows from `agents`. For each row it builds an `AgentRecord` — identity from the row, plus `auditPhase` / `requiresIndexing` / `runtime` looked up from `AGENT_PIPELINE` (defaults: `phase` null, `requiresIndexing` false, `runtime` `inproc_graph`). The dependency `edges` and `adjList` are derived from each entry's `dependsOn`.
  </Step>
  <Step title="assertNoCycle()">
    Kahn's algorithm over the edges; throws if the dependency graph has a cycle.
  </Step>
  <Step title="assertEdgesPointToLowerPhase()">
    Every edge's source phase must be a non-null integer strictly greater than the target's phase. Both endpoints must have non-null phases.
  </Step>
  <Step title="assertNoDrift()">
    Every non-hyphenated registered graph name (from `GraphRegistryService.getRegisteredGraphs()`) and every `EXECUTOR_AGENT_KEYS` entry must be an **active** catalog row, or boot throws. This keeps the seed and the code in sync.
  </Step>
</Steps>

`EXECUTOR_AGENT_KEYS` is a hard-coded static used by the drift guard:

```ts
["piiValidation", "documentHealth", "urlaExtraction", "propertyData", "redFlags", "creditEvaluation"]
```

### Registry accessors

The registry exposes the catalog joined with the code map. The router, orchestrator, and the entitlement service read it rather than touching `AGENT_PIPELINE` directly.

| Method | Returns |
|---|---|
| `getRecord(agentType)` | The full `AgentRecord` (identity + placement + runtime), or `undefined`. |
| `getActive()` | All active records. |
| `getAllRecords()` | Every record, active and inactive (for admin views). |
| `getEdges()` | The full dependency edge list (`{ agentType, dependsOn }` per `dependsOn`). |
| `getDirectDependencies(agentType)` | The direct `dependsOn` predecessors (for the orchestrator's dependency-aware skip). |
| `byPhase(phase)` | Active records in a given phase. |
| `requiredClosure(requested)` | The transitive required closure (below). |

## The required closure

`requiredClosure(requested: string[]): string[]` is the keystone the router uses. It computes the transitive required closure of the requested agents:

- Standalone agents (`phase === null`) and unknowns contribute nothing — they have no edges.
- A DFS (`collectDeps`) collects each agent plus all transitive `dependsOn`.
- The result is filtered to **active** rows only.
- It is sorted phase-ascending (phase 0 first; NULL sorts to Infinity), with a `localeCompare` tiebreak.

For example, `requiredClosure(["redFlags"])` returns `["preAuditClassification", "urlaExtraction", "propertyData", "redFlags"]`. The router consumes this closure when it builds a plan — see [Agent router and entitlements](/agents/router-and-entitlements).
