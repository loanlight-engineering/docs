The Appraisal-Evaluation agent (`appraisalEvaluation`) is a fully implemented, 12-node compiled
LangGraph that extracts and enriches appraisal data. Unlike
[Credit-Evaluation](/agents/credit-evaluation), rule evaluation does **not** run inside the
graph â€” it runs in the processor, after the graph returns.

<Note>
The graph performs extraction and enrichment only. Its output type
(`AppraisalEvaluationGraphOutput`) carries no `complianceResults`. The processor
(`appraisal.processor.ts`) re-runs `evaluateWithCustomRules` with its dependency-injected DB and
persists results. `LangGraphService.createAppraisalEvaluationGraph` does **not** persist
`appraisal_runs` â€” the processor does. This is a deliberate divergence from the credit agent,
whose `aggregate` node persists.
</Note>

## How it runs

Runs are created via `POST /api/v1/portal/appraisals/run` (`PortalAppraisalController`, base
path `api/v1/portal/appraisals`). The DTO `CreateRunDto` carries `s3Key` (required), optional
`xmlS3Key`, `loanProgramId`, `loanNumber`, and `agencySources?` (in `fannie_mae | freddie_mac`,
max 2). The processor (`@Processor("appraisal-extraction")`) claims the run with a heartbeat and
`claimToken` (the same recovery pattern as credit), calls
`LangGraphService.createAppraisalEvaluationGraph`, persists the graph-returned `enrichmentData`,
then runs the rules and findings (see below).

### EPC audit path (LOA-1675)

The agent also runs on the EPC audit path, triggered post-completion by
`triggerAppraisalEvaluation` in `integration-orchestration.service.ts`. After `WORKFLOW.COMPLETE`
fires, the orchestrator enumerates classified appraisal documents (`appraisal_standard`,
`desk_review`) from the S3 manifest, copies the appraisal PDF into the portal bucket, mints an
`appraisal_runs` row stamped with `loan_id`, `lender_id`, and `audit_run_id`, and dispatches
the job by identity. The EPC audit completes synchronously; the appraisal job runs as a
detached best-effort task (up to ~89 min) and does not gate `WORKFLOW.COMPLETE`.

Read the EPC-linked run via `GET /api/v1/audit_runs/:id/appraisal` â€” returns `PortalAppraisalPayload`
or `{ data: null }` (never 404). The `AppraisalAuditProjectionService` applies the manual-override
overlay (LOA-1685) to the projection. See [Integration orchestration pipeline](/integrations/orchestration-pipeline)
and [Reading audit results](/pipeline/results).

`LangGraphService.createAppraisalEvaluationGraph` takes an options object, returns the
extraction result plus the graph-owned `enrichmentData`, and races the graph against
`APPRAISAL_GRAPH_DEADLINE_MS`. The builder `buildAppraisalEvaluationGraph` closes over
`pdfBytes`, optional `xmlCandidates`, `onPartial`, `onIteration`, and `inputFilename` â€” these
stay closure-delivered and are never placed in graph state or serialized progress.

## The graph

```text
__start__ â†’ prepare â†’ classify â†’ detectScanned
  â†’ (routeExtractionStrategy) â†’ ocr? â†’ reductoExtract
  â†’ resolveFields â†’ normalizeAdapter â†’ computeCoverage
  â†’ (attomGate) â†’ attomFetch? â†’ (routeAttomReplay) â†’ validateProject
  â†’ mashvisorFetch â†’ aggregate â†’ __end__
```

The canonical node order is pinned in `APPRAISAL_EVALUATION_NODE_IDS`. There is no
`spatialGenerate` node â€” the legacy pdfjs spatial-candidate path was removed when extraction was
consolidated onto Reducto (PR #2463), so `reductoExtract` is now the only extractor and
`resolveFields` merges Reducto candidates with XML and corrections.

Routing:

- `routeExtractionStrategy(detectScanned)`: `scannedPdf === true ? "ocr" : "reductoExtract"`;
  `ocr â†’ reductoExtract`.
- `attomGate(computeCoverage)`: route to `attomFetch` only when **not** in `enrichmentPass`,
  `ENABLE_ATTOM_ENRICHMENT === "true"`, the second pass fits the lock budget
  (`attomSecondPassExceedsLockBudget`), and `hasAttomAddressInputs`; otherwise go straight to
  `validateProject`.
- `routeAttomReplay(attomFetch)`: when `hasReplayableEnrichmentData` (any of `avmDetail`,
  `attomAvmDetail`, `avmHistory`, `comparableSales`, `salesTrend`), back-edge to `detectScanned`
  for a second extraction pass; otherwise go to `validateProject`.
- After `validateProject` the graph always runs `mashvisorFetch`, then `aggregate`.

Every node is wrapped by `instrumentNode`, which emits `node:<name>` started / complete / failed
milestones.

blocking the fetch.

- `strPerformance` â€” STR aggregate (from Lookup), or `null` on miss.
- `strComps` â€” STR comp array (from List Comps).
- `ltrComps` â€” LTR comp array (from Long-Term Comps).

  the retired `confidence` field â€” no real confidence score exists in the Mashvisor API.

</Accordion>

### R10 appraisal-gated completion race (LOA-1751)

When an EPC audit is appraisal-gated, the orchestration service (`integration-orchestration.service.ts`) used to rely solely on the appraisal processor's own `tryCompleteAuditRun` call to close the audit. That single attempt no-oped if appraisal finished before its barrier-awaited siblings, leaving the audit permanently stuck in `processing`. Since LOA-1751 (#2709), after `orchestrator.run` resolves the gated branch unconditionally retries `tryCompleteAuditRun` (1 initial + 2 retries at 100ms and 300ms), closing the gap without risking double-emission of `workflow:complete`. The follow-up durable backstop was delivered in LOA-1752 (#2719) as `AuditCompletionReaperService` â€” see [Orphan reapers](/agents/runs-ledger#orphan-reapers).

### agenticRefine has been deleted

The `agenticRefine` node (factory and spec) was removed entirely in LOA-1043 â€” it OOM-killed a
production worker on scanned multi-page documents. Do not reintroduce an agentic refine pass
without addressing that memory profile. The `spatialGenerate` node was likewise removed when the
legacy pdfjs extraction path was retired.

### ATTOM budgeting and deadlines

`APPRAISAL_GRAPH_DEADLINE_MS` defaults to 89 minutes (`envMinutesMs`) and sits strictly between
the legitimate enrichment floor and the worker lock
(`APPRAISAL_EXTRACTION_LOCK_DURATION_MS`, about 90 minutes). The floor
(`APPRAISAL_GRAPH_DEADLINE_LEGIT_FLOOR_MS`) is now split 85 min (`SUBSCHEMA_OVERALL_CEILING_MS`) +
2 min ATTOM + 1 min Mashvisor (`MASHVISOR_ENRICHMENT_CEILING_MS`, LOA-1449) = 88 minutes, so the
pinned ordering invariant is `upload(10) < single-call(60) < split+ATTOM+mashvisor(88) <
graph(89) < lock(90)`. `ATTOM_SECOND_PASS_CEILING_MS = SINGLE_CALL_DXT_CEILING_MS +
SINGLE_CALL_DXF_TIMEOUT_MS`. `attomSecondPassExceedsLockBudget` budgets against elapsed time since
`graphStartedAt` (a missing value means refuse). Relevant env: `ENABLE_ATTOM_ENRICHMENT`,
`APPRAISAL_GRAPH_DEADLINE_MS`.

<Warning>
LangGraph appraisal runs own ATTOM replay state. The processor must **not** perform legacy ATTOM
reruns â€” that is a trap door called out in the agent's `CLAUDE.md`.
</Warning>

### Appraisal rules

The rule definitions live in `lib/appraisal-pipeline/rules/definitions` (registry
`rule-registry.ts`) and span **67 rules** across these prefixes: `SUBJ_` (15), `SITE_` (8),
`EXH_` (6), `NEIGH_` (6), `COMP_` (6), `CONT_` (5), `IMP_` (6), `APPR_` (3), `CONDO_` (3),
`REC_` (3), `INCOME_` (2), `RENT_` (2), `GEN_` (1), `UNIT_` (1). The rules are agency-aware
(`AgencySource`: `fannie_mae`, `freddie_mac`), support compound conditions (`compound-walker`,
`condition-schema`), and use `feature-gates`.

**`IMP_006` â€” Unique / atypical property (LOA-1713)** is the newest global built-in. It flags when the appraiser characterizes the **property itself** as unique or atypical (e.g. a note saying "a unique property with limited comparables") by reading `improvements.has_unique_property_mention` (boolean tri-state). Outcomes: `true` â†’ WARNING, `false` â†’ PASS, `null` â†’ NEEDS_INFO. The field is extracted from the narrative (comments, addendum, improvements, neighborhood, sales-comparison, reconciliation sections); boilerplate like "a unique opportunity" does not set it. Migration `0216` seeds the template row and **backfills it into all existing active rule sets** (same pattern as `0193`). This rule is complementary to Deephaven's `DH_019` (floor-plan-specific); both are kept because they measure different things.

The `COMP_` family is `COMP_001`â€“`COMP_005` plus `COMP_007` (there is no `COMP_006`). `COMP_007`
(**Stated vs Actual Proximity**, LOA-1200 / PR #2315) verifies each comp's stated proximity
against the geocoded actual distance; it exists in the in-code registry but production evaluates
the DB-backed rules, so migration `0192_seed_comp_007_rule_template` seeds it into
`appraisal_rule_templates` and backfills it into existing rule sets. Migration
`0193_seed_missing_appraisal_rule_templates` likewise reconciles other in-code rules that were
never seeded. Migration `0213_comp_007_tolerance_10pct` (LOA-1200) tightened the proximity
tolerance from 25% to 10%.

The `appraisal_rules.operator` CHECK (`appraisal_rules_operator_check`) is the **16-value** union
(adding `abs_less_than` and `abs_less_than_or_equal` to the base 14). Credit reached the same
16-operator parity in migration `0190_credit_operator_abs_parity`, so the two unions now match.
The appraisal union is drift-pinned by `operator-check-drift.spec.ts`; the Drizzle schema must
mirror the latest migration SQL.

### Lender-specific rule bundles

Lender-specific overlays are authored as staged bundles in the `portal-appraisal-rules` module
(`staged-bundles/`). Applied via `/admin/staged-rules` â†’ dry-run preview â†’ promote.

#### Deephaven Phase 1.5 bundle

The **Deephaven Phase 1.5** bundle
LOA-1367..1378) adds the single Deephaven lender's compound rules `DH_012`â€“`DH_052` (property
condition, site/hazard + ADU, property-type classification, condo project, sales comparison, and
occupancy/solar/income/deferred-maintenance groups) on top of new Phase 1.5 extraction signals in
the adapter and `appraisal-data` schema. These DH rules are compound rules evaluated by the shared
`compound-walker`; they are written only into the Deephaven lender's active rule set and are not
part of the 67-rule core registry.

The accessory-dwelling-unit (ADU) rules in Group B are `DH_025` (**No accessory dwelling unit** â€”
a flat predicate that warns when `improvements.has_accessory_unit` is `true`) and `DH_026`
(**ADU minimum area** â€” a conditional that, when an ADU is present, fails if `cost_approach.adu_sqft`
is under 500 sq ft). Both read fields that originally existed only as Zod defs and so returned
`NEEDS_INFO` on every appraisal; LOA-1497 (PR #2506) wired `improvements.has_accessory_unit` and
`cost_approach.adu_sqft` through the Reducto/LLM extraction path plus adapter coercion (no
rule-definition or Zod change) so the two rules actually evaluate. The XML bridge maps
`â€¦improvements[0].units[0].detail.is_adu â†’ improvements.has_accessory_unit`, so DH_025 evaluates
from MISMO too, but DH_026 stays `NEEDS_INFO` on XML-only ADU appraisals â€” MISMO carries an ADU
count, not the cost-grid area, so only the PDF path fills `adu_sqft`.

Property-type characteristic flags were renamed from their original "required-state" naming
(e.g. â€śNot a row homeâ€ť, â€śNo unusual floor planâ€ť) to characteristic-first naming that describes
the property trait rather than a prohibition (LOA-1680). `DH_029` (`CUST_DEE00029`, the row-home
rule) was also downgraded from `FAIL` to `WARNING` â€” row homes and townhouses are eligible
property types; the rule flags for underwriter review, not disqualification. `DH_019`
(`CUST_DEE00019`) and related rules (`CUST_DEE00025`, `00030`, `00031`, `00049`) were renamed
similarly. Genuine disqualifiers (condotel, lava zone, multiple dwellings, condition defects,
recent conversion) retain `FAIL` severity and are unchanged.

<Note>
The fields were renamed between spec and as-built: LOA-1368/1374 spec'd `improvements.has_adu` and
`adu_size_sqft`, but the rule-read paths are `improvements.has_accessory_unit` and
`cost_approach.adu_sqft`. Grepping `has_adu` finds nothing.
</Note>

#### First Colony (Sharp Advantage) bundle updates (LOA-1713)

Two additions to the First Colony (`sharp_advantage`) bundle:

- **`FCAPP_010` â€” Atypical design style (WARNING, lender-wide):** flags any appraisal whose Design (Style) field isn't one of First Colony's accepted types â€” Traditional, Colonial, Contemporary, Ranch, Duplex, or Split Entry. - **`IMP_006` backfill:** the new global rule is automatically included in First Colony's active rule set via migration `0216` (same as all active sets).

#### Luxury WSE Credit bundle update (LOA-1652)

The `luxury-wse-credit` bundle now **disables `CRED_011`** (hard-inquiry count check) lender-wide, including DSCR programs. This replaces the earlier `report_only` approach from #2542 (never applied in prod). Rationale: `CRED_011` with no threshold adds no signal for Luxury â€” virtually every borrower has at least one inquiry, so a threshold-less WARNING would mark every loan Medium risk without identifying a real underwriting concern. Re-apply the bundle via `/admin/staged-credit-rules` on deploy.

## Review summary export

`POST /:id/export` (`PortalAppraisalController.export`) renders the appraisal **Review summary**
as a branded PDF â€” property facts, an **Issues Found** section (the failed and warning rules,
surfaced near the top), the full findings list, and program-rule outcomes. The payload comes from
the same `getRunDetail` projection the review page renders (`appraisal-export-data.service.ts` â†’
`renderAppraisalReviewHtml` / `appraisal-pdf-template.ts`), so the PDF matches the on-screen
review. It reuses the `bank-statement-export` Puppeteer writer and signed-URL service.

- **Roles**: `client_admin`, `client_user`, and `super_admin` (PR #2477). Throttled to 5
  requests/minute per user; the run must be `completed` or `reviewed` or the export is refused.
- **Findings gate** (PR #2496): the Findings section's empty state is keyed on whether enrichment
  actually ran for **this** run â€” `findingsEvaluated = run.enrichmentData != null` â€” not the live
  `ENABLE_ATTOM_ENRICHMENT` env flag. A run done while ATTOM was off (or where `attomGate` skipped
  enrichment) shows no "No findings" block, so a later flag flip cannot mislabel historical runs.

## Reliability and recovery

The appraisal processor uses the same heartbeat / `claimToken` / bounded-recovery pattern as
credit, with an in-process graph deadline below the BullMQ lock. The
`sweepOrphanedRuns` reaper recovers stale `pending`/`running` runs, re-enqueues them up to
`APPRAISAL_MAX_RECOVERY_ATTEMPTS`, then terminally fails them â€” the same shape as the credit
orphan sweep.

<Tip>
Investigating an appraisal run that stalled or failed? See
[Investigating a failed run](/runbooks/investigating-a-failed-run).
</Tip>
