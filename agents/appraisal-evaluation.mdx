The Appraisal-Evaluation agent (`appraisalEvaluation`) is a fully implemented, 12-node compiled
LangGraph that extracts and enriches appraisal data. Unlike
[Credit-Evaluation](/agents/credit-evaluation), rule evaluation does **not** run inside the
graph ŌĆö it runs in the processor, after the graph returns.

<Note>
The graph performs extraction and enrichment only. Its output type
(`AppraisalEvaluationGraphOutput`) carries no `complianceResults`. The processor
(`appraisal.processor.ts`) re-runs `evaluateWithCustomRules` with its dependency-injected DB and
persists results. `LangGraphService.createAppraisalEvaluationGraph` does **not** persist
`appraisal_runs` ŌĆö the processor does. This is a deliberate divergence from the credit agent,
whose `aggregate` node persists.
</Note>

## How it runs

Runs are created via `POST /api/v1/portal/appraisals/run` (`PortalAppraisalController`, base
path `api/v1/portal/appraisals`). The DTO `CreateRunDto` carries `s3Key` (required), optional
`xmlS3Key`, `loanProgramId`, `loanNumber`, and `agencySources?` (in `fannie_mae | freddie_mac`,
max 2). The processor (`@Processor("appraisal-extraction")`) claims the run with a heartbeat and
`claimToken` (the same recovery pattern as credit), calls
`LangGraphService.createAppraisalEvaluationGraph`, persists the graph-returned `enrichmentData`,
then runs the rules and findings (see below).

### EPC audit path (LOA-1675)

The agent also runs on the EPC audit path, triggered post-completion by
`triggerAppraisalEvaluation` in `integration-orchestration.service.ts`. After `WORKFLOW.COMPLETE`
fires, the orchestrator enumerates classified appraisal documents (`appraisal_standard`,
`desk_review`) from the S3 manifest, copies the appraisal PDF into the portal bucket, mints an
`appraisal_runs` row stamped with `loan_id`, `lender_id`, and `audit_run_id`, and dispatches
the job by identity. The EPC audit completes synchronously; the appraisal job runs as a
detached best-effort task (up to ~89 min) and does not gate `WORKFLOW.COMPLETE`.

Read the EPC-linked run via `GET /api/v1/audit_runs/:id/appraisal` ŌĆö returns `PortalAppraisalPayload`
or `{ data: null }` (never 404). The `AppraisalAuditProjectionService` applies the manual-override
overlay (LOA-1685) to the projection. See [Integration orchestration pipeline](/integrations/orchestration-pipeline)
and [Reading audit results](/pipeline/results).

`LangGraphService.createAppraisalEvaluationGraph` takes an options object, returns the
extraction result plus the graph-owned `enrichmentData`, and races the graph against
`APPRAISAL_GRAPH_DEADLINE_MS`. The builder `buildAppraisalEvaluationGraph` closes over
`pdfBytes`, optional `xmlCandidates`, `onPartial`, `onIteration`, and `inputFilename` ŌĆö these
stay closure-delivered and are never placed in graph state or serialized progress.

## The graph

```text
__start__ ŌåÆ prepare ŌåÆ classify ŌåÆ detectScanned
  ŌåÆ (routeExtractionStrategy) ŌåÆ ocr? ŌåÆ reductoExtract
  ŌåÆ resolveFields ŌåÆ normalizeAdapter ŌåÆ computeCoverage
  ŌåÆ (attomGate) ŌåÆ attomFetch? ŌåÆ (routeAttomReplay) ŌåÆ validateProject
  ŌåÆ mashvisorFetch ŌåÆ aggregate ŌåÆ __end__
```

The canonical node order is pinned in `APPRAISAL_EVALUATION_NODE_IDS`. There is no
`spatialGenerate` node ŌĆö the legacy pdfjs spatial-candidate path was removed when extraction was
consolidated onto Reducto (PR #2463), so `reductoExtract` is now the only extractor and
`resolveFields` merges Reducto candidates with XML and corrections.

Routing:

- `routeExtractionStrategy(detectScanned)`: `scannedPdf === true ? "ocr" : "reductoExtract"`;
  `ocr ŌåÆ reductoExtract`.
- `attomGate(computeCoverage)`: route to `attomFetch` only when **not** in `enrichmentPass`,
  `ENABLE_ATTOM_ENRICHMENT === "true"`, the second pass fits the lock budget
  (`attomSecondPassExceedsLockBudget`), and `hasAttomAddressInputs`; otherwise go straight to
  `validateProject`.
- `routeAttomReplay(attomFetch)`: when `hasReplayableEnrichmentData` (any of `avmDetail`,
  `attomAvmDetail`, `avmHistory`, `comparableSales`, `salesTrend`), back-edge to `detectScanned`
  for a second extraction pass; otherwise go to `validateProject`.
- After `validateProject` the graph always runs `mashvisorFetch`, then `aggregate`.

Every node is wrapped by `instrumentNode`, which emits `node:<name>` started / complete / failed
milestones.

`MASHVISOR_ENRICHMENT_CEILING_MS` (1 minute). The service fans out **three parallel endpoints**
(PR #2752):

- **Lookup** ŌĆö STR aggregate performance: `median_rental_income`, `median_annual_revenue`,
  `median_occupancy_rate`, `sample_size`.
- **List Comps** ŌĆö STR comparable listings: short-term rental comps filtered, ranked, and capped
  at **5** (see below).
- **Long-Term Comps** ŌĆö LTR comparable listings: an array of long-term rental comps
  (`LtrComp`: `monthlyRent`, `bedrooms`, `bathrooms`, `sqft`, `address`, `listingDate`,
  `distanceMiles`, `url`).

`NormalizedAddress` now carries `latitude`/`longitude` (resolved from ATTOM enrichment data in
the node) so each comp's `distanceMiles` can be computed via haversine; the field is `null`
when ATTOM coordinates are unavailable ŌĆö comps render with `distanceMiles: null` rather than
blocking the fetch.

#### STR comps filtering and ranking (LOA-1940, PR #2980)

The raw List Comps response is filtered and ranked before it reaches the portal:

- **Bedroom filter** ŌĆö comps must exactly match the subject's bedroom count(s). Mashvisor's `beds=N` param is fuzzy (returns ŌēźN-ish), so a client-side exact match (`num_of_beds Ōłł targets`) is applied after fetching. Multi-unit Form 1025 appraisals use the distinct bedroom counts across `units[].bedrooms`; single-unit forms use the scalar `improvements.bedrooms` (or `unit_description.bedrooms` for condos). Per-target fetches run in parallel with adaptive paging ŌĆö page 2 is requested only when a target's page-1 exact-match count is below the display cap.
- **Entire-home only** ŌĆö comps where `room_type !== "Entire home/apt"` (private-room, shared-room) are dropped.
- **Best 5** ŌĆö the table is capped at 5 comps. Quota is allocated deterministically across bedroom targets (`floor(5/n)` + nearest-remainder + thin-pool redistribution) so each unit type retains representation, then sorted by proximity (closest first).
- **Location column** ŌĆö shows a real `"City, State"` locality string (e.g. `"Brooklyn, NY"`) instead of a placeholder repeated on every row.

Two discriminators flow from backend ŌåÆ DTO ŌåÆ portal:
- `strCompsFilteredEmpty` ŌĆö list-comps returned results but bedroom/entire-home filters left none. The finding is relabelled **"Rental Comps Filtered"** (never asserts "no rental market").
- `bedTargetsResolved` ŌĆö whether subject bedroom count was derivable. Gates "matched" vs. "nearby comparable listings" copy so legacy/underivable rows do not falsely claim a bedroom match.

The `RentalDataResult` shape is:
- `strPerformance` ŌĆö STR aggregate (from Lookup), or `null` on miss.
- `strComps` ŌĆö STR comp array (from List Comps), after bedroom/entire-home filtering, quota ranking, and 5-comp cap.
- `ltrComps` ŌĆö LTR comp array (from Long-Term Comps), **bounded to the nearest 10** (proximity-ranked
  by haversine distance, nearest-first; comps without coordinates keep API order at the tail).
  Added in LOA-1869 (#2849) ŌĆö a median over the full unbounded page-1 slice (up to 30
  unranked comps) was noisy. The bound is cardinality, not sort-only: the sort picks which 10.
- `sampleSize` ŌĆö number of STR comps behind the Lookup aggregate (`null` on miss). Replaces
  the retired `confidence` field ŌĆö no real confidence score exists in the Mashvisor API.

<Note>
**Occupancy format (LOA-1869):** the Mashvisor API returns `median_occupancy_rate` and the
per-comp `occupancy` field as a **percent** (0ŌĆō100, e.g. `42`), not a fraction. The adapter
converts to a ratio (0ŌĆō1) before writing to `strPerformance.occupancy` and
`RentalComp.occupancy`. Values outside `[0,1]` after conversion are nulled by a downstream
sanity guard. The live API also wraps every response in a `{ status, content: ŌĆ” }` envelope;
the adapter unwraps `content` before schema-parsing and treats a missing or non-object
`content` as an unrecognized shape (distinct from a genuine-empty `{}` response).
</Note>

The node never throws (per-endpoint failures surface in `_errors`; a timeout on the whole
request yields the null-sentinel for all fields). The result rides graph state as `rentalData`;
the processor merges rental findings (`mergeRentalFindings`) and persists
`appraisal_runs.rental_data`.
</Accordion>
<Accordion title="aggregate">
Assembles the final `result`, preserves `_reductoResult`, and keeps the latest-pass warnings.
</Accordion>
</AccordionGroup>

### R10 appraisal-gated completion race (LOA-1751)

When an EPC audit is appraisal-gated, the orchestration service (`integration-orchestration.service.ts`) used to rely solely on the appraisal processor's own `tryCompleteAuditRun` call to close the audit. That single attempt no-oped if appraisal finished before its barrier-awaited siblings, leaving the audit permanently stuck in `processing`. Since LOA-1751 (#2709), after `orchestrator.run` resolves the gated branch unconditionally retries `tryCompleteAuditRun` (1 initial + 2 retries at 100ms and 300ms), closing the gap without risking double-emission of `workflow:complete`. The follow-up durable backstop was delivered in LOA-1752 (#2719) as `AuditCompletionReaperService` ŌĆö see [Orphan reapers](/agents/runs-ledger#orphan-reapers).

### agenticRefine has been deleted

The `agenticRefine` node (factory and spec) was removed entirely in LOA-1043 ŌĆö it OOM-killed a
production worker on scanned multi-page documents. Do not reintroduce an agentic refine pass
without addressing that memory profile. The `spatialGenerate` node was likewise removed when the
legacy pdfjs extraction path was retired.

### ATTOM budgeting and deadlines

`APPRAISAL_GRAPH_DEADLINE_MS` defaults to 89 minutes (`envMinutesMs`) and sits strictly between
the legitimate enrichment floor and the worker lock
(`APPRAISAL_EXTRACTION_LOCK_DURATION_MS`, about 90 minutes). The floor
(`APPRAISAL_GRAPH_DEADLINE_LEGIT_FLOOR_MS`) is now split 85 min (`SUBSCHEMA_OVERALL_CEILING_MS`) +
2 min ATTOM + 1 min Mashvisor (`MASHVISOR_ENRICHMENT_CEILING_MS`, LOA-1449) = 88 minutes, so the
pinned ordering invariant is `upload(10) < single-call(60) < split+ATTOM+mashvisor(88) <
graph(89) < lock(90)`. `ATTOM_SECOND_PASS_CEILING_MS = SINGLE_CALL_DXT_CEILING_MS +
SINGLE_CALL_DXF_TIMEOUT_MS`. `attomSecondPassExceedsLockBudget` budgets against elapsed time since
`graphStartedAt` (a missing value means refuse). Relevant env: `ENABLE_ATTOM_ENRICHMENT`,
`APPRAISAL_GRAPH_DEADLINE_MS`.

<Warning>
LangGraph appraisal runs own ATTOM replay state. The processor must **not** perform legacy ATTOM
reruns ŌĆö that is a trap door called out in the agent's `CLAUDE.md`.
</Warning>

## Processor-side rule evaluation

After the graph returns, `appraisal.processor.ts`:

<Steps>
<Step title="Persist enrichment">
Writes the graph-returned `enrichmentData`.
</Step>
<Step title="Evaluate rules">
Runs `evaluateWithCustomRules` (`lib/appraisal-pipeline/rules/evaluator.ts`) with the
processor's DI'd DB.
</Step>
<Step title="Generate findings">
`AppraisalFindingsService.generateFindings`.
</Step>
<Step title="Cross-document validation (step 7d)">
Once per extraction completion (gated on `auditRunId` + `loanId`; portal-only uploads skip
silently): retrieves three source documents ŌĆö the URLA/1003 (`urla_1003` or `ed_urla`
classification type), the 1008 Transmittal (`transmittal_1008`), and the Flood Certificate
(`flood_certificate`) ŌĆö via per-document deterministic LlamaCloud retrieval, then runs one
structured-output extraction call per source doc (3 LLM calls/run, no LLM query planner).
Compares 10 appraisal-extracted fields against the source values using deterministic
normalize-then-exact TypeScript (never fuzzy). Disagreements surface as `XDOC-*` **warning**
findings in the existing appraisal findings stream. Results are persisted in the new
`appraisal_runs.cross_doc_validation` jsonb column (migration `0264`), which stores only source
values ŌĆö verdicts are always re-derived so corrections remain correct. The LLM step uses the
`appraisalEvaluation.crossDocExtraction` config (gpt-4.1, structuredOutput). On for all lenders;
not lender-configurable.
</Step>
<Step title="Write the run">
Writes `appraisal_runs` with `extractedData`, `complianceResults`, `findings`, coverage fields,
the field maps, and `cross_doc_validation`.
</Step>
</Steps>

### Appraisal rules

The rule definitions live in `lib/appraisal-pipeline/rules/definitions` (registry
`rule-registry.ts`) and span **68 rules** across these prefixes: `SUBJ_` (15), `SITE_` (8),
`EXH_` (6), `NEIGH_` (6), `COMP_` (6), `CONT_` (6), `IMP_` (6), `APPR_` (3), `CONDO_` (3),
`REC_` (3), `INCOME_` (2), `RENT_` (2), `GEN_` (1), `UNIT_` (1). The rules are agency-aware
(`AgencySource`: `fannie_mae`, `freddie_mac`), support compound conditions (`compound-walker`,
`condition-schema`), and use `feature-gates`.

**`CONT_006` ŌĆö Appraised value below purchase price (LOA-1753)** is the newest global built-in. It flags when the appraiserŌĆÖs `reconciliation.final_opinion_of_value` (alias-aware via `resolveAppraisedValue`) comes in below `contract.contract_price`. Purchase transactions only; refinances return `SKIPPED`. Default severity `WARNING` (configurable per lender). A `tolerancePercent` threshold (default `0` = flag any shortfall) suppresses rounding noise. Missing operands return `NEEDS_INFO` rather than double-flagging ŌĆö presence is owned by `REC_002` and `CONT_002`. Lender-neutral (no lender-code gate, no loan-context dependency) so it also evaluates on standalone portal appraisal runs. Migration `0246` seeds the template and backfills it into all active rule sets.

**`IMP_006` ŌĆö Unique / atypical property (LOA-1713)** is the previous newest global built-in.

The `COMP_` family is `COMP_001`ŌĆō`COMP_005` plus `COMP_007` (there is no `COMP_006`). `COMP_007`
(**Stated vs Actual Proximity**, LOA-1200 / PR #2315) verifies each comp's stated proximity
against the geocoded actual distance; it exists in the in-code registry but production evaluates
the DB-backed rules, so migration `0192_seed_comp_007_rule_template` seeds it into
`appraisal_rule_templates` and backfills it into existing rule sets. Migration
`0193_seed_missing_appraisal_rule_templates` likewise reconciles other in-code rules that were
never seeded. Migration `0213_comp_007_tolerance_10pct` (LOA-1200) tightened the proximity
tolerance from 25% to 10%.

The `appraisal_rules.operator` CHECK (`appraisal_rules_operator_check`) is the **16-value** union
(adding `abs_less_than` and `abs_less_than_or_equal` to the base 14). Credit reached the same
16-operator parity in migration `0190_credit_operator_abs_parity`, so the two unions now match.
The appraisal union is drift-pinned by `operator-check-drift.spec.ts`; the Drizzle schema must
mirror the latest migration SQL.

### Lender-specific rule bundles

Lender-specific overlays are authored as staged bundles in the `portal-appraisal-rules` module
(`staged-bundles/`). Applied via `/admin/staged-rules` ŌåÆ dry-run preview ŌåÆ promote.

#### Deephaven Phase 1.5 bundle

The **Deephaven Phase 1.5** bundle
LOA-1367..1378) adds the single Deephaven lender's compound rules `DH_012`ŌĆō`DH_052` (property
condition, site/hazard + ADU, property-type classification, condo project, sales comparison, and
occupancy/solar/income/deferred-maintenance groups) on top of new Phase 1.5 extraction signals in
the adapter and `appraisal-data` schema. These DH rules are compound rules evaluated by the shared
`compound-walker`; they are written only into the Deephaven lender's active rule set and are not
part of the 67-rule core registry.

The accessory-dwelling-unit (ADU) rules in Group B are `DH_025` (**No accessory dwelling unit** ŌĆö
a flat predicate that warns when `improvements.has_accessory_unit` is `true`) and `DH_026`
(**ADU minimum area** ŌĆö a conditional that, when an ADU is present, fails if `cost_approach.adu_sqft`
is under 500 sq ft). Both read fields that originally existed only as Zod defs and so returned
`NEEDS_INFO` on every appraisal; LOA-1497 (PR #2506) wired `improvements.has_accessory_unit` and
`cost_approach.adu_sqft` through the Reducto/LLM extraction path plus adapter coercion (no
rule-definition or Zod change) so the two rules actually evaluate. The XML bridge maps
`ŌĆ”improvements[0].units[0].detail.is_adu ŌåÆ improvements.has_accessory_unit`, so DH_025 evaluates
from MISMO too, but DH_026 stays `NEEDS_INFO` on XML-only ADU appraisals ŌĆö MISMO carries an ADU
count, not the cost-grid area, so only the PDF path fills `adu_sqft`.

Property-type characteristic flags were renamed from their original "required-state" naming
(e.g. ŌĆ£Not a row homeŌĆØ, ŌĆ£No unusual floor planŌĆØ) to characteristic-first naming that describes
the property trait rather than a prohibition (LOA-1680). `DH_029` (`CUST_DEE00029`, the row-home
rule) was also downgraded from `FAIL` to `WARNING` ŌĆö row homes and townhouses are eligible
property types; the rule flags for underwriter review, not disqualification. `DH_019`
(`CUST_DEE00019`) and related rules (`CUST_DEE00025`, `00030`, `00031`, `00049`) were renamed
similarly. Genuine disqualifiers (condotel, lava zone, multiple dwellings, condition defects,
recent conversion) retain `FAIL` severity and are unchanged.

<Note>
The fields were renamed between spec and as-built: LOA-1368/1374 spec'd `improvements.has_adu` and
`adu_size_sqft`, but the rule-read paths are `improvements.has_accessory_unit` and
`cost_approach.adu_sqft`. Grepping `has_adu` finds nothing.
</Note>

#### First Colony (Sharp Advantage) bundle updates (LOA-1713)

Two additions to the First Colony (`sharp_advantage`) bundle:

- **`FCAPP_010` ŌĆö Atypical design style (WARNING, lender-wide):** flags any appraisal whose Design (Style) field isn't one of First Colony's accepted types ŌĆö Traditional, Colonial, Contemporary, Ranch, Duplex, or Split Entry. Keyed on `improvements.design_style` using a generated case-insensitive regex (`FC_DESIGN_STYLE_PATTERN`) to handle casing, all-caps, and compound values (e.g. "2 Story/Colonial"). Variants normalized: Split Level / Split Foyer / Bi-Level ŌåÆ Split Entry; Raised Ranch ŌåÆ Ranch.
- **`IMP_006` backfill:** the new global rule is automatically included in First Colony's active rule set via migration `0216` (same as all active sets).

Apply via `/admin/staged-rules` ŌåÆ First Colony ŌåÆ dry-run ŌåÆ promote.

#### Luxury WSE Credit bundle update (LOA-1652)

The `luxury-wse-credit` bundle now **disables `CRED_011`** (hard-inquiry count check) lender-wide, including DSCR programs. This replaces the earlier `report_only` approach from #2542 (never applied in prod). Rationale: `CRED_011` with no threshold adds no signal for Luxury ŌĆö virtually every borrower has at least one inquiry, so a threshold-less WARNING would mark every loan Medium risk without identifying a real underwriting concern. Re-apply the bundle via `/admin/staged-credit-rules` on deploy.

## Review summary export

`POST /:id/export` (`PortalAppraisalController.export`) renders the appraisal **Review summary**
as a branded PDF ŌĆö property facts, an **Issues Found** section (the failed and warning rules,
surfaced near the top), the full findings list, and program-rule outcomes. The payload comes from
the same `getRunDetail` projection the review page renders (`appraisal-export-data.service.ts` ŌåÆ
`renderAppraisalReviewHtml` / `appraisal-pdf-template.ts`), so the PDF matches the on-screen
review. It reuses the `bank-statement-export` Puppeteer writer and signed-URL service.

- **Roles**: `client_admin`, `client_user`, and `super_admin` (PR #2477). Throttled to 5
  requests/minute per user; the run must be `completed` or `reviewed` or the export is refused.
- **Findings gate** (PR #2496): the Findings section's empty state is keyed on whether enrichment
  actually ran for **this** run ŌĆö `findingsEvaluated = run.enrichmentData != null` ŌĆö not the live
  `ENABLE_ATTOM_ENRICHMENT` env flag. A run done while ATTOM was off (or where `attomGate` skipped
  enrichment) shows no "No findings" block, so a later flag flip cannot mislabel historical runs.

## Reliability and recovery

The appraisal processor uses the same heartbeat / `claimToken` / bounded-recovery pattern as
credit, with an in-process graph deadline below the BullMQ lock. The
`sweepOrphanedRuns` reaper recovers stale `pending`/`running` runs, re-enqueues them up to
`APPRAISAL_MAX_RECOVERY_ATTEMPTS`, then terminally fails them ŌĆö the same shape as the credit
orphan sweep.
