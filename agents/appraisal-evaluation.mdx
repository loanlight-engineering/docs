The Appraisal-Evaluation agent (`appraisalEvaluation`) is a fully implemented, 13-node compiled
LangGraph that extracts and enriches appraisal data. Unlike
[Credit-Evaluation](/agents/credit-evaluation), rule evaluation does **not** run inside the
graph — it runs in the processor, after the graph returns.

<Note>
The graph performs extraction and enrichment only. Its output type
(`AppraisalEvaluationGraphOutput`) carries no `complianceResults`. The processor
(`appraisal.processor.ts`) re-runs `evaluateWithCustomRules` with its dependency-injected DB and
persists results. `LangGraphService.createAppraisalEvaluationGraph` does **not** persist
`appraisal_runs` — the processor does. This is a deliberate divergence from the credit agent,
whose `aggregate` node persists.
</Note>

## How it runs

Runs are created via `POST /api/v1/portal/appraisals/run` (`PortalAppraisalController`, base
path `api/v1/portal/appraisals`). The DTO `CreateRunDto` carries `s3Key` (required), optional
`xmlS3Key`, `loanProgramId`, `loanNumber`, and `agencySources?` (in `fannie_mae | freddie_mac`,
max 2). The processor (`@Processor("appraisal-extraction")`) claims the run with a heartbeat and
`claimToken` (the same recovery pattern as credit), calls
`LangGraphService.createAppraisalEvaluationGraph`, persists the graph-returned `enrichmentData`,
then runs the rules and findings (see below).

`LangGraphService.createAppraisalEvaluationGraph` takes an options object, returns the
extraction result plus the graph-owned `enrichmentData`, and races the graph against
`APPRAISAL_GRAPH_DEADLINE_MS`. The builder `buildAppraisalEvaluationGraph` closes over
`pdfBytes`, optional `xmlCandidates`, `onPartial`, `onIteration`, and `inputFilename` — these
stay closure-delivered and are never placed in graph state or serialized progress.

## The graph

```text
__start__ → prepare → classify → detectScanned
  → (routeExtractionStrategy) → ocr? → reductoExtract
  → spatialGenerate → resolveFields → normalizeAdapter → computeCoverage
  → (attomGate) → attomFetch? → (routeAttomReplay) → validateProject
  → mashvisorFetch → aggregate → __end__
```

Routing:

- `routeExtractionStrategy(detectScanned)`: `scannedPdf === true ? "ocr" : "reductoExtract"`;
  `ocr → reductoExtract`.
- `attomGate(computeCoverage)`: route to `attomFetch` only when **not** in `enrichmentPass`,
  `ENABLE_ATTOM_ENRICHMENT === "true"`, the second pass fits the lock budget
  (`attomSecondPassExceedsLockBudget`), and `hasAttomAddressInputs`; otherwise go straight to
  `validateProject`.
- `routeAttomReplay(attomFetch)`: when `hasReplayableEnrichmentData` (any of `avmDetail`,
  `attomAvmDetail`, `avmHistory`, `comparableSales`, `salesTrend`), back-edge to `detectScanned`
  for a second extraction pass; otherwise go to `validateProject`.

Every node is wrapped by `instrumentNode`, which emits `node:<name>` started / complete / failed
milestones.

<Accordion title="prepare">

<Accordion title="classify">

<Accordion title="spatialGenerate">

</Accordion>

</Accordion>
<Accordion title="computeCoverage">
Coverage percent plus the missing mandatory fields.
</Accordion>
<Accordion title="attomFetch">
ATTOM property enrichment, ceiling 2 minutes.
</Accordion>
<Accordion title="validateProject">
Project / condo validations.
</Accordion>

</Accordion>

### agenticRefine is intentionally unwired

The `agenticRefine` node factory exists but is **not** wired into the graph. It OOM-killed a
production worker on scanned multi-page documents (run `803ada7d`, 2026-06-09). The factory is
retained for a future reseed; do not re-wire it without addressing the memory profile.

### ATTOM budgeting and deadlines

`APPRAISAL_GRAPH_DEADLINE_MS` defaults to 88 minutes (`envMinutesMs`) and sits strictly between
the legitimate split + ATTOM + Mashvisor floor (`SUBSCHEMA_OVERALL_CEILING_MS` + 2 min ATTOM +
1 min Mashvisor) and the worker lock (`APPRAISAL_EXTRACTION_LOCK_DURATION_MS`, about 90 minutes).
`ATTOM_SECOND_PASS_CEILING_MS = SINGLE_CALL_DXT_CEILING_MS + SINGLE_CALL_DXF_TIMEOUT_MS`.
`attomSecondPassExceedsLockBudget` budgets against elapsed time since `graphStartedAt` (a missing
value means refuse). Relevant env: `ENABLE_ATTOM_ENRICHMENT`, `APPRAISAL_GRAPH_DEADLINE_MS`.

<Warning>
LangGraph appraisal runs own ATTOM replay state. The processor must **not** perform legacy ATTOM
reruns — that is a trap door called out in the agent's `CLAUDE.md`.
</Warning>

## Rental data enrichment (Mashvisor)

For **DSCR programs only**, the graph runs an additive STR/LTR rental-data enrichment
(`mashvisorFetch`) parallel in spirit to `attomFetch`. The feature is **flag-gated and dark by
default** — it stays inert until both env vars are set:

| Env var | Default | Purpose |
|---|---|---|
| `MASHVISOR_ENABLED` | unset (off) | Opt-in flag. Must be the literal `"true"` to arm the source. |
| `MASHVISOR_API_KEY` | unset | Mashvisor Core 5 API key. Without it the adapter never fires. |

When the gate is open, the `MashvisorService` adapter calls the Mashvisor Core 5 endpoint with a
normalized subject address (street, city, state, optional zip, optional unit). The result is a
`RentalDataResult` with `strPerformance` (monthly/annual revenue, occupancy), up to a few STR
`strComps`, an `ltrFallback` (long-term-rent estimate), a `confidence` score, and an internal
`_endpointStatus` map.

### Persistence and redaction

`rental_data` is persisted as a jsonb column on `appraisal_runs` and survives reruns through an
append-only `mergeRentalFindings` that strips stale `RF-*` rows before re-adding the fresh set.
The processor allowlist-picks the DTO surfaced to the portal, so the upstream `propertyProfile`
field is **never exposed to clients** — it is retained only for adapter-side debugging.

### Informational findings

The `RentalFindingsService` emits **info-severity** findings keyed by stable RF-* codes carried
in `ruleReference`. These are advisory only; they never block a run.

| Code | When it fires |
|---|---|
| `RF-STR-INCOME` | STR performance present (monthly/annual revenue and occupancy). |
| `RF-STR-LIMITED` | STR comps present but performance is partial or thin. |
| `RF-LTR-RENT` | Long-term-rent fallback (`ltrFallback`) present. |
| `RF-DATA-PARTIAL` | Some Mashvisor endpoints succeeded, others did not. |
| `RF-DATA-UNAVAILABLE` | No data returned, or the provider errored / timed out. |
| `RF-CONFIDENCE` | A non-null confidence score was returned. |

`RF-DATA-UNAVAILABLE` distinguishes a provider failure (error or timeout) from a true "no rental
data exists for this subject" outcome — the explanation text reflects which case applied.

### Behaviour

- **DSCR gate**: a missing or non-DSCR `programDocType` (resolved by the processor from
  `loan_programs.match_doc_type`) skips the fetch entirely.
- **Address guard**: missing street / city / state skips the fetch; a captured subject unit is
  folded into the address line so condo and multifamily lookups are not building-level.
- **Never throws**: any adapter error or timeout produces `RF-DATA-UNAVAILABLE` and leaves the
  rest of the run untouched.
- **Wall-clock ceiling**: 60 s per run; cleared in `finally` to avoid lingering abort signals.
- **Zero impact on non-DSCR flows**: ATTOM, extraction, and rule evaluation are unchanged.

## Processor-side rule evaluation

After the graph returns, `appraisal.processor.ts`:

<Steps>
<Step title="Persist enrichment">
Writes the graph-returned `enrichmentData`.
</Step>
<Step title="Evaluate rules">
Runs `evaluateWithCustomRules` (`lib/appraisal-pipeline/rules/evaluator.ts`) with the
processor's DI'd DB.
</Step>
<Step title="Generate findings">
`AppraisalFindingsService.generateFindings`.
</Step>
<Step title="Write the run">
Writes `appraisal_runs` with `extractedData`, `complianceResults`, `findings`, coverage fields,
and the field maps.
</Step>
</Steps>

### Appraisal rules

The rule definitions live in `lib/appraisal-pipeline/rules/definitions` (registry
`rule-registry.ts`) and span roughly 65 rules across many prefixes: `SUBJ_` (15), `SITE_` (8),
`EXH_` (6), `NEIGH_` (6), `CONT_` (5), `COMP_` (5), `IMP_` (5), `APPR_` (3), `CONDO_` (3),
`REC_` (3), `INCOME_` (2), `RENT_` (2), `GEN_` (1), `UNIT_` (1). The rules are agency-aware
(`AgencySource`: `fannie_mae`, `freddie_mac`), support compound conditions (`compound-walker`,
`condition-schema`), and use `feature-gates`.

The `appraisal_rules.operator` CHECK (`appraisal_rules_operator_check`) is the **16-value** union,
which — unlike credit's 14-value union — adds `abs_less_than` and `abs_less_than_or_equal`. This union is drift-pinned by
`operator-check-drift.spec.ts`; the Drizzle schema must mirror the latest migration SQL.

## Reliability and recovery

The appraisal processor uses the same heartbeat / `claimToken` / bounded-recovery pattern as
credit, with an in-process graph deadline below the BullMQ lock. The
`sweepOrphanedRuns` reaper recovers stale `pending`/`running` runs, re-enqueues them up to
`APPRAISAL_MAX_RECOVERY_ATTEMPTS`, then terminally fails them — the same shape as the credit
orphan sweep.
