Datadog is the source of truth for **system behaviour around the agents**: HTTP
traffic, errors, infrastructure and ECS metrics, third-party calls, logs, RUM, and
node-level timing. Reach for it after LangSmith and the code, when you need a
concrete runtime fact. See [Observability overview](/ops/observability) for the
doctrine.

## Service names: APM vs logs

The single most important Datadog fact for LoanLight is that **APM service names do
not match log service tags.**

| Repo | APM `DD_SERVICE` | Log service tag |
|---|---|---|
| loanlight-api | `loanlight-api` | `llapi` |
| loanlight-integrations | `loanlight-integrations` | `llintegrations` |

<Warning>
When you query logs, filter on the **log** tag (`llapi`, `llintegrations`), not the
APM service name. The committed dashboard
(`packages/api/observability/datadog/README.md`) notes that appraisal milestone
logs surfaced under `service:llapi`, so its log panels query
`service:(llapi OR loanlight-api)` to be safe. Mirror that pattern in ad-hoc
queries.
</Warning>

### Log status and levels

loanlight-api logs reach Datadog through the CloudWatch Lambda **forwarder** (`source:cloudwatch`), which bypasses Datadog's built-in pino pipeline — so pino's numeric `level` (50) was never mapped to a log status and every backend 500 surfaced as `status:info`, invisible to `status:error` searches and monitors. Two fixes (LOA-1656, #2551) corrected this:

- The pino logger config (`app.module.ts`, `loggerConfig.pinoHttp.formatters.level`) now emits a string `status` (Datadog's reserved log-status attribute) alongside the numeric `level` (preserved for existing `custom.level` queries).
- The global exception filter (`common/filters/http-exception.filter.ts`) logs **4xx as `warn` and 5xx as `error`**, ships the exception `stack` as a structured field (object-first log call), and strips query-string values from the `req` serializer (PII the `redact` list cannot scrub inside a URL).

<Tip>
Query real server errors with `status:error` (5xx). Routine 401/404/422 client errors land under `status:warn`, so they no longer bury 500s.
</Tip>

</Note>

</Note>

## APM tracing

`dd-trace/init` is the very first import in `packages/api/src/main.ts` (with the
comment "Must be first import!"). It auto-starts APM and, when
`DD_PROFILING_ENABLED=true`, the Continuous Profiler. There is no explicit
`tracer.init({...})`; configuration is entirely env-driven.

The portal initializes tracing in `packages/app/src/instrumentation.ts`
(`register()` imports `dd-trace/init` only when `NEXT_RUNTIME === "nodejs"`), and
also defines `onRequestError()` for server-error logging.

`main.ts` adds the Datadog distributed-trace headers to its CORS allowlist:
`traceparent`, `tracestate`, `x-datadog-trace-id`, `x-datadog-parent-id`,
`x-datadog-sampling-priority`, `x-datadog-tags`, `x-datadog-origin`.

Extra DD env on the API app (`getApplicationEnvironment()`):

- `DD_PROFILING_ENABLED=true` (continuous profiler)
- `DD_GIT_COMMIT_SHA` and `DD_GIT_REPOSITORY_URL` (source code integration)
- `DD_DBM_PROPAGATION_MODE=full` (SQL-to-trace linking)

## The Datadog agent sidecar

ECS Fargate services that emit telemetry use the
`FargateTaskDefinitionWithDataDog` construct from
[shared-cdk](/ops/shared-packages)
(`loanlight-shared/packages/shared-cdk/src/compute/fargate-task-definition-with-datadog.ts`).
It runs a `public.ecr.aws/datadog/agent:latest` sidecar (512 MiB) alongside the app
container.

- App container port `3000`; agent ports `8126` (APM) and `8125` (DogStatsD).
- App container env injected by the construct: `DD_AGENT_HOST=localhost`,
  `DD_TRACE_AGENT_PORT=8126`, `DD_APM_ENABLED=true`, `DD_SERVICE`, `DD_ENV`,
  `DD_VERSION`, `DD_TAGS`.
- DD API key from Secrets Manager secret `LoanLightDatadogAPIKey`.
- `DD_TAGS` always includes `env:<environment>`, `service:<serviceName>`,
  `version:<sha>`, `commit:<sha>`, plus `project:loanlight`, `component:ecs`.
- Logs subscribe to the Datadog Forwarder Lambda via the CFN export
  `DatadogIntegration-DatadogForwarderArn`. App log retention defaults to 30 days,
  agent logs to 7 days.

Octy is the exception: it uses the plain `FargateTaskDefinition` (no sidecar) and
therefore has no APM, as noted above.

## Source maps (LOA-1060)

Source maps let Datadog Error Tracking show unminified, source-resolved stack frames. Each surface uses a different approach:

**loanlight-api backend (service `loanlight-api`)**
Maps are kept in the production Docker image (the backend ECS service is not client-served) and Node is started with `NODE_OPTIONS=--enable-source-maps` so the runtime remaps stack traces to TypeScript source automatically. No `datadog-ci` upload is needed — `dd-trace` performs backend remapping at runtime.

**Portal frontend (service `PortalFrontend`)**
Next.js emits client-side source maps (`productionBrowserSourceMaps: true`). The `@datadog/webpack-plugin` uploads them to Datadog RUM during the CI build, keyed to `service: PortalFrontend` and `releaseVersion: NEXT_PUBLIC_GIT_COMMIT_SHA`. In production, `.map` files are stripped from the Docker image after upload so they are not downloadable from `/_next/static`; INT keeps the maps. `DATADOG_API_KEY` is a build-arg (server-side only, never `NEXT_PUBLIC_`) injected by the `cd-app-prod.yml` GitHub Actions workflow.

<Note>
Prior to LOA-1060, `NEXT_PUBLIC_GIT_COMMIT_SHA` was never set in the CI build, so `datadogRum.init({ version: process.env.NEXT_PUBLIC_GIT_COMMIT_SHA ?? "unknown" })` reported `version:"unknown"` on every RUM event and source map correlation was broken. The fix sets `NEXT_PUBLIC_GIT_COMMIT_SHA=github.sha` in both CD workflows.
</Note>

**loanlight-integrations backend (service `loanlight-integrations`)**
The encompass `Dockerfile.prod` starts Node with `--enable-source-maps`. Maps are already emitted by `tsconfig sourceMap: true` (including the `audit-pdf` workspace package after LOA-1060 enabled `sourceMap: true` on its `tsconfig.json`) and are kept in the container — the encompass service is not publicly served. No upload step is needed for the backend; Datadog APM Error Tracking resolves frames at ingest.

**loanlight-integrations frontend (service `frontend`)**
Next.js emits source maps (`productionBrowserSourceMaps: true`) and the `deploy-web-app-v2-{int,prod}.yml` CI workflows run `datadog-ci sourcemaps upload` keyed on `--service=frontend --release-version=$GIT_COMMIT_SHA` after the build. Prod only: `.map` files are stripped from `dist/` after upload, before the CDK S3 sync, so the public CloudFront bundle ships no source maps. INT keeps them. The CLI is used instead of the webpack plugin because the integrations frontend uses Turbopack (Next 16), which does not support custom webpack plugins.

## LangGraph node spans

Central instrumentation gives every agent node a Datadog span with zero per-agent
code. `instrumentLangGraphNodes()`
(`packages/api/src/langgraph/observability/instrument-langgraph.ts`)
monkey-patches `StateGraph.prototype.addNode` once (idempotent via a `Symbol`),
called at import time in `langgraph.module.ts`.

`traceNode(nodeName, fn)`
(`packages/api/src/langgraph/observability/trace-node.ts`) does the work per node:

- Opens a `langgraph.node` span with `resource = nodeName` and tag
  `langgraph.node=<nodeName>`.
- Lifts state fields onto the span as searchable `langgraph.<key>` tags:
  `loanId`, `auditRunId`, `documentId`, `pipelineId`, `classifySplitRunId`,
  `runId`, `lenderId`.
- **Stall watchdog** — `LANGGRAPH_NODE_STALL_MS` (default 5 min; `0` disables). On a
  stall it tags `langgraph.node.stalled=true` and `error.type=LangGraphNodeStall`
  and force-closes the span so APM records the hang (the underlying work keeps
  running). The per-node override `NODE_STALL_OVERRIDES_MS = { reductoExtract: 95 *
  60_000 }` gives Reducto extraction 95 minutes, just above the 90-minute BullMQ
  appraisal worker lock.
- On error it tags `error`, `error.type`, `error.message`, `error.stack`, plus
  facet tags (`agent`, `step`, `lender`, `run_id`, `audit_id`, `error_class`), and
  emits a structured `langgraph.node.terminal_failure` log event.

<Tip>
To find hung agents, search APM for `langgraph.node.stalled:true`. To find node
failures, search the `langgraph.node.terminal_failure` log event or filter the
`langgraph.node` resource by `error:true`. Cross-reference the same node in
[LangSmith](/ops/langsmith) to see what it was reasoning about.
</Tip>

## Pipeline metrics

Custom point-in-time gauges live under the `loanlight.pipeline.*` namespace
(`packages/api/src/modules/pipeline-metrics/pipeline-metrics.constants.ts`),
emitted via dd-trace DogStatsD on a cadence of `PIPELINE_METRICS_INTERVAL_MS`
(default 15s, floor 1s).

| Metric | Tags |
|---|---|
| `loanlight.pipeline.queue.depth` | `pipeline` + `queue_state` (waiting/active/delayed/failed) |
| `loanlight.pipeline.jobs.in_flight` | `pipeline` |
| `loanlight.pipeline.worker.concurrency` | `pipeline` |
| `loanlight.pipeline.runs.by_status` | `pipeline` + `run_status` |
| `loanlight.pipeline.runs.stuck` | `pipeline` |

`pipeline` is either `appraisal` or `credit`. The constants file also defines
`QUEUE_STATES` and `APPRAISAL_RUN_STATUSES` for zero-filling gauges.

## Monitors as code

Monitors and dashboards for the `loanlight-api` service are committed to `packages/api/observability/datadog/` as JSON; monitors for the `encompass` service live in `packages/encompass/observability/datadog/`. The pattern in both repos is identical.

### loanlight-api monitors

Committed monitor files in `packages/api/observability/datadog/`:

| File | What it pages on |
|---|---|
| `appraisal-reducto-queued-stalled.monitor.json` | Appraisal Reducto queue backlog stalled |
| `bank-statement-reducto-queued-stalled.monitor.json` | Bank-statement Reducto queue backlog stalled |
| `appraisal-reducto-circuit-read-failed.monitor.json` | Circuit snapshot reads fail (break-glass override may not be enforced) |
| `appraisal-reducto-circuit-opened.monitor.json` | Reducto breaker trips OPEN (vendor is down) |
| `appraisal-break-glass-audit-write-failed.monitor.json` | Break-glass mutation applied but audit-trail insert failed |
| `llapi-ecs-memory-oom.monitor.json` | llapi ECS Fargate task memory ≥85% of container limit (warn at 75%), grouped by `task_arn` (LOA-965) |

`monitors.registry.json` contains one entry per `*.monitor.json` basename with `{ "deprecated": boolean }`. Setting `deprecated: true` causes `apply-monitors.ts` to delete that monitor from Datadog on the next apply rather than updating it. The registry shape is validated by `monitor-schema.ts` (Zod) at both PR time and apply time.

`monitor-schema.ts` also enforces that every `*.monitor.json` has **exactly one** of `env:int` or `env:prod` in its `tags` array. For log-alert monitors this tag must also appear in the `query` string (Datadog tags and log-facet filters are different mechanisms).

`apply-monitors.ts` is idempotent by exact `name` match against Datadog's monitor list — no committed id file. Renaming a monitor's `name` field orphans the old one in Datadog; coordinate a deliberate rename with a manual delete of the orphan.

### encompass monitors

Committed monitor files in `packages/encompass/observability/datadog/`:

| File | What it pages on |
|---|---|
| `document-push-dlq-depth.monitor.json` | Encompass document-push DLQ depth exceeds threshold |

The same `monitors.registry.json` / `monitor-schema.ts` / `apply-monitors.ts` pattern applies.

### CI and apply workflows

Every PR and push validates that all `*.monitor.json` files are valid JSON, match the required schema, and have a matching `monitors.registry.json` entry (and vice versa). A new or malformed monitor fails the PR before it can merge.

Two separate GitHub Actions workflows apply monitors to Datadog on push to `main`:

| Workflow | Trigger |
|---|---|
| `.github/workflows/apply-datadog-monitors-api.yml` | Changes to `packages/api/observability/datadog/` |
| `.github/workflows/apply-datadog-monitors-encompass.yml` | Changes to `packages/encompass/observability/datadog/` |

Both can also be triggered manually via `workflow_dispatch`. They run the full reconciliation (apply or delete based on the registry) and self-heal any out-of-band Datadog UI edits. Deliberately **not** wired into the main deploy workflows — a Datadog outage or an invalid `DD_APP_KEY` never blocks an app or infra deploy.

### Dashboards as code

The committed dashboard lives in `packages/api/observability/datadog/`:

- `appraisal-credit-pipelines.dashboard.json` — the source-of-truth definition.
- `apply-dashboard.ts` — applies it (POST on first create, PUT thereafter):

```bash
DD_API_KEY=… DD_APP_KEY=… pnpm exec tsx observability/datadog/apply-dashboard.ts
```

  Supports `--dry-run`, `--site` (default `datadoghq.com`), `--only <basename>` to apply a single dashboard.
- `README.md` — the operational runbook, including the APM-vs-log service tag note.
