`loanlight-shared` is a pnpm-workspace monorepo that publishes versioned, scoped
npm packages under `@loanlight-engineering/*` to **GitHub Packages**
(`https://npm.pkg.github.com`). Every other LoanLight repo consumes them as normal
versioned dependencies. The repo is the single source of truth for cross-repo
TypeScript contracts, reusable CDK constructs, and LocalStack dev tooling.

<Note>
The repo now tracks **all seven packages** at their current versions —
`shared-types@1.13.0`, `shared-cdk@1.15.0`, `shared-local@0.1.0`,
`shared-auth@0.3.0`, `shared-api-auth@0.2.0`, `shared-ui@0.8.0`, and
`shared-lambda-auth@0.2.0`. (Older snapshots of this doc warned that only three
packages existed on disk and that types like `RelatedTrace` were published-only;
that is no longer true — `RelatedTrace` lives in `shared-types/src/octy.types.ts`,
and the `cd.yml` publish matrix releases every package.) The `shared-ui` package
lives under `packages/ui/` (its package name is still
`@loanlight-engineering/shared-ui`). A consumer's installed version can still lag
the latest publish, so cross-check the consumer's `package.json` when a contract
looks missing.
</Note>

## Publishing and authentication

All packages publish to GitHub Packages under the `@loanlight-engineering` scope
with `publishConfig.access: "restricted"`. The `.npmrc` wires the registry and
token:

```ini
@loanlight-engineering:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${GITHUB_PACKAGES_TOKEN}
strict-peer-dependencies=false
auto-install-peers=true
```

<Info>
Every consumer needs `GITHUB_PACKAGES_TOKEN` (a GitHub PAT) set in the
environment: `read:packages` to install, `write:packages` to publish. Without it,
`pnpm install` cannot resolve `@loanlight-engineering/*`.
</Info>

### Versioning policy

Every change to a package must include a SemVer bump in that package's
`package.json`:

- `patch` — non-breaking additions (new optional fields, fixes)
- `minor` — new features / new exports
- `major` — breaking changes (removed or renamed fields)

After bumping, consumers update their dependency version and reinstall. Publishing
is idempotent and gated on a version bump: the `cd.yml` workflow (push to `main`,
filtered to `packages/**`) runs `npm view <name>@<version>` and publishes only if
that version does not already exist, otherwise it skips.

<Note>
The `cd.yml` publish matrix covers **all seven** packages: `shared-types` first
(downstream packages depend on its build), then `shared-cdk`, `shared-local`,
`shared-ui` (`packages/ui`), `shared-lambda-auth`, `shared-auth`, and
`shared-api-auth`. Each entry is gated on a version bump, so an unchanged package
is skipped.
</Note>

### Local dev override

To point a consumer at local source, use pnpm overrides:

```json
{ "pnpm": { "overrides": {
  "@loanlight-engineering/shared-types": "file:../loanlight-shared/packages/shared-types",
  "@loanlight-engineering/shared-cdk": "file:../loanlight-shared/packages/shared-cdk"
}}}
```

## shared-types: the cross-repo contract

`shared-types` is the contract the audit pipeline produces and the portals
consume. The audit pipeline emits `AuditRunResults`, `DataConsistencyResponse`,
`DocumentHealthResponse`, `PropertyDataResult` / `RedFlagResult`,
`UrlaRulesEngineResult`, and more; the portals render them into tables.

`shared-types@1.13.0` exports these modules (all re-exported from `src/index.ts`):
`field-value.types`, `table.types`, `audit-response.types`, `credit.types`,
`common.types`, `api-response.types`, `property-data.types`,
`urla-completeness.types`, `octy.types`, `portal-audits.types`, `role.types`,
`access-policy.types`, and `epc-document-events.types`. The contract-bearing ones:

<AccordionGroup>
  <Accordion title="credit.types — the credit-evaluation contract (LOA-1461)">
    Producer: loanlight-api. Consumers: the portal and the integrations frontend
    (the shared `CreditLiabilitiesView`).

    - `CreditComplianceResult` (rule, category, `CreditRuleOutcome`, message),
      `CreditReportPayload` (incl. `auditedCreditDocumentFileName`), and
      `PortalCreditReportPayload = Omit<CreditReportPayload, "triggeredBy" | "inputS3Key">`
      — the redacted wire shape both EPC and portal audit endpoints project.
    - `CreditAuditBlock` (a `BaseAuditResponse`) and `CreditTableRow`.
    - `AuditRunResults.credits?: CreditAuditBlock[]` and
      `AuditSummaryResponse.creditEvaluationFindingsCount?` were added (optional, for
      backwards compat with 1.8.0 consumers) so the plural-credit shape reaches the
      shared contract.
  </Accordion>
  <Accordion title="epc-document-events.types — the EPC document round-trip events">
    The cross-service contract for the classify → carve → materialize round trip.
    Exports `DOCUMENTS_CLASSIFIED_DETAIL_TYPE` / `DOCUMENTS_MATERIALIZED_TYPE`, the
    `DocumentsClassifiedEvent` (outbound, api → integrations) and
    `DocumentsMaterializedEvent` (inbound return leg, integrations → api). The api
    messaging consumer validates the parsed EventBridge `detail` against a Zod
    schema mirroring `DocumentsMaterializedEvent` before persisting split locations.
  </Accordion>
  <Accordion title="octy.types — the Octy bug-report contract">
    Producer: loanlight-api. Consumers: integrations frontend and the Octy
    dashboard.

    - `BUG_REPORT_AGENT_TYPES = ["URLA-Extraction", "PII-Validation",
      "Document-Health", "Property-Data", "Property-Red-Flags"]` and the derived
      `BugReportAgentType`.
    - **`RelatedTrace`** = `{ trace_id: string; agent_type: BugReportAgentType;
      run_name: string }` — links an Octy issue to a LangSmith run, where
      `run_name` is typically `"{agentType}-Loan-{loanId}"`. It lives in
      `shared-types/src/octy.types.ts`.
    - `BugReportPayload` = `{ description: string; selected_agent_types:
      BugReportAgentType[] }`.
    - `BugReportResult` = `{ id, audit_run_id, loan_id, description, intake_status,
      created_at }`.
    - `BugReportAgentOption` and `BUG_REPORT_AGENT_OPTIONS` map UI labels
      ("1003 Completeness", "Data Consistency", "Document Health", "Property
      Data") to agent types.
  </Accordion>
  <Accordion title="role.types">
    `Role = "client_admin" | "client_user" | "read_only" | "super_admin"`.
  </Accordion>
  <Accordion title="portal-audits.types">
    `SectionName = "document-health" | "data-consistency" | "property-data" |
    "urla-completeness" | "checks-completed"`; `SECTIONS`; `PortalAuditsErrorCode`
    (`missing_lender_id | missing_token | invalid_token | expired_token |
    audit_scope_mismatch | audit_not_found | upstream_error`); `PortalAuditsError`;
    `ChecksCompletedResponse = { checksCompleted: number; checksTotal: 4 }`.
  </Accordion>
</AccordionGroup>

The audit response/table types (`api-response.types`, `common.types`,
`table.types`, `audit-response.types`, `property-data.types`,
`urla-completeness.types`) are the original contract surface; the credit, EPC-event,
octy, role, portal-audits, access-policy, and field-value modules were layered on
since.

## shared-cdk: reusable constructs

`shared-cdk` is consumed by each repo's `infra/` app to build identical, hardened
AWS resources. Its peer dependencies pin `aws-cdk-lib@2.214.0`, `constructs@^10`,
and `@aws-sdk/client-eventbridge@^3.916.0`, and it ships CloudFormation snapshot
tests. See [Infrastructure and CI/CD](/ops/infrastructure) for how the constructs
compose into stacks, and [Datadog](/ops/datadog) for the
`FargateTaskDefinitionWithDataDog` sidecar.

Construct families (`shared-cdk/src/`): `base` (`BaseStack`), `compute`
(`FargateService`, `FargateTaskDefinition`, `FargateTaskDefinitionWithDataDog`,
`EcsSecurityGroup`), `networking` (`NetworkStack`, `NetworkLoadBalancer`),
`api-gateway` (`LoanLightApiGateway`, `ApiGatewayDomain`, `ApiGatewayVpcLink`,
`AuthLambda`, CORS helpers), `messaging` (`SqsQueue`,
`publishEventToEventBridge`), `database` (`DatabaseBastionHost`), `lambda`
(`datadog-lambda-config`), `monitoring` (Datadog Lambda/RDS log-subscription
constructs), `environment` (`parseEnvironment`), and `common`
(`applyStandardTags`, snapshot helpers).

## shared-local: LocalStack tooling

`shared-local` standardizes the LocalStack bootstrap so api, integrations, and
octy all talk to the same local event-driven topology. `setupLocalstack()`
idempotently provisions:

- S3 buckets `loanlight-integration-files` and `loanlight-web-app-local`.
- SQS queues `audit-requested-events`, `health-check`,
  `transaction-event-handling`, `handle-aus-updates`.
- EventBridge bus `loanlight-events` with the
  `loanlight-encompass-audit-requested-rule` (source `loanlight.encompass`,
  detail-type `Audit Requested` → `audit-requested-events`) and
  `loanlight-aus-all-events-rule` (source `loanlight.aus` → `handle-aus-updates`)
  rules.
- A `.init-complete` marker object.

It also exports docker/env helpers and ships the LocalStack `docker-compose` file
(`localstack/localstack:latest`, container `loanlight-localstack`, services
`sqs,s3,events,secretsmanager`).
