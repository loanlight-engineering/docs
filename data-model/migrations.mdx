LoanLight uses `drizzle-kit` to generate and apply schema migrations. The workflow is: edit the schema TypeScript, generate a numbered SQL migration from the diff, review it, and apply it. **Never hand-write migration SQL from scratch** — let `generate` author it from the schema, then audit the output.

## The scripts

All scripts live in `loanlight-api/packages/api/package.json`.

| Script | Command | Purpose |
|---|---|---|
| `db:generate` | `drizzle-kit generate` | Diff `schema/index.ts` against the journal and emit a new numbered SQL migration in `db/migrations/`. **The primary authoring command.** |
| `db:migrate` | `drizzle-kit migrate` | Apply pending migrations. |
| `db:check` | `drizzle-kit check` | Detect collisions / consistency issues in the migration set. |
| `validate:migrations` | `ts-node scripts/validate-migrations.ts` | CI lint (see below). |
| `test:migration` | `jest --config ./jest.containers.config.json` | Live-Postgres (testcontainers, `postgres:16-alpine`) migration tests. |

## Authoring a migration

<Steps>
<Step title="Edit the schema">
Change the table file under `src/database/schema/` and make sure it is re-exported from `index.ts`.
</Step>
<Step title="Generate">
Run `db:generate`. Drizzle diffs the schema against `db/migrations/meta/_journal.json` and writes the next numbered `.sql` file (e.g. `0180_*.sql`).
</Step>
<Step title="Review the SQL">
Read the generated SQL. Confirm it does what you intended and did not emit an unexpected narrowing revert (a common symptom of schema/migration drift — see canaries below).
</Step>
<Step title="Validate and test">
Run `validate:migrations` and `test:migration`. Register P0 migrations in the container test suite.
</Step>
<Step title="Apply">
Run `db:migrate` against the target environment.
</Step>
</Steps>

The journal (`db/migrations/meta/_journal.json`, version 7, postgresql) tracks every migration from `0000_initial_schema` through the latest (`0242_add_closing_date_inconsistent_issue_kind` at time of writing).

<Warning>
**The migration runner owns the outer transaction.** Do not add raw `BEGIN;` / `COMMIT;` inside a migration `.sql` — it broke atomicity in `0143`.
</Warning>

## CI lint rules (validate:migrations)

`scripts/validate-migrations.ts` enforces:

- Journal `when` timestamps are strictly increasing.
- Every `.sql` file has a journal entry and vice-versa.
- Enum value migrations must use `ADD VALUE IF NOT EXISTS` so they are idempotent on non-transactional re-runs.

## Migration-only constructs

Some constraints cannot be expressed in Drizzle and live only in the SQL, with a comment in the schema file where they would otherwise go:

- `UNIQUE (rule_set_id, code, COALESCE(loan_program_id, nil_uuid))` on `appraisal_rules` / `credit_rules` / `income_rules`.
- `UNIQUE (lender_id, rule_code, COALESCE(loan_program_id, nil_uuid))` on `income_lender_rule_overrides`.

## Drift canaries

Because `drizzle-kit generate` re-derives SQL from the schema, a schema file that drifts from the latest migration causes a silent revert on the next generate. A set of spec canaries pins the high-risk cases (`src/database/schema/*.spec.ts`):

| Spec | What it pins |
|---|---|
| `timestamp-invariants.spec.ts` | Every timestamp column uses `{ withTimezone: true, mode: "string" }`. Omitting it breaks API date formats. |
| `index-direction.spec.ts` | DESC indexes must be declared `` sql`${t.col} DESC` `` or generate reverts prod to ASC. |
| `fk-ondelete-policy.spec.ts` | Drizzle `onDelete` mirrors the migration `ON DELETE` action exactly (e.g. `credit_rules.source_template_id` is not an FK). |
| `fk-repoint.spec.ts` | `relations.ts` repoints match schema `foreignKeys` on both sides. |
| `operator-check-drift.spec.ts` | The appraisal operator union (incl. `abs_less_than*`) matches the latest migration. |
| `migration-0161-classification-split-modes.spec.ts` | The classification-mode CHECK set. |
| `migration-0203-agent-platform.spec.ts`, `migration-0204-audit-runs-not-null.spec.ts` | The agent-platform and `audit_runs` NOT NULL migrations. |
| `migration-0202-audit-runs-lender-anchor.spec.ts` | The `audit_runs` lender-anchor EXPAND migration. |
| `migration-0201-asset-runs-loanless-runnum.spec.ts` | The `asset_runs` loanless `run_number` uniqueness. |
| `result-readers.spec.ts` | `red_flag_results` reads use the triple `orderBy`. |

<Note>
The timestamp invariant is why nearly every timestamp column is declared `timestamp(..., { withTimezone: true, mode: "string" })`. A `pg.types.setTypeParser` override returns raw ISO strings (no JS `Date` coercion) for timestamptz/timestamp OIDs.
</Note>

## Squashing migrations (quarterly) — LOA-1495

The migration chain only grows. `pnpm migrate:squash <cutoff-idx>` collapses the settled
prefix (entries `0..cutoff` in journal order) into a **baseline set** without touching the
database schema:

- One transactional `0000_baseline.sql` (all non-enum bodies, wrapped in `BEGIN;…COMMIT;`).
- N isolated enum baselines — each `ALTER TYPE … ADD VALUE` is split into its own file because
  PostgreSQL forbids `ADD VALUE` inside a transaction.

After collapsing, the script rewrites the journal head, deletes the collapsed `.sql` files,
preserves and renumbers the tail (keeping each entry’s original `version`/`breakpoints`/`when`
— only `idx` is rewritten), and emits an **append-only reconciliation manifest**
(`meta/_squash_manifest.json`). Re-running the same cutoff is a manifest-detected no-op.

The full operator flow:

```bash
pnpm migrate:squash <cutoff-idx>   # collapse entries[0..idx] → baseline set
pnpm validate:migrations            # exit 0; check 6 validates the manifest
pnpm squash:equivalence-proof       # schema + pg_dump diffs are empty
pnpm db:generate                    # expect "No schema changes"
pnpm test && pnpm test:migration    # both green
```

The cutoff is a journal **idx** (not a filename tag). Run `migrate:squash` on a branch freshly
cut from `main` so the captured pre-squash base commit stays a durable ancestor.

**Production is a zero-schema-DDL no-op.** Deployed databases already applied the collapsed
prefix. On the first CD deploy after merge, the `DbMigrate` Lambda reconciles the
`drizzle.__drizzle_migrations` ledger — an idempotent `INSERT` of the baseline-set hashes.
Superseded rows are retained (rollback safety). No application schema DDL runs. Every subsequent
deploy sees the same hashes and skips. End-to-end: **author + verify locally → commit →
merge → first INT deploy reconciles INT ledger → promote → first PROD deploy reconciles PROD
ledger.**

<Warning>
Never hand-collapse migrations or hand-edit `meta/_journal.json`, the
`drizzle.__drizzle_migrations` ledger, or `meta/_squash_manifest.json`. Run
`pnpm migrate:squash` and let it do this deterministically. After pulling a squash onto a
populated local DB, drop and rebuild, clearing the **`drizzle` schema** (not just `public`).
</Warning>

## Two migration ledgers

Do not confuse:

- The **Drizzle DDL journal** (`_journal.json` + the numbered `.sql` files) — schema structure.
- The **`migration_runs` table** — a ledger for *data-backfill* migration scripts (PK `id`, `name`, `status` CHECK `pending | complete | failed`, `counts` jsonb, `notes`, `started_at`, `completed_at`). This tracks application-level backfills, not DDL.
