LoanLight expresses enumerated values three ways: Postgres native `pgEnum` types (`enums.ts`), TypeScript `const`-array unions backed by CHECK constraints, and CHECK constraints declared inline on columns. The CHECK-backed sets are effectively enums but live in migrations, so they drift more easily — several have dedicated spec canaries (see [migrations](/data-model/migrations)).

## Run status vocabularies

The two run-ledger families use **different terminal-success spellings**. Get this wrong and writes silently miss the status filter.

<Warning>
`agent_runs.status` terminal success is **`complete`**. The standalone `appraisal_runs` / `credit_runs` / `income_runs` use **`completed`**. `audit_runs.status` and the newer `asset_runs.status` use **`complete`**.
</Warning>

| Table | Status set |
|---|---|
| `audit_runs.status` | `pending | processing | complete | failed` |
| `agent_runs.status` | `pending | processing | complete | failed | skipped` (CHECK `agent_runs_status_check`) |
| `credit_runs.status` | `pending | running | completed | failed | reviewed` |
| `appraisal_runs.status` | `pending | running | completed | failed | reviewed` (service vocabulary) |
| `income_runs.status` | default `pending` (`completed` on success) |
| `asset_runs.status` | `pending | processing | complete | failed` (CHECK `asset_runs_status_check`; note `complete`, not `completed`) |
| `loans.processing_status` | `processing | ready_for_audit | ready | error` |
| `processing_jobs.status` | `queued | running | succeeded | failed` |
| `classify_split_runs.status` | `pending | running | complete | failed | skipped | partial` |
| `migration_runs.status` | `pending | complete | failed` |

## Finding-type unions (bank-statement pipeline)

`findings.finding_type` (the `borrower_files` family, 16-value CHECK `finding_type_check`): `large_deposit`, `anomaly`, `fraud_flag`, `income_classification`, `validation_balance_mismatch`, `validation_date_out_of_range`, `validation_duplicate`, `validation_invalid_amount`, `validation_missing_field`, `fraud_pdf_integrity`, `fraud_font_consistency`, `fraud_continuity`, `llm_source_mismatch`, `nsf_overdraft`, `income_trend`, `negative_balance`. Migration `0209` added `nsf_overdraft` / `income_trend` and re-asserted `validation_balance_mismatch` / `validation_duplicate`. Migration `0244` (LOA-1688) added `negative_balance` (emitted when any statement's `closing_balance` or running balance goes negative). `findings.severity`: `info | warning | critical | error`.

## Bank-statement and asset pipeline

These CHECK-backed sets belong to the `borrower_files` bank-statement family and the asset subsystem. The asset pgEnums (`asset_source`, `asset_type_normalized`, `asset_class`, `asset_account_state`, `large_deposit_threshold_mode`, `large_deposit_override_mode`) are listed in the [pgEnum table](#postgres-native-enums-pgenum) above.

| Column | Values |
|---|---|
| `bank_statements.extraction_method` | `pdfjs | azure | hybrid | llamaparse | reducto` (`reducto` added in migration `0194`) |
| `borrower_files.large_deposit_threshold_mode` | `fixed | pct_avg_qualifying_income | pct_avg_gross_deposits` (text + CHECK, migration `0186` — distinct from the asset-side pgEnum of the same name) |
| `transactions.type` | `credit | debit` |
| `asset_conditions.status` | `open | satisfied | waived | superseded` |

<Note>
`transactions.category` is **free text** (Ntropy-derived), not a CHECK. Migration `0208` normalized the legacy value `missing account holder information` to `not enough information` in both `category` and the single-element `category_hierarchy`.
</Note>

<Note>
CHECK-backed value sets are not pgEnums — they live in migration SQL. Drizzle's `generate` can emit a narrowing revert if the schema file's union drifts from the latest migration, so several of these are pinned by spec canaries (`operator-check-drift.spec.ts`, `migration-0161-classification-split-modes.spec.ts`). See [migration workflow](/data-model/migrations).
</Note>
