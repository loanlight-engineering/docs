These modules back the internal-admin and observability surface: the tenant audit log, the lender YAML config viewer, the shadow-audit divergence monitor, lender-config cache invalidation, per-finding feedback, and JIT provisioning through the WorkOS webhook. Most are super-admin only and use [`PortalAuthGuard`](/portal/api-auth); the WorkOS webhook is signature-verified instead.

## portal-feedback

Controller: `portal-feedback/portal-feedback.controller.ts` at `api/v1/portal/feedback`. `@UseGuards(PortalAuthGuard)`. Captures per-finding feedback from reviewers.

### POST /api/v1/portal/feedback

Record feedback on a finding. Returns HTTP 201. Uniqueness is keyed including `audit_run_id`.

<ParamField body="auditRunId" type="uuid" required>The audit run.</ParamField>
<ParamField body="findingId" type="string" required>The finding within the run.</ParamField>
<ParamField body="findingType" type="string" required>
One of `pii_validation`, `doc_expiration`, `doc_completeness`, `property_data`, `red_flag`, `urla_rules`, `appraisal`.
</ParamField>
<ParamField body="verdict" type="string" required>`correct` or `incorrect`.</ParamField>
<ParamField body="reason" type="string">
One of `missing_data`, `wrong_rule`, `reasoning_error`, `false_positive`.
</ParamField>
<ParamField body="comment" type="string">Free text, ≤500 characters.</ParamField>

### GET /api/v1/portal/feedback?audit_run_id=&lt;uuid&gt;

List feedback for a run. `audit_run_id` requires `ParseUUIDPipe`. Ordered `created_at, id`.

## release-notes admin

Controller: `release-notes/release-notes-admin.controller.ts` at `api/v1/portal/admin/release-notes`. Every route requires `super_admin` with internal lender — see [release notes](/portal/release-notes) for the full lifecycle model and data schema.

| Method / path | Purpose |
|---|---|
| `POST /api/v1/portal/admin/release-notes` | Create a draft (`title`, `bodyMarkdown`, `versionTag?`, `releaseDate?`). HTTP 201. |
| `GET /api/v1/portal/admin/release-notes` | List all notes. Query: `?page`, `?limit`, `?status` (`draft\|published\|archived`). |
| `GET /api/v1/portal/admin/release-notes/:id` | Fetch one note (any status). |
| `PATCH /api/v1/portal/admin/release-notes/:id` | Edit content. `slug` and `published_at` are immutable; archived notes: 403. |
| `POST /api/v1/portal/admin/release-notes/:id/publish` | `draft → published`. |
| `POST /api/v1/portal/admin/release-notes/:id/unpublish` | `published → draft`. |
| `DELETE /api/v1/portal/admin/release-notes/:id` | Soft-archive (`status → archived`, `deleted_at` set). |
| `POST /api/v1/portal/admin/release-notes/:id/restore` | `archived → prior status`. |

Every mutation writes a `portal_audit_log` row in the same transaction (CUST-11 evidence channel). `VALID_ACTIONS` in `PortalAuditLogService` includes `release_note.created`, `release_note.updated`, `release_note.published`, `release_note.unpublished`, `release_note.archived`, and `release_note.restored`.

## portal-audit-log

Controller: `portal-audit-log/portal-audit-log.controller.ts` at `api/v1/portal/audit-log`. Class-level `@Roles("super_admin")`.

### GET /api/v1/portal/audit-log

Paginated log for the effective lender.

<ParamField query="action" type="string">Filter by action.</ParamField>
<ParamField query="startDate" type="string">`YYYY-MM-DD` or full ISO with `Z`/offset; normalized to UTC bounds.</ParamField>
<ParamField query="endDate" type="string">Same format as `startDate`.</ParamField>
<ParamField query="page" type="integer">Page number.</ParamField>
<ParamField query="limit" type="integer">Page size.</ParamField>

Results are ordered `desc(createdAt), desc(id)`.

<Warning>
`portal-audit-log` is read-only by design — there is intentionally no write route. Writes happen only internally via `PortalAuditLogService.create()`, called by other modules. Its `VALID_ACTIONS` includes `agency_adoption.enabled` / `.disabled`, `audit.shadow_diff_viewed`, `audit.shadow_diff_replay`, plus currently-unused `rule.toggled`, `feature_flag.toggled`, and `admin.client_created`.
</Warning>

## yaml-viewer

Controller: `yaml-viewer/yaml-viewer.controller.ts` at `api/v1/internal-admin/yaml-configs`. Class-level `@Roles("super_admin")`. Sections are validated against `{ document_expiration, red_flags, urla_completeness_rules }`. See [YAML config](/lender-config/yaml-config).

| Method / path | Purpose |
|---|---|
| `GET /api/v1/internal-admin/yaml-configs/:lenderId/:section` | Active config (relies on a partial unique index for determinism) |
| `GET /api/v1/internal-admin/yaml-configs/:lenderId/:section/history` | Version history (ordered `desc(updatedAt), desc(version), desc(id)`) |
| `GET /api/v1/internal-admin/yaml-configs/:lenderId/:section/versions/:version` | A specific version (`ParseIntPipe`) |

## shadow-audit-diff

Controller: `shadow-audit-diff/shadow-audit-diff.controller.ts` at `api/v1/portal/admin/shadow-audit-diffs`. All `@Roles("super_admin")`. This monitors divergence between real and shadow audit runs.

| Method / path | Query / body | Purpose |
|---|---|---|
| `GET /api/v1/portal/admin/shadow-audit-diffs` | `page, limit, diverged(true\|false), correlationId, lenderId, section, asymmetry` | List diff records |
| `GET /api/v1/portal/admin/shadow-audit-diffs/:id` | — | Diff detail; writes `audit.shadow_diff_viewed` to the audit log; 404 if missing |
| `POST /api/v1/portal/admin/shadow-audit-diffs/replay` | `{ since, until }` (ISO, ≤7 days, `until > since`) | Re-run diffs over a window → `{ ok: true }` (HTTP 200); writes `audit.shadow_diff_replay` |

`shadow-audit-diff.service.ts` runs a watermark-driven tick that compares real vs shadow runs section by section using outcome-only normalizers (`normalizers/urla-completeness-rules.ts`, `red-flags.ts`, `document-expiration.ts`, `NORMALIZER_VERSION v2`) and canonical hashing (`lib/canonical-hash.ts`). Watermark advance is capped to avoid dropping tied or pending rows.

## admin/cache-invalidation

Controller: `admin/cache-invalidation.controller.ts` at `api/v1/internal-admin`. Class-level `@Roles("super_admin")`.

### POST /api/v1/internal-admin/clear-cache

Clears lender-config caches. Returns HTTP 200 `{ cleared: true }`. It clears both the DI `LenderConfigLoaderService` cache and the module-level `sharedLenderConfigLoader` singleton — the singleton is what the propertyRedFlags and documentHealth agents use.

## workos-webhook: JIT provisioning

Controller: `portal-lenders/workos-webhook.controller.ts` at `api/v1/workos/webhooks`. Not auth-guarded; it verifies the `workos-signature` header via `ApiAuthService.constructWebhookEvent` (missing signature → 401) and returns `{ received: true }`.

The controller was refactored to a single `dispatch()` switch with structured logging (#2277): it logs `workos_webhook_received` on entry and `workos_webhook_handler_failed` if a handler throws. `createLenderFromWorkosOrg` logs `workos_org_created_name_conflict` when a same-named lender is already bound to a different WorkOS org (see [the three-branch provisioning logic](/portal/api-lenders-users-programs)).

The webhook is the just-in-time provisioning path: WorkOS organization and user lifecycle events create and update lenders and users without an explicit portal action.

| Event | Handler | Effect |
|-------|---------|--------|
| `organization.created` | `createLenderFromWorkosOrg` | Provision a lender from a new WorkOS org; also publishes `LenderUpserted` EventBridge event (LOA-1014) |
| `organization.updated` | `updateLenderWorkosOrgName` | Sync the lender name; also publishes `LenderUpserted` EventBridge event (LOA-1014) |
| `user.created` | `linkWorkosUser` | Link a WorkOS user to a portal user |
| `user.updated` | `updateLastSignIn` | Record last sign-in |
| `organization_membership.created` / `.deleted` | membership handlers | Add/remove tenant membership |

The `LenderUpserted` (`LENDER_UPSERTED_DETAIL_TYPE`) and `LenderClientLinked` (`LENDER_CLIENT_LINKED_DETAIL_TYPE`) events are published on the `loanlight.aus` EventBridge bus via `EventPublishingService` so that `loanlight-integrations` (LOA-1015) can mirror lender identity without re-deriving it from raw WorkOS payloads. A `LenderClientLinked` event is also emitted when a `lender_auth_clients` row is inserted. Every publish call is `try/catch`-wrapped; a publish failure never fails the webhook response.

See [lenders, users, and programs](/portal/api-lenders-users-programs) for the full webhook table and the user-management endpoints it complements.

## Audit-results projections and summary counts

The `portal-audits` projection service (`services/audit-data-projection.service.ts`) feeds the audit-results sections, and several count/summary fixes landed here:

- **Consistent rows are now included.** `getDataConsistency` and `getDocumentHealth` call the mapper with `includeNonIssues=true`, so clean (non-issue) rows reach the projection. Without it, `mapPiiValidationData` filtered `WHERE has_issue=true`, dropped every consistent row, and a clean loan rendered "0 of 0 consistent" instead of "X of Y consistent" (#2494, LOA-1619). The `issueCount` re-filters on `hasIssue`, so it stays correct.
- **Summary chips count only issues.** The PII / document-validation summary chips count issue rows only, not the now-included consistent rows (#2493).
- **No more synthetic `doc_expiration` rows.** `document_completeness` findings are no longer double-written as synthetic `doc_expiration` rows (#2494).

<Note>
The dashboard's `totalIssues` summary (`portal-audits-list.controller.ts`) sums `pii_validation_results`, `doc_expiration_results` (where `issue_kind <> 'valid'`), and non-transient `audit_errors` across all non-shadow runs. The double-write removal is a **write-side, go-forward fix**: audits run before it keep their inflated `doc_expiration` counts, so legacy runs may still over-report. `totalIssues` is also cumulative across re-audits by design (it matches the "Issues Found" tile), not an outstanding-per-loan count.
</Note>
