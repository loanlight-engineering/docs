These modules manage tenants (lenders), their users, and their loan programs. All endpoints are guarded by [`PortalAuthGuard`](/portal/api-auth) and scope tenant data by `ctx.effectiveLenderId`. The WorkOS webhook (`api/v1/workos/webhooks`) is the one exception — it is not auth-guarded and instead verifies a WorkOS signature.

<Info>
Rule-override CRUD on `portal-lenders` was removed in LOA-820 T3. The `lender_rule_overrides` table is being dropped, and the only honored include on `GET /lenders/:id` is `usage`. The appraisal rule set is now the single source of truth for rule customization — see [rules authoring](/portal/api-rules-authoring).
</Info>

## portal-lenders

Controller: `portal-lenders/portal-lenders.controller.ts` at `api/v1/portal/lenders`. Class-level `@UseGuards(PortalAuthGuard, PortalThrottlerGuard)`, default `@Roles("super_admin")`, throttle 100/min. Many GET/PATCH methods override roles to also allow tenant roles so non-admin users can render read-only views.

Two access helpers gate sensitive paths:

- `requireInternal(ctx)` — super-admin / internal only.
- `requireLenderAccess(ctx, lenderId)` — internal, or own-tenant; throws `403` on a foreign lender.

### Lender CRUD

#### GET /api/v1/portal/lenders

List lenders. Query `ListLendersQueryDto`; ordered `asc(name), asc(id)`. Role: `super_admin`.

#### POST /api/v1/portal/lenders

Create a lender. Role: `super_admin`.

<ParamField body="CreateLenderDto" type="object" required>
Includes an `InitialAdminDto` for the first admin user. The initial admin role excludes `read_only`.
</ParamField>

#### GET /api/v1/portal/lenders/:id

Get a lender. Role: `super_admin`. `?include=usage` is the only honored include.

#### PATCH /api/v1/portal/lenders/:id

Update a lender (`UpdateLenderDto`). Role: `super_admin`.

#### POST /api/v1/portal/lenders/:id/logo

Multipart logo upload (`FileInterceptor`). Filenames are sanitized to `[a-zA-Z0-9._-]`. Role: `super_admin`.

### Agency adoptions (Fannie/Freddie rule packs)

| Method / path | Roles | Purpose |
|---|---|---|
| `GET /api/v1/portal/lenders/:lenderId/agency-adoptions` | tenant + super_admin | Lazily materializes via `helpers/ensure-lender-agency-adoptions.ts` (idempotent `ON CONFLICT DO NOTHING`) |
| `POST /api/v1/portal/lenders/:lenderId/agency-adoptions` | requireInternal | `CreateAgencyAdoptionDto { agencySource: fannie_mae \| freddie_mac, isEnabled? }`; pg `23505` → `ConflictException` |
| `PATCH /api/v1/portal/lenders/:lenderId/agency-adoptions/:agencySource` | client_admin, super_admin | `UpdateAgencyAdoptionDto { isEnabled }` |

The `PATCH` upserts while preserving the first `adoptedBy` (write-once) and runs inside a `db.transaction` that also writes a `portal_audit_log` row (`agency_adoption.enabled` / `.disabled`). Valid sources are `fannie_mae` and `freddie_mac`.

## portal-auth: session identity

Controller: `portal-auth/portal-auth.controller.ts` at `api/v1/portal/auth`.

| Method / path | Roles | Purpose |
|---|---|---|
| `GET /api/v1/portal/auth/me` | any authed | Returns `AuthMeResponse` (effective user + lender, `isInternal`, `isImpersonating`) |
| `POST /api/v1/portal/auth/impersonate` | super_admin | `ImpersonateDto { lenderId }` → `ImpersonationResponse` |

| `GET /api/v1/portal/auth/widget-token` | client_admin, super_admin | WorkOS widget token for the lender's WorkOS org; rejected during impersonation (`BadRequestException`) |

## portal-users: tenant user management

Controller: `portal-auth/portal-users.controller.ts` at `api/v1/portal/users`. Class-level `@Roles("client_admin", "super_admin")`.

| Method / path | Purpose |
|---|---|
| `GET /api/v1/portal/users` | List users for `effectiveLenderId` |
| `POST /api/v1/portal/users/invite` | `InviteUserDto { email, lenderId? }`; internal admins may target any `lenderId`, others are forced to the effective tenant; sends a WorkOS invite |
| `PATCH /api/v1/portal/users/:id/role` | `ChangeRoleDto { role }`, validated against `VALID_ROLES` |
| `POST /api/v1/portal/users/:id/deactivate` | Deactivate user |
| `POST /api/v1/portal/users/:id/reactivate` | Reactivate user |
| `DELETE /api/v1/portal/users/revoke-invite` | `RevokeInviteDto { email, lenderId? }`; revokes a pending WorkOS invite |

`managementLenderId(ctx)` returns `undefined` for internal, non-impersonating admins (bypassing the tenant `lender_id` filter for cross-lender management); otherwise it returns `effectiveLenderId`.

## workos-webhook: inbound WorkOS events

Controller: `portal-lenders/workos-webhook.controller.ts` at `api/v1/workos/webhooks`. Not auth-guarded.

#### POST /api/v1/workos/webhooks

Verifies the `workos-signature` header via `ApiAuthService.constructWebhookEvent`, then dispatches events. A missing signature returns `401`. The handler returns `{ received: true }`.

| Event | Handler |
|-------|---------|
| `organization.created` | `createLenderFromWorkosOrg` |
| `organization.updated` | `updateLenderWorkosOrgName` |
| `user.created` | `linkWorkosUser` |
| `user.updated` | `updateLastSignIn` |
| `organization_membership.created` / `.deleted` | membership handlers |

<Note>
`createLenderFromWorkosOrg` was hardened to three branches (#2277, LOA-1178):

1. **Already linked** — a lender already bound to this `workos_organization_id` is an idempotent no-op (it just heals the org→client mapping).
2. **Same-named, null org** — `lenders.name` is `UNIQUE`, so a same-named lender with no org is atomically claimed for this org (a conditional `WHERE` closes the TOCTOU race). If that lender is already bound to a *different* org, it logs `workos_org_created_name_conflict` and returns the existing row rather than clobbering it — the webhook is still acked so it does not retry-loop forever.
3. **Brand new** — otherwise a fresh lender is created.

This fixes the prior path where a same-named org would hit the `lenders.name` UNIQUE constraint and `500`.
</Note>

Just-in-time provisioning details are on [internal admin and observability](/portal/api-admin-observability).
