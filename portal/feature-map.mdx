The portal uses the Next.js App Router with four route groups under `src/app/`. This page maps the routes, the feature flags that gate them, and the redirects that catch legacy or non-page paths.

## Feature flags

The `FeatureFlags` object (`src/lib/feature-flags.ts`) has these keys:

`classification`, `appraisal`, `agencyRules`, `chat`, `settingsPrograms`, `bankStatements`, `creditReview`, `keyFieldReview`, `enableGuidelineUploads`, `darkMode`, `updatedAppraisalEnabled`, `dscrAgent`, `statementAnalyzer`, `documentRouting`, `releaseNotes`.

`getFeatureFlags(lenderId)` GETs the backend `/lenders/${lenderId}/feature-flags`, which returns `{ featureKey, enabled }[]`; the result is transformed via `KEY_MAP` onto an env-default base. An empty array or any failure falls back to `getEnvDefaults()`.

<Warning>
Feature defaults were tightened (LOA-907). These flags are now **default-OFF** (opt-in), mirroring the backend resolvers — an absent per-lender row does NOT light up the UI: `appraisal`, `creditReview`, `keyFieldReview`, `enableGuidelineUploads`, `darkMode`, `updatedAppraisalEnabled`, `dscrAgent`, `statementAnalyzer`, `documentRouting`, `releaseNotes`.
</Warning>

`feature-gates.ts` composes flags: `isChatEnabled(flags, canBypassFlags) = flags.chat === true || canBypassFlags`; `isAgencyRulesEnabled = appraisal && agencyRules`. The second parameter of `isChatEnabled` was renamed `isInternal` → `canBypassFlags` so chat respects impersonation — callers now pass `getEffectiveContext().bypassFeatureFlags` (`isInternal && !isImpersonating`), not raw `isInternal`. See [impersonation](/portal/impersonation).

`TopNav` nav items are keyed by flag: Appraisal→`appraisal`, Credit→`creditReview`, Income→`bankStatements`, DSCR→`dscrAgent`, Statement Analyzer→`statementAnalyzer`, What's New→`releaseNotes`.

## List pagination

The shared `<Pagination>` component (`components/ui/pagination.tsx`) and the `useListPagination` hook (`lib/hooks/use-list-pagination.ts`) are wired into five list views: appraisal runs, credit runs, clients (admin), loan programs, and statement-analyzer runs. The income list is intentionally not paginated this way.

## Standalone-run dashboard routing

The dashboard distinguishes a pre-underwriting audit from a standalone agent run via the `AuditOverview` fields `isPreUnderwriting` and `linkedRuns` (`lib/api-types.ts`). A pre-underwriting audit filters on `audit.isPreUnderwriting`; a standalone run is matched through `audit.linkedRuns` by `agent`, and the row links to its first linked run. `AuditSidebar` renders the same `linkedRuns` as per-agent sidebar entries.

<Warning>
The staff properties-check page (`(detail)/loan-audits/[auditRunId]/properties-check`) passes `proxyImages={false}` to `PropertiesCheckView`. The partner results pages already opted out of the Encompass image proxy; this staff page had missed it (LOA-894).
</Warning>

## Redirects (non-page paths)

Several legacy or non-page routes redirect rather than render:

| From | To | Why |
|---|---|---|
| `/guideline-chat` | `/dashboard` | Chat is a floating FAB/drawer, not a page. |
| `/bank-statements` | `/income` | Legacy; income list is unified. |
| `/loan-audits` | `/dashboard` | Dashboard is the canonical landing page. |

## DSCR agent: V1 manual only

The `/dscr-agent` route renders only a manual DSCR calculator. It is gated on `flags.dscrAgent` (default-OFF per LOA-907) and renders a "not enabled" message otherwise. There is no LOS integration in V1 — it is manual entry only.

## Theming

Per-lender dark mode is opt-in via the `darkMode` flag. The CSS/dark-class plumbing is unconditional (every page must be dark-mode compatible using semantic tokens like `bg-card`, `bg-muted`, `text-foreground`). `ChromeShell` passes `forcedTheme="light"` unless `darkMode` is on; the theme toggle in `TopNav` only appears when `darkMode` is enabled.
