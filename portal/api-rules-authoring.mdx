Three modules author the rules that the audit pipeline evaluates: `portal-appraisal-rules`, `portal-credit-rules`, and `portal-income-rules`. They share a template-adoption pattern: a lender adopts a rule set from cross-module templates, then adds, updates, or deletes individual rules within a set. All endpoints use [`PortalAuthGuard`](/portal/api-auth).

Custom rules span four concerns: authoring (these modules), override (`portal-lenders`), display (`portal-appraisal` / `portal-credit`), and evaluation (`lib/*-pipeline/rules`).

## Rule codes, templates, and operators

Rule code format: built-in rules use `[A-Z]{2,5}_\d{3}`; custom rules use `CUST_[A-F0-9]{8}`; credit also uses `CRED_*` template codes. Templates live in the cross-module table `appraisal_rule_templates`, filtered by `module` (`appraisal` or `credit`).

The appraisal custom-rule evaluator supports these operators: `equals`, `not_equals`, `contains`, `not_contains`, `greater_than`, `greater_than_or_equal`, `less_than`, `less_than_or_equal`, `abs_less_than`, `abs_less_than_or_equal`, `regex`, `in`, `not_in`, `date_within_days`.

<Info>
USPAP rules were removed from active rule sets (LOA-726). `getAgencyAvailability` no longer surfaces `uspap`; only `fannie_mae` and `freddie_mac` are adoptable agency sources, even though `'uspap'` is still permitted in the type/CHECK for rendering historical rows.
</Info>

## isCompoundRulesEnabled gate

Compound (multi-condition) custom appraisal rules are gated by `isCompoundRulesEnabled(lenderId)`, which is **opt-in**, not default-enabled. At head it resolves from the umbrella `appraisal` flag (`isAppraisalEnabled`), so an absent flag row resolves to disabled (LOA-994). The `POST .../rule-sets/:setId/rules` endpoints honor this gate when adding a compound rule.

Credit has its own independent compound gate, `isCreditCompoundEnabled` (`credit_compound`, also default-OFF). Both live in `lib/appraisal-pipeline/rules/feature-gates.ts`. See [authentication and authorization](/portal/api-auth) for the full gate list.

<Note>
The `use_client_rule_sets` subsystem was fully removed (LOA-820 T3). Anything referencing client-rule-set opt-out or `isClientRuleSetsExplicitlyDisabled` is stale. The only remaining rule write-gate is the compound gate above.
</Note>

<Warning>
`POST /api/v1/portal/appraisal-rules` returns HTTP 410 Gone. Rule creation is via `POST /rule-sets/:setId/rules`. The sole custom-rule writer is `rule-sets.service.ts::addCustomRule` (writing to `appraisal_rules`); `appraisal_custom_rules` is no longer read and is being dropped.
</Warning>

Services: `rule-sets.service.ts` (custom-rule writer), `nl-parse.service.ts`, `preview.service.ts`, `split-drafts.service.ts`, `custom-rule-evaluator.ts`, `field-registry.ts`.

### Staged bundles (super-admin)

A second controller under `portal-appraisal-rules/staged-bundles` mounts at `api/v1/portal/admin`. Class-level `@Roles("super_admin")`, throttle 100/min. Staged bundles are curated rule sets that internal staff apply to a lender.

| Method / path | Purpose |
|---|---|
| `GET /api/v1/portal/admin/staged-rule-bundles` | List staged appraisal rule bundles (`StagedBundleListItem[]`) |
| `POST /api/v1/portal/admin/lenders/:lenderId/staged-rule-bundles/:bundleId/apply` | Apply a curated bundle to a lender (HTTP 200) |

The `apply` endpoint upserts both `loan_programs` rows and the associated `appraisal_rules` (the credit twin `apply` does the same for `credit_rules`). As of LOA-1872 (migration `0257`), apply also seeds `loan_programs.match_program_aliases` — an array of canonical program-name residuals used by the alias-first EPC program resolver. A row whose bundle definition carries no aliases binds SQL `NULL` (not `'{}'`) so a subsequent re-apply cannot clobber previously-seeded aliases with an empty array. Re-apply is append-only: `DO UPDATE COALESCE(excluded.match_program_aliases, loan_programs.match_program_aliases)` preserves existing aliases on conflict.

## portal-credit-rules

Controller: `portal-credit-rules/portal-credit-rules.controller.ts` at `api/v1/portal/credit-rules`. The credit twin of the appraisal-rules module: templates come from `appraisal_rule_templates WHERE module='credit'`, rule sets/rules live in `credit_rule_sets` / `credit_rules`, and the evaluator is `lib/credit-pipeline/rules/evaluator.ts::evaluateWithDb`.

| Method / path | Purpose |
|---|---|
| `GET /api/v1/portal/credit-rules/field-registry` | `CreditFieldRegistryResponse` (includes `compoundRulesEnabled`) |
| `POST /api/v1/portal/credit-rules/parse-nl` | NL parse (`ParseNlDto`, `credit-nl-parse.service.ts`) |
| `POST /api/v1/portal/credit-rules/preview` | Preview a rule against samples (`PreviewCreditRuleDto`, `credit-preview.service.ts`) |
| `GET /api/v1/portal/credit-rules/templates` | List templates |
| `POST /api/v1/portal/credit-rules/rule-sets` | Adopt |
| `GET /api/v1/portal/credit-rules/rule-sets` | List |
| `GET /api/v1/portal/credit-rules/rule-sets/update-status` | Update status |
| `POST /api/v1/portal/credit-rules/rule-sets/import-updates` | Import updates |
| `GET /api/v1/portal/credit-rules/rule-sets/:id` | Get (404 on wrong tenant) |
| `POST /api/v1/portal/credit-rules/rule-sets/:setId/rules` | Add a custom rule |
| `PATCH /api/v1/portal/credit-rules/rule-sets/:setId/rules/:ruleId` | Update |
| `PATCH /api/v1/portal/credit-rules/rule-sets/:setId/rules` | Bulk update |
| `DELETE /api/v1/portal/credit-rules/rule-sets/:setId/rules/:ruleId` | Delete rule (HTTP 200) |
| `DELETE /api/v1/portal/credit-rules/rule-sets/:id` | Delete set (HTTP 200) |
| `GET /api/v1/portal/credit-rules/split-drafts` | List active split-rule drafts |
| `POST /api/v1/portal/credit-rules/split-drafts` | Create a split draft (`CreateSplitDraftDto`) |
| `GET /api/v1/portal/credit-rules/split-drafts/:id/resume` | Resume a draft |
| `POST /api/v1/portal/credit-rules/split-drafts/:id/dismiss` | Dismiss a draft (HTTP 200) |

All endpoints except the public list/get/template reads are admin-gated by `assertAdmin(ctx)` (`client_admin` or internal). Credit reached rule-authoring parity with appraisal in #2266: `preview` and the `split-drafts` wizard mirror the appraisal endpoints.

<Note>
Credit split drafts are backed by a **separate** table, `credit_rule_split_drafts` (migration `0189`, schema `credit-rule-split-drafts.ts`) — distinct from the appraisal `rule_split_drafts` table.
</Note>

<Warning>
Adding a compound credit rule (one carrying a `condition`) while `credit_compound` is off returns `409 compound_rules_unavailable_legacy_evaluator` (`COMPOUND_RULES_LEGACY_ERROR_CODE`). The same code surfaces read-side as a `SKIPPED` evaluator outcome, so telemetry can join the two.
</Warning>

BFF proxies for these live under `packages/app/src/app/api/credit-rules/*`.
