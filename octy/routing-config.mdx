Routing Config at `/routing-config` is Octy's editor for the **EPC document-routing rule sets** â€” the configuration the integrations service uses to route Encompass documents. It is a thin, versioned config console over the integrations service's routing-config admin API; Octy stores none of this data itself. The page lives in `src/app/(app)/routing-config/` (`page.tsx` â†’ `RoutingConfigPage.tsx`) and the typed API client in `src/lib/integrations-routing-client.ts`.

<Note>
The integrations routing-config admin API is live. Octy can read and author routing rule sets against the integrations service. Per-lender document-push routing (AUS report + property PDFs) is activated by the `document_push` lender feature flag â€” see [Audit lifecycle](/integrations/audit-lifecycle).
</Note>

## Layout

Two tabs, URL-driven (`?tab=`), with the selected lender (`?orgId=`) preserved across tab switches. The default tab is **Global**.

| Tab | `?tab=` | Edits |
|---|---|---|
| Global | `global` | The org-wide default routing rule set. |
| Per-Lender | `lenders` | A single lender's routing rule set, selected by `orgId`. |

Each tab edits the rule set as JSON in a Monaco editor (`RoutingJsonEditor.tsx`) and shows the version history (`VersionHistory.tsx`). Edits go through server actions in `actions/` (`global.ts`, `lenders.ts`, shared error mapping in `shared.ts`).

<Note>
The Monaco editor is why the app's Content-Security-Policy allows `'unsafe-eval'`, `blob:`, `worker-src 'self' blob:`, and the `cdn.jsdelivr.net` script/style source. See [authentication and access control](/octy/auth-and-access).
</Note>

## The versioned config model

Both the global and per-lender rule sets are **versioned and activatable** rather than edited in place. The lifecycle, exposed by the `routingConfig` client, is the same on both tabs:

<Steps>
<Step title="Read the active config">
`global.get()` / `lenders.get(orgId)` returns the current rule set. The per-lender read accepts `includeGlobal` (fold in the global defaults) and `resolve` (return the effective merged config) options.
</Step>
<Step title="Create a new version">
`createVersion(payload, { activateOnCreate })` posts the edited rule set as a new version. With `activateOnCreate: true` the new version goes live immediately.
</Step>
<Step title="Activate or deactivate">
`activateVersion(version)` promotes a prior version; `deactivate()` turns the rule set off (falling back to the global default, for a lender).
</Step>
</Steps>

These map to the integrations service's admin API under the `/integrations/routing-config` prefix:

| Client call | Integrations endpoint |
|---|---|
| `global.get()` | `GET /integrations/routing-config/global` |
| `global.createVersion()` | `POST /integrations/routing-config/global/versions` |
| `global.activateVersion(v)` | `POST /integrations/routing-config/global/versions/{v}/activate` |
| `global.deactivate()` | `POST /integrations/routing-config/global/deactivate` |
| `lenders.list()` | `GET /integrations/routing-config/lenders` |
| `lenders.get(orgId)` | `GET /integrations/routing-config/lenders/{orgId}` |
| `lenders.createVersion(orgId)` | `POST /integrations/routing-config/lenders/{orgId}/versions` |
| `lenders.activateVersion(orgId, v)` | `POST /integrations/routing-config/lenders/{orgId}/versions/{v}/activate` |
| `lenders.deactivate(orgId)` | `POST /integrations/routing-config/lenders/{orgId}/deactivate` |

## How Octy reaches integrations

The client is a close cousin of the loanlight-api client, but points at a **separate service** over its own private internal route (`src/lib/integrations-routing-client.ts`). See [environment variables](/octy/environment-variables):

| Variable | Purpose |
|---|---|
| `INTEGRATIONS_INTERNAL_URL` | Base URL of the integrations internal API. Deployed value `https://integrations-internal.{zone}` (TLS in transit). |
| `INTEGRATIONS_INTERNAL_API_KEY` | Shared secret sent as `x-internal-api-key`. In deployed envs both services read the same `LLIntegrations/INTERNAL_API_KEY` Secrets Manager secret. |

Every call also carries the acting WorkOS user for the integrations audit trail â€” `x-acting-user-id` (WorkOS user id) and `x-acting-user-email` â€” and uses a 120-second timeout. Non-2xx responses surface as `ApiClientError`; `actions/shared.ts` maps the contract's **401 / 400 / 404 / 409** statuses to UI messages, and keeps 401s generic so the internal key is never echoed back to the browser.

## Document-router outcomes on classification cards (LOA-1714)

Octy surfaces document-router routing outcomes directly on classification cards in the **Loan Lookup** modal (the `LoanClassificationsPane` / `DocumentExplorer` views). When a classification has a routing outcome, a **Show routing** toggle appears on the `ClassificationCard` and expands a `ClassificationRoutingSection` showing:

- the outcome details (container name, document type, push state),
- the S3 split-storage path that produced the outcome, and
- a link to the relevant Routing Config page.

Outcomes are fetched by `RoutingOutcomesProvider` via `getRoutingOutcomesByClassificationId`, which fans out two distinct lookups (LOA-1671):

- **`listClassifications`** â€” accepts a `loanSearch` param that matches `loans.loan_number` / `loans.encompass_loan_id` (human-facing loan identifiers) to resolve the segment rows and their `split_storage_path`.
- **`documentPush.getOutcomes`** â€” accepts a `correlationId` param equal to `audit_runs.id` (the EPC `transaction_state.id`). The integrations outcomes endpoint joins on `transaction_state.id` â€” **not** on loan number or `encompassLoanId`. Before LOA-1671 all call sites passed `encompassLoanId`, which is null for effectively all loans, so the join always missed and routing data was never shown.

<Note>
Pass the `correlationId` (= `audit_runs.id`) to `getOutcomes`, not the Encompass loan ID or loan number. Passing the wrong identifier silently empties the outcomes side of the join without an error.
</Note>

Fetch errors yield an empty result â€” the classification card renders normally without the routing section. The `integrations-routing-client.ts` and `integrations-routing-types.ts` modules were extended to carry routing-outcome types and the new `getDocumentRoutingOutcomes` / `getRoutingOutcomesByClassificationId` actions.
