The Bug Reports console at `/bug-reports` is the operator surface where engineers triage reported audit issues and delegate them to Oz, the autonomous coding agent. It is the human side of the [Oz run lifecycle](/octy/oz-run-lifecycle): you pick a skill, attach context, launch a run, then come back to read the agent's summary and any PRs it opened.

The page lives in `src/app/(app)/bug-reports/page.tsx` plus its panel components (`LinearTicketPanel.tsx`, `RunDetailPanel.tsx`, `RunTimeline.tsx`, `ContextRunSelector.tsx`, `ModelPicker.tsx`, `IssueComments.tsx`).

## The issue list

The left side is a paginated table of `octy_issues` rows backed by `GET /api/bug-reports`. Each issue references an `audit_run_id` (always present) and a `loan_id` (optional — loanless portal-appraisal runs have none), and carries aggregate counts (run count, latest run status, PR count, comment count).

Filters are URL-driven, so a filtered view is shareable:

| Filter | Query param | Notes |
|---|---|---|
| Status | `status` / `intake_status` | Comma list. Default shows `new,acknowledged,delegated,ai_complete,in_review` (hides `closed`/`cancelled`). |
| Agent | `agents` | Comma list of agent types, each normalized to its canonical `agent_type` + LangSmith aliases (catalog-tolerant), matched against `related_traces[].agent_type`. |
| Lender | `lenders` | Comma list of lender UUIDs (resolved against `audit_runs.lender_id`). |
| Search | `q` | ILIKE over loan number or `audit_run_id`. |
| Sort | `sort` | `date_asc` or `date_desc`. |

The default page size is 20 (options 10/20/50/100). The `StatusChangerBadge` lets you change an issue's `intake_status` inline; it PATCHes `/api/octy-issues/[id]`.

<Info>
Issues are usually created upstream from the loanlight-api portal (or from Octy's own Lookup BugReportPanel) via `POST /api/bug-reports`. See [Issues, runs, labels API](/octy/issues-runs-labels-api) for the intake contract.
</Info>

## The issue detail panel

Selecting an issue opens a right-hand drawer (`PanelShell`) with the issue meta (loan, audit run, dates, status, labels), the description, related agent runs as LangSmith trace chips, the comment thread, and the Trigger Octy form.

### Trigger Octy

<Steps>
<Step title="Name the run (optional)">
A free-text run name, max 80 characters. If you set it, the agent cannot overwrite it on completion — your label wins.
</Step>
<Step title="Attach previous run context (optional)">
`ContextRunSelector` lists prior runs on this issue that succeeded and have a non-null `summary`. Selected runs are loaded into a "Previous Run Context" section of the prompt, so a fix-and-PR run can build on an earlier root-cause run.
</Step>
<Step title="Review MCP servers">
LangSmith CLI access and the LangChain Docs MCP are shown as read-only checked. The Datadog MCP attaches automatically when the Datadog keys are configured (see [Environment variables](/octy/environment-variables)).
</Step>
<Step title="Pick a model">
`ModelPicker` is a cascading provider menu. The default is `claude-4-8-opus-xhigh`. The chosen `modelId` is passed straight through to Warp's `config.model_id`.
</Step>
<Step title="Pick an affected area">
The **Affected Area** dropdown is populated by `GET /api/v1/audit_runs/:id/bug-reports/agent-options` (LOA-1669). Options are derived per audit run as the union of the run's expected plan (`audit_runs.agent_config`) and the observed `agent_runs` for that audit. Adding a new area is a backend change with no package release or frontend redeploy required.
</Step>
<Step title="Pick a tentacle (skill)">
A radio over `OZ_SKILLS`: Root Cause Analysis, RCA Fix & PR, or Ad Hoc. See [Oz skills](/octy/oz-skills) for what each does.
</Step>
<Step title="Add custom instructions / attach comments">
Custom instructions apply to the fixed tentacles (RCA and fix-and-PR) as an "Additional User Instructions" section. The Attach comments checkbox folds the issue's comments into the prompt.
</Step>
<Step title="Launch">
`handleLaunchOcty` POSTs to `/api/octy-issues/[id]/oz-run`, which builds the prompt, launches the Warp run, writes an `octy_runs` row, and moves the issue to `delegated`.
</Step>
</Steps>

### Live prompt preview

The panel assembles a live preview of the exact prompt the server will build. It calls `buildBasePrompt` client-side with placeholder tokens (and `datadogAvailable: true`) and appends the context, comments, and skill sections, mirroring the server. For the Ad Hoc tentacle the preview is an editable textarea seeded with the skill's pre-fill, which becomes the `customPrompt`.

<Tip>
The preview is the same code path the server runs, so what you see is what the agent receives (minus the real bearer token, which is injected server-side).
</Tip>

## Reviewing run history

The detail panel also surfaces the issue's runs (from `GET /api/octy-issues/[id]/runs`, newest first). `RunDetailPanel` and `RunTimeline` render each run's status, model, skill, and timeline. While a run is active, the UI polls `GET /api/octy-runs/[runId]/status`, which retrieves the live Warp state and persists changes (including the `oz_session_url` Warp session link).

`ACTIVE_OZ_STATUSES` is `queued`, `pending`, `claimed`, `in_progress`. When a run is active the panel warns that a run is in progress but still lets you launch another.

When a run finishes, the completion webhook stores its markdown `summary` and `pr_urls`, and advances the issue to `ai_complete` (only if it was still `delegated`). The summary renders as markdown in the run detail; PR links open against loanlight-api. From there engineers move the issue to `in_review`, `closed`, or `cancelled`.

<Note>
Per-run completion never silently overwrites an engineer's manual status — the `delegated → ai_complete` transition only fires when the issue is still `delegated`. See [Completion webhook](/octy/completion-webhook).
</Note>

## Linear ticket integration

Every Octy bug report owns at most one Linear ticket, regardless of how many Oz runs it accumulates. The dedup unit is the bug report, not the run.

**How a ticket is created:** at the end of an Oz run the agent calls the `attach-linear-ticket` Oz tool. Octy writes the ticket anchor in a race-safe conditional update (`WHERE linear_ticket_identifier IS NULL`) so concurrent runs on the same bug report can never create duplicate tickets.

**Where ticket data lives:** five columns on `octy_issues` (migration `0255`):

| Column | Purpose |
|---|---|
| `linear_ticket_id` | Internal Linear UUID. |
| `linear_ticket_identifier` | Human-readable id (e.g. `LOA-1234`). |
| `linear_ticket_url` | Deep-link into Linear. |
| `linear_ticket_attached_at` | When the anchor was written. |
| `linear_ticket_attached_by_run_id` | The `octy_runs.id` of the Oz run that attached the ticket. FK ON DELETE SET NULL. |

`linear_snapshot` (jsonb) is a read-through display cache of the ticket (state, assignee, priority, cycle, labels) refreshed by Octy on each Linear fetch. Linear stays the source of truth; staleness is display-only.

**`LinearTicketPanel`:** the issue detail panel shows a Linear tab when a ticket is attached. It displays the live ticket state (via `GET /api/linear/meta` and `GET /api/linear/search`) and lets engineers navigate directly to Linear for label and assignee management.

**Retiring in-house labels:** migration `0256` dropped `octy_labels` and `octy_issue_labels`. Labels now live on the Linear ticket and are mirrored in `linear_snapshot`. The previous in-house label assignments were not migrated — Linear provides a fresh taxonomy.
